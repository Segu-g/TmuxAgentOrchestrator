## §10.98 v1.2.23 — ワークフローテンプレートのパラメータ継承 (Workflow Template `defaults:` + Context Packs)

### Step 0 — 選択根拠

**選択**: ワークフローテンプレートのパラメータ継承 (`examples/workflows/*.yaml` への `defaults:` セクション追加) + エージェントコンテキストパックの `planner.md` / `spec-writer.md` / `architect.md` ロール別docs追加

**理由**: ユーザー指示「強めるべきはワークフローのテンプレート、プリセットとそれに付随するエージェントコンテキストパック」に最も直接的に応える。具体的な問題:
1. 現行の `examples/workflows/*.yaml` では `task_timeout`, `required_tags` のパターンを各 `phases[]` エントリに個別に書く必要があり、テンプレートの保守コストが高い
2. `examples/workflows/` に 15+ ワークフローがあるが、agent コンテキストパック (`role:` フィールドで参照できる `.claude/rules/*.md`) はまだ `tester.md` / `coder.md` / `reviewer.md` / `worker.md` / `director.md` の 5 つのみ
3. Argo Workflows の `templateDefaults` や GitLab CI の `default:` セクションが「フェーズ共通設定の一箇所定義」を実現しており、TmuxAgentOrchestrator でも同じパターンを採用できる

**選択しなかった候補**:
- チェックポイント永続化 (高優先度だが実装規模が大きく、SQLite スキーマ設計と `--resume` フラグが必要。単イテレーションで完結しない)
- CleanArch 層別ディレクトリ移行 (アーキテクチャ品質改善だが、ユーザー戦略「ワークフローテンプレート優先」と方向が逆)
- Stop フック完了検出 (重要だが tmux/claude CLI のバージョン依存があり、安定性より機能拡充を優先)

**実装範囲**:
1. `workflow_defaults.py` の `deep_merge_defaults()` に `defaults:` YAML セクションを処理する機能を追加 — `POST /workflows` 受付時に `defaults` を各フェーズに merge する前処理
2. `examples/workflows/*.yaml` テンプレート全てに `defaults:` セクションを追記
3. `planner.md` / `spec-writer.md` / `architect.md` の 3 ロール別 docs を `.claude/rules/` に追加
4. `web/schemas.py` の `WorkflowSubmit` に `defaults: dict | None` フィールドを追加

### Step 1 — 調査記録 (2026-03-15)

#### Query 1: workflow YAML template defaults inheritance multi-agent orchestration 2025

**Source**: Microsoft Foundry Blog "Introducing Multi-Agent Workflows in Foundry Agent Service"
https://devblogs.microsoft.com/foundry/introducing-multi-agent-workflows-in-foundry-agent-service/ (2025)

**Source**: Argo Workflows Documentation "Template Defaults"
https://argo-workflows.readthedocs.io/en/latest/template-defaults/ (2025)

**Key findings**:
- Argo Workflows の `templateDefaults` は workflow spec レベルで全テンプレート共通のデフォルト値を定義する
- 優先度: Template固有の値 > `templateDefaults` > コントローラーレベルのデフォルト
- Kubernetes strategic merge patch を使用; テンプレート固有の値が常に優先される
- timeout, retryStrategy などを一箇所で定義できる

#### Query 2: CrewAI LangGraph workflow template presets agent context packs 2025

**Source**: CrewAI Changelog (docs.crewai.com/en/changelog)
https://docs.crewai.com/en/changelog (2025-2026)

**Source**: IBM Developer "Comparing AI agent frameworks: CrewAI, LangGraph, and BeeAI"
https://developer.ibm.com/articles/awb-comparing-ai-agent-frameworks-crewai-langgraph-and-beeai/ (2025)

**Key findings**:
- CrewAI v0.120.0+ では Agent を YAML + repository ベースで定義でき、複数ワークフローで再利用できる
- 「ドキュメントとしての設定ファイル」(YAML-as-documentation) アプローチが 2025 の主流 — CrewAI・LangGraph 両者が採用
- system_prompt テンプレートと response_template の分離により、同一エージェント定義を複数コンテキストで使い回せる
- Context window management と state persistence が v1.0.0+ で強化

#### Query 3: YAML configuration inheritance defaults merging best practices 2025

**Source**: GitLab Docs "Optimize GitLab CI/CD configuration files"
https://docs.gitlab.com/ci/yaml/yaml_optimization/ (2025)

**Source**: GitHub Changelog "Actions: YAML anchors and non-public workflow templates"
https://github.blog/changelog/2025-09-18-actions-yaml-anchors-and-non-public-workflow-templates/ (2025)

**Key findings**:
- GitLab CI の `default:` セクションが全ジョブに適用されるデフォルト値を一箇所で定義する最良の参照実装
- `extends:` キーワードによる多段継承 (最大11段、推奨3段) でテンプレートの再利用が可能
- 後の定義が常に優先 (last-writer-wins); 配列は置換 (merge しない)
- YAML アンカー (`&` / `*`) は同一ファイル内のみ有効; `extends` はファイルをまたいで使える
- GitHub Actions 2025-09 にも YAML アンカーが導入され、共通ステップの再利用が標準化

### Design Decisions

1. **`WorkflowSubmit.defaults`**: `dict[str, Any] | None` フィールドを追加。`task_timeout`, `required_tags`, `priority` 等のフェーズ共通設定を一箇所で定義できる
2. **`deep_merge_defaults(defaults, phases)`**: 既存 `workflow_defaults.py` の関数を拡張。Phase-level 値が `defaults` より常に優先される (phase-wins-over-defaults)
3. **`examples/workflows/*.yaml` の `defaults:` セクション**: 全テンプレートに `defaults: { task_timeout: 300 }` を追加して冗長な個別設定を削除
4. **ロール別 docs 追加**: `planner.md` (計画立案) / `spec-writer.md` (仕様作成) / `architect.md` (アーキテクチャ設計) を `.claude/rules/` に追加
5. **CLAUDE.md ロールテーブル更新**: 新しいロール docs への参照を追記
