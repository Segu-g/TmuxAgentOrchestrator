**選択**: `POST /workflows/socratic` — 3エージェント ソクラテス的対話ワークフロー (questioner → responder → synthesizer)

**理由**:
1. ユーザー指示「ワークフローテンプレートの拡充」の継続。DESIGN.md §11 の中優先度候補として明示されている
2. ソクラテス的対話はLLMエージェントでの実装研究が豊富 (SocraSynth arXiv:2402.06634, KELE EMNLP 2025)

5. 既存のエージェントロールdocs (worker.md, director.md, reviewer.md) を活用可能

- Competitive strategy criteria customization: API拡張のみで新ワークフローを追加しない。テンプレート拡充の方向性と異なる
- DriftMonitor semantic similarity: インフラ改善で新デモが困難

**実装範囲**:

### Step 1 — 調査記録 (2026-03-15)

#### Query 1: SocraSynth LLM Socratic multi-agent dialogue workflow 2024 2025

**Source**: SocraSynth: Multi-LLM Reasoning with Conditional Statistics — arXiv:2402.06634
https://arxiv.org/abs/2402.06634

**Key findings**:
- SocraSynth: 複数LLMエージェントが構造化討論を行うプラットフォーム
- 2段階: Knowledge Generation (モデレーターがトピック設定・各エージェントが論拠形成) → Reasoning Evaluation (ソクラテス的推論・論理評価)
- Contentiousness level (対立強度) を段階的に「対立→協調」に調整
- Conditional statistics + systematic context enhancement
- Multi-LLM Agent Collaborative Intelligence (ACM Book 2025) に発展

**Source**: Stanford MACI Lab SocraSynth page
http://infolab.stanford.edu/~echang/SocraSynth.html

**Key findings**:
- Human moderator + 2 LLM agents representing opposing viewpoints
- Moderator adjusts contentiousness from adversarial → collaborative
- ソクラテス的推論で各エージェントの論拠を形式論理で評価

#### Query 2: Socratic questioning LLM agent requirement elicitation design review 2025

**Source**: "Reflecting in the Reflection: Integrating a Socratic Questioning Framework into Automated AI-Based Question Generation" — arXiv:2601.14798 (2025)
https://arxiv.org/abs/2601.14798

**Key findings**:
- ソクラテス的問答をAI質問生成に統合: 単純一発質問より反復的Socratic対話が高品質な反省的回答を引き出す
- 構造化問答プロトコル (Socratic Interaction Protocol) が重要
- modest model + Socratic protocol > 大型モデル + single prompt

#### Query 3: KELE Socratic teaching agent EMNLP 2025 arXiv 2409.05511

**Source**: "KELE: A Multi-Agent Framework for Structured Socratic Teaching with Large Language Models" — EMNLP Findings 2025
https://aclanthology.org/2025.findings-emnlp.888/
Authors: Xian Peng, Pan Yuan, Dong Li, Junlong Cheng, Qin Fang, Zhi Liu

**Key findings**:
- 「consultant-teacher」デュアルエージェント: consultant (teaching planning) + teacher (execution)
- SocRule: 構造化ソクラテス教授規則システム (34 teaching strategies)
- SocratDataset: 42,000+ dialogue turns のソクラテス対話データセット
- 9次元評価システム (単一ターン → 複数ターン教授プロセス)
- SocratTeachLLM が GPT-4o を全ソクラテス能力で上回る

**Source**: "Enhancing Critical Thinking in Education by means of a Socratic Chatbot" — arXiv:2409.05511 EMNLP 2025
https://arxiv.org/abs/2409.05511

**Key findings**:
- ソクラテス的チャットボットがクリティカルシンキングを向上
- questioner (問いを発する) → responder (精緻化された回答) → synthesizer (構造化結論抽出) のパターンが効果的

### Design Decisions

1. **`SocraticWorkflowSubmit`**: `topic` (required), `context=""`, `rounds=1`, `language="english"`, `scratchpad_prefix=""`, `agent_timeout=300`, `questioner_tags=[]`, `responder_tags=[]`, `synthesizer_tags=[]`, `reply_to=None`
2. **DAG構造**: questioner_r1 → responder_r1 → questioner_r2 → responder_r2 → ... → synthesizer (rounds×2+1エージェント)
   - ただし最小構成 (rounds=1): questioner → responder → synthesizer (3エージェント)
3. **エージェントロール**:
   - questioner: `reviewer` ロール — 前提・定義・論拠を問うマイウティカ法
   - responder: `coder` ロール — 問いに答えて考えを精緻化
   - synthesizer: `reviewer` ロール — 対話ログから構造化結論を抽出
4. **スクラッチパッドキー**: `{prefix}_q_r{n}` (questioner), `{prefix}_a_r{n}` (responder), `{prefix}_synthesis`
5. **プロンプト**: questioner は SocraSynth の "contentiousness → collaborative" 移行を模倣。synthesizer はSocraSynth arXiv:2402.06634 と KELE EMNLP 2025 の知見を適用


**修正**: `/workflows/socratic` は既に v1.0.25 で実装済みと確認。`§10.101` を code-audit ワークフローに変更する。

## §10.101 v1.2.26 — コード監査ワークフロー (POST /workflows/code-audit) [REVISED]

### Step 0 — 選択根拠 (2026-03-15)

**選択**: `POST /workflows/code-audit` — 3エージェント コード監査ワークフロー
(implementer → [security-auditor ∥ performance-auditor] → synthesizer)

**理由**:
1. ユーザー指示「ワークフローテンプレートの拡充」の継続
2. `/workflows/socratic` は既に実装済みと判明。新規ワークフローが必要
3. コード監査は実際のソフトウェア開発で需要が高い自動化対象
4. 並列監査 + 統合 (fan-out/fan-in) の新しいトポロジー: 既存の mob-review (4並列) や redblue (対立型) とは異なるパターン
5. 実装エージェント → 並列専門監査 → 統合という流れが明確なデモを可能にする

**パターン**: implementer → [security-auditor ∥ performance-auditor] → synthesizer
- implementer: 機能を実装してコードをスクラッチパッドに書き込む
- security-auditor: セキュリティ観点でコードを監査 (OWASP Top 10 + CWE)
- performance-auditor: パフォーマンス観点でコードを監査 (complexity, caching, algorithms)
- synthesizer: 両監査結果を統合し優先度付きの AUDIT_REPORT.md を作成

**選択しなかった候補**:
- socratic multi-round強化: 既存実装との互換性が複雑
- DECISION.md標準フォーマット: 全ワークフロー横断の大規模変更
- DriftMonitor semantic similarity: インフラ改善でデモが困難

**実装範囲**:
1. `web/schemas.py` に `CodeAuditWorkflowSubmit` モデル追加
2. `web/routers/workflows.py` に `POST /workflows/code-audit` エンドポイント追加
3. `examples/workflows/code-audit.yaml` テンプレート追加
4. `tests/test_workflow_code_audit.py` — 40+ テスト


### Step 1 — 調査記録 (2026-03-15)

#### Query 1: LLM automated code audit security performance multi-agent 2025

**Source**: RepoAudit: An Autonomous LLM-Agent for Repository-Level Code Auditing — arXiv:2501.18160 ICML 2025
https://arxiv.org/abs/2501.18160

**Key findings**:
- RepoAuditは15ベンチマークプロジェクトで78.43%精度、40バグ検出 (0.44時間, $2.54/プロジェクト)
- On-demand codebase exploration (data-flow analysis along feasible paths)
- Validator module でハルシネーション抑制 (path condition satisfiability check)
- 3バグ種別特化: null pointer dereference, memory leak, use-after-free
- 複数特化エージェントが並列監査を実施する設計が高精度の鍵

**Source**: iAudit: Combining Fine-tuning and LLM-based Agents for Smart Contract Auditing — ICSE 2025
https://daoyuan14.github.io/papers/ICSE25_iAudit.pdf

**Key findings**:
- 2段階: fine-tuning (localization) + LLM agent (explanation)
- multi-agent conversational architecture で 98% accuracy
- セキュリティ特化エージェントとパフォーマンス特化エージェントの分離が精度向上に寄与

**Source**: Automating Security Audit Using Large Language Models — arXiv:2505.10732 (2025)
https://arxiv.org/pdf/2505.10732

**Key findings**:
- 専門化されたセキュリティ監査エージェント (OWASP Top 10 分類) が汎用モデルより大幅に高精度
- セキュリティ観点とパフォーマンス観点を分離した並列監査が最良実践

#### Query 2: RepoAudit ICML 2025 — confirmed key design principles

**Source**: ICML 2025 Poster: RepoAudit
https://icml.cc/virtual/2025/poster/45170

**Key findings**:
- Specialist agent roles (auditor types) → parallel scan → synthesis of findings
- 各監査タイプ専門エージェントが並列実行し、validator/synthesizer が統合
- このパターンを `code-audit` ワークフローの設計に直接適用

### Design Decisions

1. **`CodeAuditWorkflowSubmit`**: `feature` (required), `language="python"`, `audit_focus=["security","performance"]`, `scratchpad_prefix=""`, `agent_timeout=300`, `implementer_tags=[]`, `security_auditor_tags=[]`, `performance_auditor_tags=[]`, `synthesizer_tags=[]`, `reply_to=None`
2. **DAG構造**: implementer → [security_auditor ∥ performance_auditor] → synthesizer (fan-out/fan-in)
   - implementer: 機能実装 → `{prefix}_impl` に書き込み
   - security_auditor: `{prefix}_impl` を読み → `{prefix}_security_audit` に書き込み
   - performance_auditor: `{prefix}_impl` を読み → `{prefix}_performance_audit` に書き込み
   - synthesizer: 両監査読み → `{prefix}_audit_report` に書き込み (優先度付きレポート)
3. **スクラッチパッドキー**: `{prefix}_impl`, `{prefix}_security_audit`, `{prefix}_performance_audit`, `{prefix}_audit_report`
4. **監査プロンプト**: OWASP Top 10 (security), Big-O / caching / DB (performance)
5. **auto-prefix**: `audit_{8hex}`
