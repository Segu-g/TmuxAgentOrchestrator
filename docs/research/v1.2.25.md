## §10.100 v1.2.25 — ミューテーションテストワークフロー (POST /workflows/mutation-test)

### Step 0 — 選択根拠 (2026-03-15)

**選択**: `POST /workflows/mutation-test` — 3エージェント変異テストパイプライン (implementer → mutant-introducer → test-improver)

**理由**:
1. ユーザー指示「ワークフローテンプレートの拡充」に応える新規ワークフロー
2. 変異テスト（Mutation Testing）はソフトウェアテスト品質の客観的評価手法として確立された研究基盤があり、LLMエージェントによる自動化は最新の研究トレンド
3. 3エージェント構成 (implementer → mutant-introducer → test-improver) は依存関係が明確な線形DAGで実装が容易
4. v1.2.24デモで発見した「bashスニペットが複雑すぎる」問題を修正: 新ワークフローではシンプルなスクラッチパッド書き込みパターンを採用
5. `tester.md` と `coder.md` の既存ロールdocsを活用する最初のワークフロー

**選択しなかった候補**:
- チェックポイント永続化: 大規模実装。単イテレーションで完結しない
- YAML テンプレート追加のみ: コンテンツ量が少なくテスト件数が増えない

**実装範囲**:
1. `web/schemas.py` に `MutationTestWorkflowSubmit` モデル追加
2. `web/routers/workflows.py` に `POST /workflows/mutation-test` エンドポイント追加
3. `examples/workflows/mutation-test.yaml` テンプレート追加
4. `tests/test_workflow_mutation_test.py` — 35+ テスト
5. スクラッチパッドスニペットを全ワークフローでシンプル化 (fixup)


### Step 1 — 調査記録 (2026-03-15)

#### Query 1: mutation testing LLM agents automated test quality improvement 2025

**Source**: "Mutation-Guided LLM-based Test Generation at Meta" — FSE 2025 Industry Papers
https://arxiv.org/abs/2501.12862

**Source**: Meta Engineering Blog "LLMs Are the Key to Mutation Testing and Better Compliance"
https://engineering.fb.com/2025/09/30/security/llms-are-the-key-to-mutation-testing-and-better-compliance/ (2025-09)

**Key findings**:
- Meta の ACH (Automated Compliance Hardening) は LLM でミュータントを生成し、そのミュータントを殺すテストを生成する2段階アプローチ
- 9,095 ミュータント → 571 プライバシー強化テストケース生成。73% がエンジニアに承認
- LLM-based Equivalence Detector でミュータントの等価性を検出 (精度0.95, 再現率0.96)

#### Query 2: LLM code mutation generation test weakness detection multi-agent 2025

**Source**: "Test vs Mutant: Adversarial LLM Agents for Robust Unit Test Generation" — arXiv:2602.08146
https://arxiv.org/html/2602.08146

**Key findings**:
- AdverTest: テスト生成エージェント(T) とミュータント生成エージェント(M) が交互に対戦する双方向フィードバックループ
- Mutation Score + Coverage フィードバックで両エージェントを進化させる
- 最先端の手法より fault detection 8.56% 向上
- Mを使った adversarial loop: T がサバイブしたミュータントに対してテストを追加 → M が coverage gap にミュータントを生成

**Source**: "A Survey on Code Generation with LLM-based Agents" — arXiv:2508.00083
https://arxiv.org/html/2508.00083v1

**Key findings**:
- パイプライン型: implementer → mutant-introducer → test-improver の3段階線形DAGが実装容易
- 変異テストはテスト品質の客観的指標として最も信頼性が高い手法

### Design Decisions

1. **DAG構造**: implementer → mutant-introducer → test-improver (線形シーケンス)
2. **エージェントロール**:
   - implementer: `coder` ロール — 機能実装 + 初期テスト作成
   - mutant-introducer: `coder` ロール — 実装にバグを意図的導入
   - test-improver: `tester` ロール — ミュータントを殺すテスト追加
3. **スクラッチパッドキー**: `{prefix}_impl`, `{prefix}_mutants`, `{prefix}_improved_tests`
4. **簡略化スニペット**: 複雑なbashを避け、Pythonワンライナーで書き込み
