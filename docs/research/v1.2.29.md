## §10.104 — v1.2.29 研究記録: YAML汎用ワークフローテンプレートライブラリ拡張 + docs/research/ 再編

### 選択根拠

v1.2.28 では `POST /workflows/from-template` エンドポイントと最初の3テンプレート (tdd, debate, review) を実装した。
次の自然な拡張はテンプレートライブラリを充実させることで、YAMLのみで追加できる仕組みの価値を実証することにある。

**選択**: YAMLテンプレートライブラリの拡張（7テンプレート追加）+ docs/research/ フォルダ再構成

**理由**:
- §11「低: ワークフローテンプレートライブラリ」のアイテムを前進させる。v1.2.28でインフラが整備されたため実施タイミングが最適。
- 新テンプレートはコード変更不要（YAMLのみ）なので実装リスクが最小。
- clean-arch テンプレートは Martin (2017) と Cockburn (2005) の研究に基づく独立した設計パターンを示す。
- `docs/research-log.md` が 9646 行に達しており、ナビゲーションが困難になっている。

**選ばなかったもの**:
- DriftMonitor embedding 強化: sentence-transformers 依存が発生する（中優先度）。
- TLA+ 形式仕様化: 研究色が強くユーザー向け即効性が低い（低優先度）。

### Step 1 — 調査結果 (2026-03-15)

#### Query 1: "Clean Architecture hexagonal ports adapters multi-agent workflow 2025"

**Source**: Martin, Robert C. "Clean Architecture: A Craftsman's Guide to Software Structure and Design" (2017)
- Dependency Rule: source code dependencies must point INWARD, toward higher-level policies.
- 4 concentric layers: Entities (domain) → Use Cases → Interface Adapters → Frameworks & Drivers.
- Applied to multi-agent systems: each agent can own one layer. Architect agent designs the ring structure; specialist agents implement each ring independently (no circular dependencies).

**Source**: Cockburn, Alistair. "Hexagonal Architecture / Ports and Adapters" (2005)
- URL: https://alistair.cockburn.us/hexagonal-architecture/
- The application has a "core" and communicates with the outside world through "ports" (abstract interfaces).
- "Adapters" implement ports for specific technologies (HTTP, DB, CLI).
- Direct mapping to agent roles: domain-agent implements core, adapter-agent implements port implementations.

**Source**: Hexagonal Architecture — Java Code Geeks (2025)
- URL: https://www.javacodegeeks.com/2025/12/hexagonal-architecture-ports-and-adapters-achieving-true-domain-independence.html
- Recent 2025 emphasis: hexagonal architecture's relevance increases as systems grow more complex.
- Pattern's isolation, testability, and adaptability remain core values.

**Source**: AgentMesh arXiv:2507.19902 (Elias, 2025)
- 4-role Planner/Coder/Debugger/Reviewer pipeline for software development.
- Each role specialises in one aspect; output of each phase feeds the next.
- Multi-agent decomposition mirrors Clean Architecture layer separation.

**Source**: From Hexagonal- to Clean Architecture — Codeartify Substack (Oliver Zihler)
- URL: https://codeartify.substack.com/p/from-hexagonal-to-clean-architecture
- Clean Architecture extends hexagonal principles with additional internal structure (entity/use-case split).

**設計への応用**: `clean-arch.yaml` テンプレートは architect → (domain-agent | usecase-agent | adapter-agent) 並列 → verifier の5フェーズDAGとして実装。architect がポート定義のスタブを書き、domain/usecase/adapter エージェントが並列に実装し、verifier が依存方向制約を静的解析で確認する。

#### Query 2: "SocraSynth Socratic dialogue multi-agent deliberation arXiv 2402.06634"

**Source**: Chang, Edward Y. "SocraSynth: Multi-LLM Reasoning with Conditional Statistics" — arXiv:2402.06634 (Stanford, 2024)
- URL: https://arxiv.org/abs/2402.06634
- SocraSynth は多数の LLM エージェントが Socratic reasoning と formal logic を使って高品質な知識を抽出するプラットフォーム。
- 2フェーズ: knowledge generation (各エージェントが論点を展開) → reasoning evaluation (Socratic cross-examination で論点を評価)。
- 「contentiousness level」を調整してエージェントが対立的 vs 協調的な姿勢で討論するよう制御。
- 最後に moderator が contentiousness を下げて最終合意文書を生成する。

**Source**: MACI (Multi-LLM Agent Collaborative Intelligence) — Stanford InfoLab
- URL: http://infolab.stanford.edu/~echang/SocraSynth.html
- SocraSynth の後継プロジェクト。multi-round Socratic dialogue で「知識生成と意思決定支援」を統合。

**Source**: SocratiQ — AI-Driven Socratic Dialogue Systems (EmergentMind, 2025)
- URL: https://www.emergentmind.com/topics/socratiq
- Programming learning における Socratic method の有効性を実証。

**Source**: "When Generative AI Meets Socratic Method" — Journal of Computer Assisted Learning (2026)
- URL: https://onlinelibrary.wiley.com/doi/10.1002/jcal.70210
- Socratic AI interaction が学習動態と認識品質を改善することを実証。

**設計への応用**: `deliberate.yaml` テンプレートは questioner (Socratic examination) → responder (considered answers) → synthesizer (actionable conclusions) の3フェーズとして実装。Chang (2024) の「cross-examination + formal logic evaluation」フェーズに対応する。

#### Query 3: "workflow template YAML variable substitution best practices 2025"

**Source**: GitHub Actions — YAML anchors and non-public workflow templates (September 2025)
- URL: https://github.blog/changelog/2025-09-18-actions-yaml-anchors-and-non-public-workflow-templates/
- YAML anchors allow reuse of configuration across workflows.
- Variable substitution `${{ variables.x }}` with explicit type declarations is the 2025 standard.

**Source**: Azure Pipelines template variable substitution — Microsoft Learn (2025)
- URL: https://learn.microsoft.com/en-us/azure/devops/pipelines/process/templates
- Parameters with type, default, and required flags are the canonical pattern.
- Security: use parameters over variables where type safety matters (for shell tasks).

**Source**: Argo Workflows — WorkflowTemplate reference (2025)
- URL: https://argo-workflows.readthedocs.io/en/latest/workflow-templates/
- `WorkflowTemplate` as a reusable resource with `templateRef` for instantiation.
- Input parameters with defaults: `{{inputs.parameters.message | default "hello"}}`.

**Source**: Google Cloud Workflows best practices (2025)
- URL: https://docs.cloud.google.com/workflows/docs/best-practice
- One substitution technique: write a single workflow definition with placeholders, deploy variants with a tool that replaces them.

**設計への応用**: v1.2.28 の `str.format_map()` 設計を継続。Azure Pipelines の `required: true/false` + `default:` パターンが既実装と一致していることを確認。

### 設計決定 (v1.2.29)

**7テンプレート追加**:

| Template | Pattern | Phases | Required Tags |
|----------|---------|--------|---------------|
| `agentmesh.yaml` | 4-stage pipeline | planner → coder → debugger → reviewer | agentmesh_{planner,coder,debugger,reviewer} |
| `refactor.yaml` | 3-stage pipeline | analyzer → refactorer → verifier | refactor_{analyzer,refactorer,verifier} |
| `redblue.yaml` | adversarial review | implement → attack → assess | redblue_{blue,red,arbiter} |
| `clean-arch.yaml` | fan-out + fan-in | architect → (domain\|usecase\|adapter) → verifier | clean_arch_{architect,domain,usecase,adapter,verifier} |
| `delphi.yaml` | multi-round consensus | expert-r1 → moderator-r1 → expert-r2 → consensus | delphi_{expert,moderator} |
| `deliberate.yaml` | Socratic dialogue | questioner → responder → synthesizer | deliberate_{questioner,responder,synthesizer} |
| `spec-first.yaml` | spec-driven dev | spec-writer → implementer → acceptance-tester | spec_{writer,implementer,tester} |

**`clean-arch.yaml` の並列フェーズ**: domain-agent, usecase-agent, adapter-agent は全て architect に `depends_on` し、verifier は全3に `depends_on`。これにより architect完了後に3エージェントが並列で稼働し、全完了後に verifier が制約確認する。

**docs/research/ 再構成**:
- `docs/research-log.md` はアーカイブとして残し、先頭に移行ヘッダーを追加。
- `docs/research/pre-v1.2.md`: v1.2.0 以前の全エントリ（v0.3.0〜v1.1.44）
- `docs/research/v1.2.{N}.md`: 各バージョンの研究記録（v1.2.1〜v1.2.29）
- `docs/research/README.md`: インデックス
