# v1.0.23 — POST /workflows/delphi — 多ラウンド合意形成ワークフロー

## 問題の背景

現在実装されている `POST /workflows/debate` は Advocate / Critic / Judge の3役割による2項対立構造。しかし実際の意思決定や設計評価では、セキュリティ・パフォーマンス・保守性・UX・コストといった**多次元の専門家視点**が必要なことが多い。Delphi 法は 1950 年代に RAND Corporation で開発された、複数の専門家が**匿名**で意見を提出し、モデレーターが集計して次ラウンドにフィードバックするという反復合意形成手法。LLM マルチエージェントシステムへの適用が 2025 年の研究で実証されている。

## 提案する解決策

`POST /workflows/delphi` エンドポイントを追加する。

### エンドポイント仕様

```
POST /workflows/delphi
```

**リクエストボディ** (`DelphiWorkflowSubmit`):

| フィールド | 型 | デフォルト | 説明 |
|---|---|---|---|
| `topic` | `str` | 必須 | 合意形成のテーマ |
| `experts` | `list[str]` | `["security", "performance", "maintainability"]` | 専門家ペルソナ名リスト (3–5個) |
| `max_rounds` | `int` | `2` | Delphi ラウンド数 (1–3) |
| `expert_tags` | `list[str]` | `[]` | 専門家エージェント候補のタグ |
| `moderator_tags` | `list[str]` | `[]` | モデレーターエージェントのタグ |
| `reply_to` | `str \| None` | `None` | 最終合意書の通知先エージェント |

**レスポンス**:

```json
{
  "workflow_id": "<uuid>",
  "name": "delphi/<topic>",
  "task_ids": {
    "expert_security_r1": "<task-id>",
    "expert_performance_r1": "<task-id>",
    "moderator_r1": "<task-id>",
    ...
    "consensus": "<task-id>"
  },
  "scratchpad_prefix": "delphi_<8chars>"
}
```

### ワークフロー DAG 構造

```
Round 1:
  expert_security_r1  ─┐
  expert_performance_r1─┼─→ moderator_r1 ─→ (Round 2 start)
  expert_maintainability_r1─┘

Round 2 (if max_rounds >= 2):
  expert_security_r2  ─┐
  expert_performance_r2─┼─→ moderator_r2 ─→ consensus
  expert_maintainability_r2─┘

Final:
  consensus (reads all moderator summaries → writes consensus.md)
```

各ラウンドのエキスパートタスクは **parallel** に実行（`depends_on` なし、または前ラウンドのモデレーターのみに依存）。

### Scratchpad キー設計

| キー | 内容 |
|---|---|
| `{prefix}_r{n}_{expert}` | 専門家ペルソナ `{expert}` の Round `n` 意見 |
| `{prefix}_r{n}_moderator` | モデレーターの Round `n` 集計・フィードバック |
| `{prefix}_consensus` | 最終合意書 (consensus.md の内容) |

### アーティファクト

- `expert_{persona}_r{n}.md` — 各専門家の各ラウンド意見
- `delphi_round_{n}.md` — モデレーターの集計・フィードバック
- `consensus.md` — 最終合意文書

## 実装計画

### 1. `DelphiWorkflowSubmit` モデル追加 (`web/app.py`)

Pydantic モデル。`experts` リストのバリデーション（3–5要素、空文字禁止）、`max_rounds` バリデーション（1–3）。

### 2. `submit_delphi_workflow` ハンドラー実装

debate と同パターンで:
1. `uuid4()` で `scratchpad_prefix` 生成
2. Round ごとに:
   - 各専門家エージェントに並列タスク投入（前ラウンドのモデレータータスクに `depends_on`）
   - モデレータータスクを全専門家タスクに `depends_on`
3. 最終 consensus タスクを最終ラウンドのモデレータータスクに `depends_on`

### 3. テスト (`tests/test_delphi_workflow.py`)

- `DelphiWorkflowSubmit` のバリデーション（空 topic、experts 数超過、max_rounds 範囲外）
- DAG 構造検証（ラウンド数・タスク数・依存関係）
- スクラッチパッドキー命名規則
- `reply_to` が設定された場合、consensus タスクに `reply_to` が渡される
- OpenAPI スナップショット更新
