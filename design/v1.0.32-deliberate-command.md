# v1.0.32 — `/deliberate` Slash Command

## Problem

Agents currently have no self-service tool to resolve design questions internally. When a worker or director agent faces a design trade-off (e.g., "SQLite vs PostgreSQL", "REST vs GraphQL"), it must either hard-code a choice, ask the orchestrator, or spawn sub-agents manually through a multi-step process.

Single-LLM decision-making suffers from confirmation bias — the agent tends to favour its initial position. Research shows multi-agent debate with a Devil's Advocate role substantially reduces this bias (DEBATE, ACL 2024).

## Solution

A new slash command `/deliberate <question>` that:
1. Discovers an available worker template
2. Spawns two sub-agents: **advocate** (pro position) and **critic** (con position)
3. Briefs each with a clear role and structured output requirements
4. Guides the calling agent to synthesize arguments into `DELIBERATION.md`

This is implemented purely as a slash command — no new REST endpoints, no new server code. It composes existing primitives: `POST /agents` (spawn), `POST /agents/{id}/message` (P2P brief), inbox polling.

## Research

### DEBATE Framework (ACL 2024, arXiv:2405.09935)
Kim, Kim, Yoon — "DEBATE: Devil's Advocate-Based Assessment and Text Evaluation"
- Multi-agent scoring system with a Devil's Advocate who criticises other agents' arguments
- Outperforms SOTA on SummEval (Spearman 0.847 vs 0.762) and TopicalChat
- **Key insight**: 2 agents, 2 rounds is the minimum effective configuration
- Forced polarisation (advocate must take pro, critic must take con) produces stronger arguments than balanced "discuss both sides" prompting

### CONSENSAGENT (ACL 2025)
Pitre, Ramakrishnan, Wang — "Towards Efficient and Effective Consensus in Multi-Agent LLM Interactions Through Sycophancy Mitigation"
- Sycophancy (agents agreeing without critical engagement) is the primary failure mode in MAD systems
- Explicit critic role with "argue AGAINST" instruction suppresses sycophancy
- Implementation detail: critic brief must say "Do NOT hedge — take the con position firmly"

### IUI 2024 — LLM-Powered Devil's Advocate
Yin et al. — "Enhancing AI-Assisted Group Decision Making through LLM-Powered Devil's Advocate"
- LLM devil's advocates improve human group decision quality (statistically significant)
- The advocate/critic pattern translates directly to autonomous agent settings

## Design Decisions

### Why a slash command (not a workflow endpoint)?
- Zero server changes required — all logic runs client-side in the agent's Python snippet
- Agents can use it ad-hoc during any task, not just pre-planned workflows
- Composable with `/plan`, `/tdd`, `/delegate` in the same agent session

### Why 2 sub-agents (not 3 with synthesizer)?
- DEBATE paper: 2-agent configuration already captures the key bias-reduction benefit
- Adding a third "synthesizer" agent adds complexity without proportional benefit
- The calling agent IS the synthesizer — it has full context on why it needed deliberation

### Why fixed roles (advocate/critic) not symmetric?
- CONSENSAGENT: symmetric "discuss both sides" produces sycophantic agreement
- Forced polarisation creates genuine tension that the synthesizer must resolve
- This matches the Devil's Advocate pattern from DEBATE framework

### Output: DELIBERATION.md
- Structured Markdown: Pro Arguments / Con Arguments / Synthesis / Decision
- Parallel to ADR format (MADR 4.0.0) but simpler — no formal status/consequences
- Written by the calling agent (synthesizer), not by sub-agents

## Implementation

### File created
- `src/tmux_orchestrator/agent_plugin/commands/deliberate.md`

### Logic flow
1. Parse `$ARGUMENTS` as the question
2. Load `__orchestrator_context__.json` for `agent_id`, `web_base_url`, `mailbox_dir`
3. Discover worker templates via `GET /agents`
4. Spawn advocate: `POST /agents` with `parent_id=my_id`, `template_id=workers[0]`
5. Poll inbox for `subagent_spawned` event → extract `advocate_id`
6. Spawn critic: repeat step 4
7. Poll inbox for second `subagent_spawned` event → extract `critic_id`
8. Brief advocate: `POST /agents/{advocate_id}/message` with advocate prompt
9. Brief critic: `POST /agents/{critic_id}/message` with critic prompt
10. Print `DELIBERATION.md` template and next-steps guide

### Key invariants
- Uses same API key resolution as all other commands (env → per-agent file → legacy file)
- Does NOT submit tasks via `POST /tasks` — uses P2P messaging so the sub-agents handle task completion themselves
- Does NOT wait for sub-agents to complete (would block the calling agent indefinitely)
- Guides the calling agent through the synthesis step explicitly
