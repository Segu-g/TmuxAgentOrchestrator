# v1.1.0 — POST /workflows/competition (Best-of-N Competitive Solver)

## 問題

多エージェントオーケストレーターが提供するワークフローテンプレートの中に「競争型（Best-of-N）」パターンが欠けていた。
CLAUDE.md にはデモシナリオとして「Competitive / best-of-N」が明示されており、AHC デモでは
3エージェントが同一問題を解いたが、スコア収集と勝者決定は手動だった。

## 解決策

`POST /workflows/competition` エンドポイントを追加し、以下の (N+1)-agent DAG を自動生成する：

```
solver_{strategy_0} ──┐
solver_{strategy_1} ──┼─→ judge
solver_{strategy_2} ──┘
```

- **solver_{strategy}**: 同一 `problem` を異なる戦略で解き、SCORE: 行を含む result.md を scratchpad に書く
- **judge**: 全 solver の結果を読み、スコアを比較し、WINNER: 行を含む COMPETITION_RESULT.md を書く

## 研究

### Query 1: "best-of-N multi-agent LLM competition parallel solver orchestrator pattern 2025"

- **MultiAgentBench** (arXiv:2503.01935, 2025): LLM マルチエージェントの協調と競争を評価する benchmark。
  競争シナリオでは「同一問題を複数エージェントが独立に解き、最良解を選択する」パターンを定義。
- **Multi-Agent Collaboration via Evolving Orchestration** (arXiv:2505.19591, OpenReview 2025):
  各層で複数 operator を並列実行する puppeteer-style orchestrator。競争型の並列フェーズに直接対応。
- **Learning Latency-Aware Orchestration for Parallel Multi-Agent Systems** (arXiv:2601.10560, 2025):
  並列マルチエージェントの低レイテンシ最適化。依存グラフ上の並列実行が最適。

### Query 2: "multi-agent competitive workflow LLM orchestration winner selection judge arXiv 2025"

- **Agent-as-a-Judge** (arXiv:2508.02994, 2025): LLM エージェントが他エージェント出力を評価する審判パターン。
  judge がツールを活用してスコアリングと評価根拠を生成する → competition judge ロールに直接適用。
- **Leveraging LLMs as Meta-Judges** (arXiv:2504.17087, 2025): 3段階メタ-judge パイプライン。
  単一 judge で十分だが、将来の拡張点として記録。
- **Difficulty-Aware Agent Orchestration** (arXiv:2509.11079, 2025): タスク難易度別エージェント割り当て。
  Best-of-N はシンプルなケースで最も有効と実証。

### Query 3: "best-of-N sampling LLM parallel agents judge scoring winner selection 2025"

- **Making, not Taking, the Best of N (FusioN)** (arXiv:2510.00931, 2025):
  BoN 選択 vs. N 候補統合（FusioN）の比較。本実装では「選択」ベース BoN を実装。
  list-wise judge（全候補を一覧渡し）が点数付け精度を最大化することを実証 → 本実装の judge プロンプト設計に採用。
- **M-A-P Parallel Test-Time Scaling** (arXiv:2506.12928, 2025):
  並列サンプリング（BoN, BoN-wise, Beam-Search, Tree search）の比較実験。
  list-wise 評価が最も精度が高い → judge が全候補をまとめて評価する設計を採用。
- **Statistical Estimation of Adversarial Risk under Best-of-N Sampling** (arXiv:2601.22636, 2025):
  候補数 N を増やすほど最終結果の信頼性が上がる → strategies リストを 2〜10 の範囲で許可する設計に反映。
