# v1.1.1 — `[Pasted text #1]` ハング修正 + Server Cleanup Helper

**選択日**: 2026-03-09

## 問題

### 1. `[Pasted text #1]` ハング

長いプロンプト（数百文字以上）を tmux ペインに送信すると、tmux が paste-preview モードに入り
ペイン上に `[Pasted text #1]` と表示される。この状態で続く Enter が内部バッファに吸収されてしまい、
Claude CLI がプロンプトを受け取れないまま永久にブロックされる。

**現在のコード (`tmux_interface.py`)**:

```python
def send_keys(self, pane: libtmux.Pane, text: str, enter: bool = True) -> None:
    pane.send_keys(text, enter=False)
    if enter:
        time.sleep(0.15)  # let paste-preview settle before Enter
        pane.send_keys("", enter=True)
```

`ensure_session()` では `set_option("assume-paste-time", "0")` を呼んでいる。

### 2. Server Cleanup の問題

デモスクリプトで `subprocess.Popen` + `proc.terminate()` パターンを使うと、
FastAPI/uvicorn プロセスが子プロセスを持つ場合にゾンビが残ることがある。
`os.killpg()` でプロセスグループ全体に SIGTERM を送る必要がある。

---

## 根本原因分析

### `assume-paste-time 0` が効かない可能性

`assume-paste-time` はセッション作成時に `set_option()` で設定している。この設定は
**新規セッション全体** に適用されるため、直後に作成されるウィンドウ・ペインには正しく伝播するはずである。

ただし調査の結果、以下の問題が浮上した:

1. **新規サブペイン (split_window) には `assume-paste-time` が継承されるか？**
   — tmux のセッションオプションはウィンドウ・ペインに継承される。`set -g` (global) 設定ではないが、
   セッション単位で設定しているため原則的に全ペインに有効。

2. **しかし `libtmux.Session.set_option` は `-g` なしで呼ぶため、セッションオプションのみ適用。**
   新しくスプリットしたペインが別セッションではないため、問題ないはずだが、
   `new_subpane()` で作成されるペインのウィンドウがどのセッションに属するかに依存する。

3. **より根本的な問題**: `assume-paste-time 0` がセッション作成後に適用されても、
   **その後に tmux send-keys で長いテキストを送ると OS の PTY バッファリングと
   tmux 内部の paste-detection が競合する**。実際には `assume-paste-time 0` が
   正しく設定されていても、libtmux の `pane.send_keys(text, enter=False)` が
   内部で大量のバイトを一括書き込みするため、tmux が paste として検出する
   可能性は排除できない。

4. **最終確認**: v1.1.0 デモログで `[Pasted text #1]` が実際に発生したことが確認されている。
   これは `assume-paste-time 0` だけでは不十分であることを示す。

### 検証済みの動作

- `time.sleep(0.15)` の後に空 Enter を送る現在の実装は、paste-preview が **既に表示されてしまった後** の対処療法に近い。
- `assume-paste-time 0` は tmux のキーバインド解釈をバイパスするが、Claude CLI 側の
  bracketed-paste 処理とは独立した問題である可能性がある。

---

## 修正方針

### Option A: pane 出力を検査して `[Pasted text` を検出したら Enter を追加送信 (採用)

`send_keys()` の後、短時間 (最大2秒) でペイン出力を監視し、
`[Pasted text` が含まれていれば自動的に Enter を追加送信する。

**利点**:
- 実際に発生したケースを確実に処理できる
- 他の修正 (assume-paste-time 0) が有効な場合は無害（フォールバックとして動作）
- 既存の sleep(0.15) + Enter パターンを拡張するだけ

**実装**:

```python
def send_keys(self, pane: libtmux.Pane, text: str, enter: bool = True) -> None:
    pane.send_keys(text, enter=False)
    if enter:
        time.sleep(0.15)  # let paste-preview settle
        # Check if paste-preview mode was triggered
        if _paste_preview_active(self.capture_pane(pane)):
            # tmux showed "[Pasted text #N]" — send Enter to confirm
            pane.send_keys("", enter=True)
            time.sleep(0.05)
        pane.send_keys("", enter=True)

def _paste_preview_active(text: str) -> bool:
    return "[Pasted text" in text
```

### Option B: テキストを行ごとに分割して送信 (補助として採用)

長いプロンプトが複数行の場合、行単位で送信する。ただしプロンプト内の改行は
Claude CLI の入力として意味を持つため、最終行の後のみ Enter を送る。

### Option C: `assume-paste-time 0` の設定確認の強化 (維持)

現在の実装を維持する（設定が正しく呼ばれていることを確認）。

## Web調査結果

### Query 1: "tmux assume-paste-time bracket paste mode [Pasted text terminal behavior 2024"

**key findings**:
- `assume-paste-time` はキーが 1ms より速く入力された場合、それをペースト入力とみなし、
  tmux キーバインドを処理しない。デフォルトは 1ms で、0 で無効化 (man page, Ubuntu manpages)。
- bracketed paste mode は tmux に 2012 年に追加。pasted text を `\033[200~` と `\033[201~` で
  囲む機能 (tmux/tmux commit f4fdddc)。
- `assume-paste-time` と bracketed paste mode は独立した機能。前者は tmux の内部 paste 検出、
  後者は端末エミュレータとアプリケーション間のプロトコル。

**Sources**:
- tmux man page: https://man7.org/linux/man-pages/man1/tmux.1.html
- tmux/tmux bracketed paste commit: https://github.com/tmux/tmux/commit/f4fdddc9306886e3ab5257f40003f6db83ac926b
- Bracketed paste mode bug: https://github.com/microsoft/terminal/issues/19418

### Query 2: "tmux send-keys long text paste preview mode enter keypress workaround"

**key findings**:
- `tmux send-keys` がセッション外から呼ばれると preview が表示される (tmux/tmux issue #467)。
- セッション内から呼ぶ場合は preview は表示されない。
- TmuxAgentOrchestrator は Python プロセスから libtmux 経由で `send-keys` を発行するため、
  セッション外からの操作とみなされる可能性がある。

**Sources**:
- tmux send-keys preview issue: https://github.com/tmux/tmux/issues/467
- libtmux send_keys issue: https://github.com/tmux-python/libtmux/issues/15

### Query 3: "tmux assume-paste-time option explanation paste detection milliseconds"

**key findings**:
- `assume-paste-time` は複数のキーストロークが高速で入力された場合に paste として検出する閾値。
  デフォルト 1ms、`0` で完全無効。
- `set -g assume-paste-time 0` を tmux.conf に追加するのが一般的な回避策。
- ただし Claude CLI は bracketed paste mode を独立して実装している可能性があり、
  この設定だけでは `[Pasted text #N]` を完全に防げない。

**Sources**:
- Ubuntu tmux manpage: https://manpages.ubuntu.com/manpages/xenial/man1/tmux.1.html
- tmux man7.org: https://man7.org/linux/man-pages/man1/tmux.1.html

## Server Cleanup Helper

### 問題

デモスクリプトの典型的なパターン:

```python
server_proc = subprocess.Popen(cmd, cwd=demo_dir, ...)
# ... demo runs ...
server_proc.terminate()
server_proc.wait(timeout=5)
```

`terminate()` は SIGTERM を直接 uvicorn プロセスに送るが、uvicorn が子プロセス
（worker プロセス）を起動している場合、子プロセスは SIGTERM を受け取らず残留する。

### 修正

`os.setsid()` で新しいプロセスグループを作成し、終了時に `os.killpg()` でグループ全体に
SIGTERM を送る。

```python
proc = subprocess.Popen(
    cmd, cwd=cwd,
    stdin=subprocess.DEVNULL,
    start_new_session=True,  # os.setsid() 相当
)
# Cleanup:
os.killpg(os.getpgid(proc.pid), signal.SIGTERM)
```
