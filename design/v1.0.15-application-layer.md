# v1.0.15 — application/ 層の抽出（クリーンアーキテクチャ）

## 目的

v1.0.11 で `domain/` (純粋ドメイン型) を抽出した。v1.0.13-14 で DI プロトコルを追加した。
本イテレーション (v1.0.15) では **application/** 層を抽出する。

## 層の定義

```
domain/          # 純粋型。外部依存ゼロ (AgentStatus, Task, Message...)
application/     # ユースケース・サービス。domain のみに依存。インフラ不可。
infrastructure/  # tmux, libtmux, subprocess, HTTP, filesystem
```

### application/ に属するもの

| 候補 | 判断 |
|------|------|
| `WorkflowService` — workflow.py の Workflow + _topological_sort | **移動対象** |
| `TaskSubmissionService` — submit_task のビジネスロジック | 今後 |
| `SupervisionService` — supervision.py の supervised_task | **移動対象** |
| `NullContextMonitor`, `NullDriftMonitor` — orchestrator.py 内 | **移動対象** |
| `ContextMonitorProtocol`, `DriftMonitorProtocol` — orchestrator.py 内 | **移動対象** |
| `Orchestrator` 本体 | 今後（大規模すぎる） |
| `AgentRegistry` | 今後（registry.py に既に分離済み） |

### application/ に属さないもの（今回は移動しない）

- `tmux_interface.py` — infrastructure (libtmux 依存)
- `agents/claude_code.py` — infrastructure (subprocess, libtmux)
- `bus.py` — infrastructure/framework (asyncio)
- `config.py` — infrastructure (YAML, filesystem)

## 今回の移動対象

### 1. `WorkflowStep` + `Workflow` + `_topological_sort` → `application/workflow_service.py`

**理由:**
- Kahn's algorithm (`_topological_sort`) はインフラ依存ゼロの純粋アルゴリズム
- `Workflow` は `Orchestrator.submit_task()` を呼ぶが、Protocol 経由にすれば application 層で完結
- `WorkflowStep` は domain 型 (`Task`) のみに依存

**実装方針:**
1. `application/workflow_service.py` に `TaskSubmitter` Protocol を定義
2. `Workflow` が `TaskSubmitter` に依存するよう変更（`Orchestrator` への直接依存を除去）
3. `workflow.py` は backward-compat shim として `application/workflow_service` を re-export

### 2. `supervised_task` → `application/supervision.py`

**理由:**
- `supervised_task` は asyncio + stdlib のみ。インフラ依存なし
- application 層のユースケース（dispatch loop の耐障害性）を実現するユーティリティ
- `supervision.py` は shim として re-export

### 3. Monitor Protocols + Null Objects → `application/monitor_protocols.py`

**理由:**
- `ContextMonitorProtocol`, `DriftMonitorProtocol` は DI インターフェース (PEP 544)
- `NullContextMonitor`, `NullDriftMonitor` は infrastructure を使わないテスト用実装
- 現在 `orchestrator.py` に混在しており、layer 汚染を起こしている

## Strangler Fig 移行計画

```
Phase 1 (v1.0.15 — 本イテレーション):
  application/workflow_service.py ← workflow.py の中身
  application/supervision.py      ← supervision.py の中身
  application/monitor_protocols.py ← orchestrator.py 内の Protocols + Null Objects

  workflow.py → shim (re-export from application.workflow_service)
  supervision.py → shim (re-export from application.supervision)
  orchestrator.py → import monitor_protocols from application/

Phase 2 (将来):
  application/task_submission.py ← submit_task ロジック
  application/dispatch_policy.py ← dispatch loop ポリシー

Phase 3 (将来):
  Orchestrator を application/ 配下に移動
```

## レビュー指摘

### 層割り当ての正確性

- `supervised_task` は asyncio のみ依存 → application 層 OK。
- `Workflow` / `WorkflowStep` / `_topological_sort` は domain.Task + Protocol のみ → application 層 OK。
- `ContextMonitorProtocol`, `NullContextMonitor` は typing + stdlib のみ → application 層 OK。
- `orchestrator.py` は infrastructure (tmux, libtmux) に依存するため application 層への移動は今回対象外 → 正しい判断。

### 循環依存チェック

- `application/` → `domain/` のみ。逆方向なし。
- `orchestrator.py` → `application/` は OK（infrastructure → application の依存）。
- `supervision.py` → `application/supervision.py` (shim) → 循環なし。
- `workflow.py` → `application/workflow_service.py` (shim) → 循環なし。

### インポート順序

- `orchestrator.py` のインポート: `application.monitor_protocols` を stdlib/domain 依存より先に追加。alphabetical 順では問題なし。
- shim ファイルの re-export: `# noqa: F401` で re-export フラグ付き → linter 対応 OK。

### 指摘事項

1. `__init__.py` の `APP_FILES` glob は `__init__.py` を除外しているが、`__init__.py` 自体も application imports を持つ。
   → purity テストは `__init__.py` を除く意図的な設計。`__init__.py` は application 内モジュールのみを import するため問題なし。

> 対応: `__init__.py` は `application/` 内部のシンボルのみを re-export しており、forbidden imports は含まない。テストは `APP_FILES` (non-underscore files) のみを検査する設計で意図的。`__init__.py` のチェックは将来追加可能。

2. `TaskSubmitter` Protocol の `**kwargs` は静的型チェックで脆弱。将来は明示的パラメータに置き換えを推奨。

> 対応: `**kwargs` は `Orchestrator.submit_task()` の多数のオプション引数を受け入れるための暫定措置。次イテレーションで Protocol をより厳密に定義する候補として `DESIGN.md §11` に追記予定。

3. `_topological_sort` はプライベート関数 (アンダースコアプレフィックス) だが `__all__` に含まれている。

> 対応: 既存の `test_workflow.py` が `_topological_sort` を直接テストしており、backward compat のために `__all__` に含める必要がある。将来は public API `topological_sort` に rename を検討。
