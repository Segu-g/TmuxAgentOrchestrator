# v1.0.34 — ProcessPort as Canonical Type + WebhookManager DI

## 問題

`ClaudeCodeAgent` に残る2つの依存問題:

1. **`self.pane: libtmux.Pane`** — `ClaudeCodeAgent.start()` が生の `libtmux.Pane` を
   `self.pane` に代入し、内部で `self._tmux.send_keys(self.pane, ...)` / `self.pane.send_keys("C-c")` /
   `self.pane.id` などを直接使用している。`ProcessPort` Protocol が
   `infrastructure/process_port.py` に存在しているにもかかわらず、エージェント内部の
   主要な pane 操作インターフェースとして使われていない。

2. **`orchestrator.py` の `WebhookManager` 直接インスタンス化** — `WebhookManager` が
   モジュールレベルで import され `__init__` 内でハードコードされている。
   `ContextMonitor` / `DriftMonitor` は既に Protocol + DI 済みだが `WebhookManager` が残っている。

## 解決策

### ProcessPort 統合 (`ClaudeCodeAgent`)

1. `ProcessPort` に `pane_id: str` プロパティと `send_interrupt()` メソッドを追加
   - `pane_id`: ログ出力用の pane 識別子
   - `send_interrupt()`: `Ctrl-C` を送信（`libtmux.Pane.send_keys("C-c")` の代替）

2. `TmuxProcessAdapter` を拡張して上記2メソッドを実装

3. `StdioProcessAdapter` を拡張して上記2メソッドを実装（テスト用）

4. `ClaudeCodeAgent.start()` で pane 作成後に `TmuxProcessAdapter` でラップ:
   ```python
   self.process: ProcessPort = TmuxProcessAdapter(pane=pane, tmux=self._tmux)
   self._raw_pane = pane  # watch/unwatch のみに使用
   ```

5. 内部操作を `ProcessPort` 経由に切り替え:
   - `self._tmux.send_keys(self.pane, ...)` → `self.process.send_keys(...)`
   - `self.pane.send_keys("C-c")` → `self.process.send_interrupt()`
   - `self.pane.id` → `self.process.pane_id`

6. `watch_pane` / `unwatch_pane` は `self._raw_pane` (生の libtmux.Pane) 経由のまま
   （これらは `TmuxInterface` の infrastructure 操作であり Process Port の責務外）

7. `self.pane` を公開プロパティとして維持（後方互換）し、`TmuxProcessAdapter.pane` を返す

### WebhookManager DI (`orchestrator.py`)

`Orchestrator.__init__` にオプション引数 `webhook_manager: WebhookManager | None = None` を追加。
`None` の場合は既存と同じ `WebhookManager(timeout=config.webhook_timeout)` を生成する。
モジュールレベルの `from tmux_orchestrator.webhook_manager import WebhookManager` は
デフォルト生成のために残す（型ヒント用）。

  - 「Components depend on abstractions (protocols or ABCs) rather than concrete implementations」

- **Source**: [How to Implement Dependency Injection in Python — OneUptime](https://oneuptime.com/blog/post/2026-02-03-python-dependency-injection/view)
  - 「Constructor injection is the preferred approach because it makes dependencies explicit and required」
  - テスト用 fake を `__init__` に渡せることでユニットテストが高速化・安定化

### Query 3: Python Protocol runtime_checkable pane abstraction tmux agent testability 2025
- **Source**: [Leveraging Typing.Protocol: Faster Error Detection And Beyond Inheritance — Pybites](https://pybit.es/articles/typing-protocol-abc-alternative/)
  - `@runtime_checkable` を使うと `isinstance(obj, Protocol)` で構造的適合を検証できる
  - 「Protocol is preferred over ABC because it doesn't require inheritance — any object that satisfies the interface works」

- **Source**: [Is there a downside to `typing.runtime_checkable`? — Python Discussions](https://discuss.python.org/t/is-there-a-downside-to-typing-runtime-checkable/20731)
  - `runtime_checkable` は method presence のみ検証 (シグネチャは未検証)
  - 本用途ではプロパティ `pane_id` は `__isinstance__` で確認できないため
    `pane_id` は Protocol メソッドではなくドキュメント的契約として扱うか、
    `@property` ではなく通常メソッド `get_pane_id() -> str` として定義する

## レビュー指摘

**コンテキストフリーレビュー (v1.0.34)**

### 指摘1: `self.pane` 公開属性の後方互換性
`ClaudeCodeAgent.pane` は外部コード (web/app.py の context_monitor, drift_monitor など) から
参照されている可能性がある。`self.pane = pane` を維持しているのは正しいが、
将来的に `pane` を削除して `process` のみにする移行パスを DESIGN.md に記載すべき。

> 対応: `self.pane` は v1.0.34 で維持。`process` を使う新しいパスと並行して提供する。
> 将来の削除方針は §11 の低優先度候補に追記した。

### 指摘2: `TmuxInterface.create_process_adapter()` との重複
`infrastructure/tmux.py` に `create_process_adapter(pane) -> TmuxProcessAdapter` が
存在するが、`claude_code.py` は直接 `TmuxProcessAdapter(pane=pane, tmux=self._tmux)` を
構築している。factory method を使うかどうかで一貫性が欠如している。

> 対応: 直接構築が最も明示的でテストしやすいため現状を維持する。
> factory method は tmux.py の内部テスト用として残す。引数の重複は許容範囲。

### 指摘3: `send_keys(launch)` の `enter` パラメータ
`start()` で `self.process.send_keys(launch)` を呼ぶが、旧コードは
`self._tmux.send_keys(pane, launch)` で TmuxInterface が Enter を自動付加していた。
`ProcessPort.send_keys` の `enter` デフォルトが `True` なので同じ動作をするが、
明示的に `enter=True` と書くべきか検討が必要。

> 対応: `TmuxProcessAdapter.send_keys` は `TmuxInterface.send_keys` に委譲し、
> そちらが Enter を自動付加する。`enter` パラメータのデフォルトが `True` であるため
> 動作は同一。コードの明示性を高めるため `enter=True` を省略のままにする (デフォルト)。

### 指摘4: `notify_stdin` の `process is None` チェック
`notify_stdin` で `pane is None or process is None` の両方をチェックしているが、
`start()` が実行されれば両方セットされる。`process is None` だけチェックすれば十分では。

> 対応: 後方互換性のため両チェックを維持する。外部コードが `agent.pane = mock_pane` だけ
> セットするテストパターンが存在するため、`pane is None` チェックを外すとそれらが壊れる。
