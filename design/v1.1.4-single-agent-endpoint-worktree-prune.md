# v1.1.4 — GET /agents/{id} + Stale Worktree Auto-Cleanup

## Problem

1. **Missing single-agent REST endpoint**: `GET /agents` returns all agents. There is no dedicated
   `GET /agents/{id}` endpoint to retrieve a single agent by ID. This is a gap in the REST API's
   resource model — demo scripts and external tooling must scan the full list and filter client-side.

2. **Stale worktree pain**: Build-log v1.0.23 recorded "stale worktree cleanup is a recurring pain".
   When a demo exits without proper cleanup (crash, timeout), `.worktrees/{agent_id}/` directories
   accumulate. On the next demo run, `git worktree add` fails because the directory already exists.
   Developers must manually run `git worktree prune` before each demo.

## Solution

### Part A: `GET /agents/{id}`

Add a dedicated single-agent endpoint following REST best practices (uniform resource interface,
sub-resource access via `/{id}` path parameter, 404 on missing resource).

- Reuses the existing `registry.list_all()` dict structure for consistency.
- Returns 404 with `{"detail": "Agent {id!r} not found"}` when the ID is unknown.
- Registered **before** `DELETE /agents/{agent_id}` to avoid path conflict with `GET /agents/tree`.
- Added to OpenAPI schema snapshot.

### Part B: `WorktreeManager.prune_stale()`

Add a `prune_stale()` method to `WorktreeManager` that runs `git worktree prune --expire now`.
Call it from `Orchestrator.start()` when a `WorktreeManager` is available, before spawning agents.

This removes stale administrative metadata for worktrees whose directories no longer exist,
allowing `git worktree add` to succeed even after an unclean prior shutdown.

## Research

### Query 1: REST API GET single resource by ID best practices 2025

**Source**: Stack Overflow Blog "Best practices for REST API design" (2020, still canonical in 2025)
- URL: https://stackoverflow.blog/2020/03/02/best-practices-for-rest-api-design/
- Pattern: `GET /collections/{id}` for single-resource retrieval; 404 on not-found.

**Source**: Microsoft Azure Architecture Center "Web API Design Best Practices"
- URL: https://learn.microsoft.com/en-us/azure/architecture/best-practices/api-design
- Confirms: Use `GET /resources/{id}` (not query params) for single-resource access.
  ETag/Cache-Control headers recommended for caching.

**Conclusion**: `GET /agents/{agent_id}` with 404 on not-found is the correct REST pattern.
The endpoint reuses the same dict shape as `list_all()` for consistency.

### Query 2: git worktree prune stale cleanup automation startup 2025

**Source**: Git official docs "git-worktree prune"
- URL: https://git-scm.com/docs/git-worktree
- `git worktree prune` removes stale administrative files. `--expire now` forces immediate cleanup.
- `gc.pruneWorktreesExpire` config controls automatic GC expiry (default 3 months).

**Source**: GitHub issue "Stale worktrees are never cleaned up" (anthropics/claude-code #26725)
- URL: https://github.com/anthropics/claude-code/issues/26725
- Confirms this is a well-known problem: no garbage collection on session restart.

**Source**: Haacked.com "tree-me: Because git worktrees shouldn't be a chore" (2025-11-21)
- URL: https://haacked.com/archive/2025/11/21/tree-me/
- Shows wrapper tools include automatic pruning; pattern: prune before creating new worktrees.

**Conclusion**: `git worktree prune --expire now` in `Orchestrator.start()` is the correct fix.

### Query 3: FastAPI single item endpoint 404 response model 2025

**Source**: FastAPI docs "Additional Responses in OpenAPI"
- URL: https://fastapi.tiangolo.com/advanced/additional-responses/
- Pattern: `responses={404: {"model": ErrorModel, "description": "..."}}` in route decorator.

**Source**: BetterStack "FastAPI Error Handling Patterns"

### レビュー: コードレビュー (context-free sub-agent review)

**指摘 1: `get_agent_dict()` は `list_all()` を全件呼ぶため O(n) スキャン**

`Orchestrator.get_agent_dict()` は `registry.list_all()` でエージェント全件を JSON dict にシリアライズしてから線形検索する。
エージェント数が多い場合（>100）、不要なシリアライズコストが生じる。
`registry.get(agent_id)` で Agent オブジェクトを取得してから単体シリアライズする方が効率的。

> 対応: 現時点のエージェント数 (<20) では無視できるコスト。
> ただし、`registry` に `get_agent_dict(agent_id)` メソッドを直接追加し、
> 単体シリアライズを行うことでこの問題を解決できる。これは v1.1.5 以降の改善候補として
> `DESIGN.md §11` に追加する。

**指摘 2: `prune_stale()` のエラーサイレンシングはデバッグを困難にする可能性**

`subprocess.run()` のエラーを完全に無視しているため、git が存在しない環境や
リポジトリではない場所で実行された場合も黙って失敗する。
ログ出力 (`logger.debug` または `logger.warning`) を追加すべき。

> 対応: 正当な指摘。`prune_stale()` にログ出力を追加した (DEBUG レベル)。
> 失敗しても orchestrator の起動を妨げないことは設計的に正しい（worktree cleanup は
> ベストエフォート）。

**指摘 3: `test_single_agent_endpoint.py` の `_MockOrchestrator` が `get_agent_dict` を持っているが、web/app.py の `get_agent` エンドポイントは orchestrator の `get_agent_dict` を呼ぶ**

テストの mock は `get_agent_dict` を実装しているが、実際の `Orchestrator` には
今回のコミットで `get_agent_dict` が追加された。既存の mock オーケストレーター
（`test_web_app.py` の `_MockOrchestrator` など）は `get_agent_dict` を持っていないため、
それらのテストで `GET /agents/{id}` が呼ばれた場合に `AttributeError` が生じる可能性。

> 対応: `web/app.py` の `get_agent` ハンドラーが `orchestrator.get_agent_dict()` を
> 呼ぶため、既存の mock が `get_agent_dict` を持たないと `AttributeError` になる。
> ただし既存テストは `GET /agents/{agent_id}` を呼ばないため実際の問題は起きていない。
> 安全策として `hasattr` ガードを追加することも検討したが、シンプルさを優先して
> `AttributeError` が出た場合は 500 エラーになる（テストが壊れることで検出可能）。
> 今回は問題なし。
