# Design: v1.0.24 — POST /workflows/redblue — Red Team / Blue Team 対抗評価ワークフロー

## 問題ステートメント

現在のワークフローエンドポイント群（`/workflows/tdd`, `/workflows/debate`, `/workflows/adr`, `/workflows/delphi`）は構造化された協調パターンを提供しているが、**対抗評価（adversarial evaluation）**パターンが欠けている。設計案・実装・アーキテクチャ選択に対して、攻撃者視点からの欠陥探索を自動化する Red-Blue チーム評価が存在しない。

セキュリティレビュー・アーキテクチャ評価・ビジネスケース検証において、意図的に反対の視点を持つエージェントを配置することで、単一エージェントや対称的討論では発見されにくい欠陥を体系的に検出できる。

## 提案する解決策

`POST /workflows/redblue` — 3エージェント構成の対抗評価ワークフロー:

1. **blue-team**: 設計案・実装・提案を作成する（構築側）
2. **red-team**: blue-teamの成果物を攻撃者視点から評価し、欠陥・脆弱性・弱点を列挙する（攻撃側）
3. **arbiter**: blue-teamの設計とred-teamの指摘を総合し、リスク評価レポートと推奨事項を生成する（評価側）

成果物はスクラッチパッド（Blackboardパターン）経由で受け渡し:
- `{prefix}_blue_design` — blue-teamの設計/実装案
- `{prefix}_red_findings` — red-teamの欠陥・リスク一覧
- `{prefix}_risk_report` — arbiterのリスク評価レポート

## 実装計画

### 1. Pydanticモデル (`RedBlueWorkflowSubmit`)
- `topic: str` — 評価対象（例: "FastAPI endpoint design for user authentication"）
- `blue_tags: list[str]` — blue-team required_tags
- `red_tags: list[str]` — red-team required_tags
- `arbiter_tags: list[str]` — arbiter required_tags
- `reply_to: str | None` — arbiter RESULT の宛先エージェント
- バリデーション: topic 非空

### 2. エンドポイント (`POST /workflows/redblue`)
- `/workflows/debate` と同じルーティング・認証パターン
- DAG: blue-team → red-team (blue依存) → arbiter (red依存)
- スクラッチパッドキー: `redblue_{uuid[:8]}_blue_design` / `_red_findings` / `_risk_report`
- 戻り値: `workflow_id`, `name`, `task_ids`, `scratchpad_prefix`

### 3. テスト (`tests/test_workflow_redblue.py`)
- `RedBlueWorkflowSubmit` バリデーション (空topic等)
- HTTP 200 レスポンスと workflow 構造
- タスク数 (3) と依存チェーン
- スクラッチパッドキー命名
- 認証 (401)
- `reply_to` 伝播
- タグ転送

## 調査結果

### Query 1: "red team blue team adversarial multi-agent LLM evaluation security review 2025"

主要知見:
- **Agent-in-the-Middle (AiTM) 攻撃**: LLM-MAS のエージェント間通信を傍受・改ざんする攻撃手法が確認されており、adversarial multi-agent 評価の重要性が高まっている。
- **自律型 Red-Blue チーム**: Farzulla (2025) が LLM-guided エージェントによる自律セキュリティテストを提案。攻撃エージェントが脆弱性を発見し、防御エージェントがより現実的なセキュリティテストを実施。
- **AIRTBench**: AI red teaming フレームワークが AI/ML CTF チャレンジで LLM の攻撃能力を評価。adversarial シナリオの証明の場として機能。

**References**:
- "Red-Teaming LLM Multi-Agent Systems via Communication Attacks", ACL Anthology 2025, arXiv:2502.14847, https://aclanthology.org/2025.findings-acl.349/
- Farzulla, "Autonomous Red Team and Blue Team AI", DISSENSUS AI Working Paper DAI-2513, 2025, https://farzulla.org/papers/Farzulla_2025_Autonomous_Red_Team.pdf

### Query 2: "adversarial multi-agent evaluation LLM bias reduction arXiv 2410.04663"

主要知見:
- **D3 (Debate, Deliberate, Decide)**: arXiv:2410.04663 が役割特化エージェント（advocate, judge, jury）による構造化討論フレームワークを提案。positional bias と verbosity bias を匿名化とロール多様化で削減。Human 判断との一致率（Cohen's kappa）でState-of-the-Art を達成。
- **MORE プロトコル**: N個の並列 advocate が各回答を擁護し、多様な視点で評価シグナルを増幅。
- **SAMRE プロトコル**: トークン予算と収束チェックを持つ単一 advocate の多ラウンド反復。

**References**:
- Harrasse, Bandi, Bandi, "Debate, Deliberate, Decide (D3): A Cost-Aware Adversarial Framework for Reliable and Interpretable LLM Evaluation", arXiv:2410.04663, 2026-01, https://arxiv.org/abs/2410.04663

### Query 3: "red team blue team arbiter LLM multi-agent security design review pattern workflow 2025"

主要知見:
- **連続的 Red Teaming**: 2025年トレンドとして自動化された敵対的テストが定期実行される継続的 Red Teaming に移行。
- **攻撃→防御→評価サイクル**: 攻撃エージェントが戦術/技法で欠陥を探し、防御エージェントがそこから防御策を学習するサイクルが有効。本提案の blue→red→arbiter 構造はこのサイクルの静的版として実装できる。
- **セキュリティ評価の自動化**: LLM が pen testing, CTF, vulnerability discovery に活用されており、構造化されたレポート生成（arbiter 相当）が重要。

**References**:
- "From Promise to Peril: Rethinking Cybersecurity Red and Blue Teaming in the Age of LLMs", arXiv:2506.13434, 2025, https://arxiv.org/html/2506.13434v1
- "Algorithmic red teaming approaches to secure LLMs", ScienceDirect, 2025, https://www.sciencedirect.com/article/pii/S2666827025001987

## テスト計画

1. `RedBlueWorkflowSubmit` バリデーション
   - 空 topic → 422
   - 空白のみ topic → 422
   - 正常なリクエスト → 200
2. DAG 構造
   - タスク数 = 3 (blue / red / arbiter)
   - red-team が blue-team に依存
   - arbiter が red-team に依存
3. スクラッチパッドキー命名規則
   - `redblue_{prefix}_blue_design`, `_red_findings`, `_risk_report`
4. 認証
   - API キーなし → 401
   - 不正なキー → 401
5. reply_to 伝播
   - `reply_to="director"` → arbiter タスクの `reply_to` に伝播
6. タグルーティング
   - `blue_tags`, `red_tags`, `arbiter_tags` が各タスクの `required_tags` に設定
