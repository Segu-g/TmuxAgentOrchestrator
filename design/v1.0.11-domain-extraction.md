# Design: v1.0.11 — domain/ 純粋型の抽出

## 概要

`tmux_orchestrator/domain/` サブパッケージを新設し、現在複数のモジュールに分散している
純粋なドメイン型を集約する。`domain/` は Python 標準ライブラリのみを使用し、外部依存をゼロに保つ。
既存の import パスは後方互換シム（re-export）で維持する。

---

## 移動する型と移動元

| 型 | 現在の場所 | 移動先 |
|----|-----------|--------|
| `AgentStatus` | `agents/base.py` | `domain/agent.py` |
| `AgentRole` | `config.py` | `domain/agent.py` |
| `Task` | `agents/base.py` | `domain/task.py` |
| `MessageType` | `bus.py` | `domain/message.py` |
| `Message` | `bus.py` | `domain/message.py` |
| `BROADCAST` | `bus.py` | `domain/message.py` |

---

## 新ファイル構造

```
src/tmux_orchestrator/
└── domain/
    ├── __init__.py        # package marker; re-exports all public symbols
    ├── agent.py           # AgentStatus, AgentRole
    ├── task.py            # Task dataclass
    └── message.py         # MessageType, Message, BROADCAST
```

### domain/__init__.py

全 public シンボルを re-export する。外部コードが `from tmux_orchestrator.domain import AgentStatus` と書けるようにする。

### domain/agent.py

`AgentStatus(str, Enum)` と `AgentRole(str, Enum)` のみを定義。
import: `from __future__ import annotations`, `from enum import Enum` のみ。

### domain/task.py

`Task` dataclass を定義。
import: `from __future__ import annotations`, `dataclasses`, `secrets`, `time`, `typing` のみ。
※ `Any` は `typing.Any`（stdlib）で OK。

### domain/message.py

`MessageType(str, Enum)`, `Message` dataclass, `BROADCAST = "*"` を定義。
import: `from __future__ import annotations`, `dataclasses`, `datetime`, `enum`, `uuid` のみ。

## シム戦略（後方互換 re-export）

既存モジュールに以下のコードを追加する（既存のクラス定義を削除し、re-export に置き換える）:

**`agents/base.py`** (AgentStatus, Task を削除 → re-export):
```python
# Backward-compat shims — do not remove
from tmux_orchestrator.domain.agent import AgentStatus  # noqa: F401
from tmux_orchestrator.domain.task import Task  # noqa: F401
```

**`config.py`** (AgentRole を削除 → re-export):
```python
from tmux_orchestrator.domain.agent import AgentRole  # noqa: F401
```

**`bus.py`** (MessageType, Message, BROADCAST を削除 → re-export):
```python
from tmux_orchestrator.domain.message import BROADCAST, Message, MessageType  # noqa: F401
```

これにより、`from tmux_orchestrator.agents.base import AgentStatus` や
`from tmux_orchestrator.bus import MessageType` などの既存 import が引き続き動作する。

## "no external deps" 不変条件の強制方法

`tests/test_domain_purity.py` に新テストを追加し、`domain/` の各モジュールが
外部ライブラリを import していないことを検証する:

```python
import importlib, sys

DOMAIN_MODULES = [
    "tmux_orchestrator.domain.agent",
    "tmux_orchestrator.domain.task",
    "tmux_orchestrator.domain.message",
]
STDLIB_MODULES = sys.stdlib_module_names  # Python 3.10+

def test_domain_no_third_party_imports():
    for mod_name in DOMAIN_MODULES:
        mod = importlib.import_module(mod_name)
        for dep in vars(mod).get("__dependencies__", []):
            top = dep.split(".")[0]
            assert top in STDLIB_MODULES, f"{mod_name} imports non-stdlib: {dep}"
```

さらに、import グラフを静的解析する方法として `ast.parse` で各ファイルの `import` 文を
抽出し、全依存が stdlib かチェックするアプローチも併用する。

## 考慮事項

### 循環 import リスク

`domain/task.py` の `Task` は現在 `agents/base.py` 内で `from tmux_orchestrator.bus import Message, MessageType` を使う `Agent` クラスの隣に定義されているが、`Task` 自体は `bus.py` を import していない。`Task` だけを `domain/task.py` に切り出せば循環 import は発生しない。

### `Task.__lt__` の比較演算子

`Task` の `__lt__` は `asyncio.PriorityQueue` で使用される。この実装は stdlib のみで書かれており問題なし。

### `Task.submitted_at` と `time.time()`

`time` は stdlib モジュール。問題なし。

### `Message.timestamp` と `datetime`

`datetime` は stdlib モジュール。問題なし。

## レビュー指摘

### 指摘 1: `agents/base.py` の循環 import リスク（重大）

`agents/base.py` は現在 `from tmux_orchestrator.bus import Message, MessageType` を top-level import している。
`bus.py` の `Message` と `MessageType` を `domain/message.py` に移動した後、`bus.py` で re-export すれば、
`agents/base.py` → `bus.py` → `domain/message.py` という import チェーンになり循環は発生しない。
ただし `domain/message.py` が `bus.py` を import しないこと、また `domain/agent.py` が `bus.py` を import しないことを必ず確認すること。

> 対応: `domain/` 内のファイルは相互に import しない設計とし、`domain/task.py` は `domain/message.py` を import せず、`domain/agent.py` も同様。`agents/base.py` は引き続き `from tmux_orchestrator.bus import Message, MessageType` を使用する（bus.py がシムとして re-export するため変更不要）。これにより循環 import は発生しない。

### 指摘 2: `__dependencies__` ベースのテストは実効性が低い（軽微）

設計文書中の `test_domain_no_third_party_imports` は `vars(mod).get("__dependencies__", [])` を参照しているが、
`__dependencies__` という属性は Python の慣例的な属性名ではなく、実際のモジュールに定義されないため、テストが何も検証しない。

> 対応: `ast.parse` を使った静的解析アプローチを採用する。各 `domain/*.py` ファイルを AST で解析し、`import X` または `from X import Y` の `X` が stdlib に含まれるかチェックする。`sys.stdlib_module_names` (Python 3.10+) を使う。

### 指摘 3: `domain/task.py` は `typing.Any` を使用するが `Any` の型ヒントが `dict[str, Any]` 形式

`Task.metadata: dict[str, Any]` の型ヒントで `Any` を使用する。`typing` は stdlib なので問題ないが、
`from __future__ import annotations` と `from typing import Any` の両方が必要なことを明記すること。

> 対応: `domain/task.py` の import リストに `from typing import Any` を明示的に含める。設計文書に記載済み。

### 指摘 4: シム re-export の `__all__` 更新が必要な場合がある

もし `agents/base.py`, `config.py`, `bus.py` が `__all__` を定義していた場合、
型を削除してシムに変えたあと `__all__` が古い参照を持ったままになる可能性がある。

> 対応: 実装前に各ファイルの `__all__` 定義の有無を確認し、存在しなければ何もしない（Python はデフォルトで public シンボルを全て export する）。実際に確認したところ `agents/base.py`, `config.py`, `bus.py` はいずれも `__all__` を定義していない。問題なし。

### 指摘 5: `domain/` の `__init__.py` の re-export が `domain/` 自身を import することで起きる重複 import の可能性

`domain/__init__.py` が `from .agent import AgentStatus, AgentRole` 等を行うと、Python の import システムは
`domain/__init__.py` の import 時に `domain.agent` も import する。これは正常動作だが、`domain/__init__.py` の
import が `domain/agent.py` より先に完了しようとした場合に partial initialization エラーが起きる可能性がある。

> 対応: `domain/__init__.py` では各サブモジュールからの re-export を直接行う（lazy import は使わない）。Python の import システムはサブモジュールを先に完全に初期化してから `__init__.py` の次の行を実行するため、問題は発生しない。これは通常の Python パッケージパターンであり安全。
