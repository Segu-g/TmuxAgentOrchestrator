# Design: v1.0.20 — REST API 充実 + worktree_path フィールド追加

## 選択理由

`GET /agents` および `GET /agents/{id}` レスポンスに `worktree_path` が含まれていないため、デモや運用スクリプトがエージェントのワークツリーを特定するには REST API 以外の手段（コンテキストファイルを読む等）に依存しなければならない。API から直接 `worktree_path` を参照できれば E2E 検証が格段に簡潔になる。また `GET /agents/{id}/stats` は文脈使用量（pane_chars 等）のみ返しており、タスク履歴サマリーや稼働時間を含めることで、外部監視ツールが単一エンドポイントで充足する。

## Research

### 1. Additive Fields と後方互換性

**出典**: Zalando RESTful API Guidelines "Compatibility" (https://github.com/zalando/restful-api-guidelines/blob/main/chapters/compatibility.adoc)

- JSON レスポンスへの新フィールド追加はクライアントがそれを無視すれば「後方互換」変更である。
- 必須フィールドを追加してはならないが、省略可能フィールドの追加はセーフ。
- 今回の `worktree_path: str | None` は省略可能かつ None 許容のため完全に後方互換。

**出典**: Zuplo Learning Center "API Backwards Compatibility Best Practices" (https://zuplo.com/learning-center/api-versioning-backward-compatibility-best-practices, 2025)

- 「Prefer additive change」— 既存フィールドを変えず新フィールドを追加する戦略が推奨。
- Deprecation は6〜12ヶ月前通知・`Deprecation:` ヘッダが標準。今回は追加のみなので不要。

### 2. Sparse Fieldsets / Field Selection

**出典**: OneUptime Blog "How to Implement API Field Selection" (https://oneuptime.com/blog/post/2026-01-30-api-field-selection/view, 2026)

- `?fields=id,status,worktree_path` のような パラメータで返却フィールドを絞る手法。
- 今回の規模（1エージェントに数十フィールドを追加するわけではない）では、まず単純な追加から始め、クライアント需要が高まれば sparse fieldsets を後から実装するのが合理的。
- セキュリティ: ホワイトリストで返却可能フィールドを明示する。`worktree_path` はローカルFS パスであり、同一ホスト内運用が前提のためセキュリティリスクは低い。

### 3. HATEOAS とリソース充実

**出典**: pradeepl.com "HATEOAS - Building Self-Documenting REST APIs That Scale" (https://pradeepl.com/blog/rest/hateoas/, 2025)

- HATEOAS では、クライアントがリソースの現在状態から次のアクションを発見できるようにリンクを埋め込む。
- `worktree_path` はリソースの「状態」（ファイルシステム上の位置）を表し、クライアントが直接参照できるようにするのは HATEOAS の精神に沿う。
- ただし今回は純粋な HATEOAS リンクではなくパス文字列として返すのが適切（tmux/ファイルシステム操作向け）。

## 現在のエージェント応答スキーマ

`registry.py` `list_all()` が返すフィールド（全エンドポイント共有）:

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | `str` | エージェントID |
| `status` | `str` | IDLE/BUSY/STOPPED/ERROR/DRAINING |
| `current_task` | `str\|null` | 実行中タスクID |
| `role` | `str` | WORKER/DIRECTOR |
| `parent_id` | `str\|null` | 親エージェントID |
| `tags` | `list[str]` | capabilityタグ |
| `bus_drops` | `int` | バスメッセージドロップ数 |
| `circuit_breaker` | `str\|null` | closed/open/half_open |

**不足しているフィールド**:
- `worktree_path`: エージェントのワークツリーパス（Agent.worktree_path）
- `started_at`: エージェント起動時刻（あれば uptime 計算が可能）

## 提案する変更

### Goal 1: `worktree_path` を list_all() に追加

`registry.py` の `list_all()` に `"worktree_path": str(a.worktree_path) if a.worktree_path else None` を追加する。

`Agent.worktree_path` は `base.py` で `Path | None` として定義済み（l.56）。
`ClaudeCodeAgent._setup_worktree()` が `self.worktree_path = path` を設定する（l.218）。

### Goal 2: `GET /agents/{id}/stats` の充実

現在の stats（context_monitor 由来）:
- `agent_id`, `pane_chars`, `estimated_tokens`, `context_window_tokens`, `context_pct`, `warn_threshold_pct`

追加するフィールド:
- `task_count`: エージェントの完了タスク数（履歴から）
- `error_count`: エラー終了タスク数（履歴から）
- `worktree_path`: ワークツリーパス（registry から引く）
- `status`: 現在のエージェント状態

`web/app.py` の `agent_context_stats` エンドポイントで `orchestrator.get_agent(agent_id)` を使い追加フィールドを注入する。context_monitor が `None` を返す場合でも最低限のフィールドを返す方が有用なため、404 を返す条件を緩める（agent が存在していれば skeleton を返す）。

### Goal 3: OpenAPI スナップショット更新

エンドポイント変更後: `UPDATE_SNAPSHOTS=1 uv run pytest tests/test_openapi_schema.py`

## 後方互換性

- 既存フィールドは変更しない。
- `worktree_path: null` は isolate=false の場合または未起動エージェントの場合。
- `task_count`, `error_count` は stats エンドポイントへの追加のみ。
- 外部クライアントが追加フィールドを無視してもデグレードしない。

## テスト計画

1. `test_registry.py` または新 `test_api_worktree_path.py`:
   - `list_all()` に `worktree_path` が含まれること
   - worktree_path が Path → str に変換されること
   - worktree_path が None のエージェントでは null を返すこと
2. `test_openapi_schema.py` スナップショット更新
3. stats エンドポイントの追加フィールドテスト

## レビュー指摘

### 指摘 1: stats エンドポイントが 404 から 200 に変わることで既存テストが壊れる可能性

> 対応: `tests/test_context_monitor.py` の `_make_mock_orch_with_stats` を更新し、
> `get_agent()` が既知の agent_id に対して mock agent を返すよう修正した。
> 既知エージェントでも context_monitor が未ポーリングの場合はスケルトン（200）を返す。
> 未知エージェントは引き続き 404 を返す（`get_agent() is None` の場合）。
> 404 テストも既知/未知の使い分けを明確にした。

### 指摘 2: `worktree_path` が Path オブジェクトのまま JSON シリアライズされると TypeError

> 対応: `registry.list_all()` で `str(a.worktree_path)` に変換している。
> `web/app.py` の stats エンドポイントでも `str(agent.worktree_path)` に変換。
> テスト `test_list_all_worktree_path_is_string_not_path` で文字列型を検証。

### 指摘 3: `started_at` / uptime_s フィールドが今回含まれていない

> 対応: `Agent` 基底クラスに `started_at` フィールドがなく、追加コストが高いため
> 今回は見送り。`task_count`・`error_count`・`status`・`worktree_path` の 4 フィールドを
> 優先した。uptime_s は §11 の次候補として記録する。
