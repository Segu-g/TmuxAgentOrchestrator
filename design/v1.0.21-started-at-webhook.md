# v1.0.21 設計資料 — started_at / uptime_s + Webhook コールバック

## 選択理由

- `started_at` / `uptime_s`: エージェントの稼働時間を API で可視化する最小コスト変更。
  依存なし、実装範囲が限定的でリスクが低い。
- Webhook コールバック: `WebhookManager` と `KNOWN_EVENTS["agent_status"]` がすでに存在する。
  `agent_status` を実際に deliver するワイヤーのみ追加すれば完成する。
  既存インフラの活用で両機能を同一イテレーションで完成させることが可能。

---

## Part A: started_at / uptime_s

### 設計方針

- `Agent` base class に `started_at: datetime | None = None` フィールドを追加。
- `start()` の具体的な実装（`ClaudeCodeAgent`）の先頭で `self.started_at = datetime.now(UTC)` を記録。
  抽象 `start()` に直接書く方法もあるが、抽象メソッドは実装を持てないため、
  具体実装でセットするか、またはテンプレートメソッドパターン（`_record_start_time()`）を使う。
  -> テンプレートメソッドとして `_record_start_time()` を base class に追加し、
     `ClaudeCodeAgent.start()` の先頭から呼ぶ。こうすれば他の Agent 実装も同じ契約に従える。
- `uptime_s` は計算フィールド（永続化不要）: `(datetime.now(UTC) - started_at).total_seconds()`。
  `started_at` が None なら `uptime_s` も None。

### 変更箇所

| ファイル | 変更内容 |
|---|---|
| `agents/base.py` | `started_at: datetime | None = None` 追加、`_record_start_time()` ヘルパー追加 |
| `agents/claude_code.py` | `start()` 冒頭で `self._record_start_time()` 呼び出し |
| `registry.py` | `list_all()` に `started_at`、`uptime_s` を追加 |
| `web/app.py` | `GET /agents/{id}/stats` の enrichment に `started_at`、`uptime_s` を追加 |
| `tests/fixtures/openapi_schema.json` | UPDATE_SNAPSHOTS=1 で再生成 |

### API レスポンス例

```json
{
  "id": "worker-1",
  "status": "IDLE",
  "started_at": "2026-03-09T12:00:00+00:00",
  "uptime_s": 42.7,
  ...
}
```

## Part B: Webhook コールバック (agent.status_changed)

### 既存インフラ

- `WebhookManager` (`webhook_manager.py`) が実装済み。
- `KNOWN_EVENTS` に `"agent_status"` が含まれている。
- `orchestrator.py` が `_webhook_manager` を保持している。
- `_publish_status_event` が `AgentStatus` 変化時に bus に STATUS を投げる。

### 差分

`agent_status` イベントを `WebhookManager.deliver()` で実際に配信するワイヤーを追加するだけ。

配信タイミング:
- IDLE → BUSY: `_run_loop` の `_set_busy` 呼び出し後
- BUSY → IDLE: `_set_idle` 後
- ERROR: `_run_loop` の except 節後

配信場所の候補:
1. `agents/base.py` の `_publish_status_event` 内 — agent が webhook_manager を知る必要がある
2. `orchestrator.py` の STATUS メッセージ受信時 — 既存の STATUS ハンドラに追加

-> **Option 2 を採用**: Orchestrator が STATUS メッセージを bus から受信して処理するハンドラ
  (`_handle_bus_message` または類似) に webhook deliver を追加。
  これにより `Agent` base class は webhook_manager を知らなくて済む（依存方向が正しい）。

### 変更箇所

| ファイル | 変更内容 |
|---|---|
| `orchestrator.py` | STATUS メッセージ処理箇所で `_webhook_manager.deliver("agent_status", {...})` を追加 |
| `config.py` | `webhooks: list[WebhookConfig]` フィールド追加 (`WebhookConfig`: `url`, `events`, `secret`) |
| `factory.py` | `webhooks` 設定から `WebhookManager.register()` を startup 時に呼ぶ |
| `tests/test_webhook_status.py` | 新規テスト |

### WebhookConfig データクラス

```python
@dataclass
class WebhookConfig:
    url: str
    events: list[str] = field(default_factory=list)
    secret: str | None = None
```

YAML 例:
```yaml
webhooks:
  - url: http://localhost:9999/hook
    events: [agent_status, task_complete]
    secret: mysecret
```

### 配信ペイロード

```json
{
  "event": "agent_status",
  "timestamp": "2026-03-09T12:00:00+00:00",
  "data": {
    "agent_id": "worker-1",
    "status": "BUSY",
    "prev_status": "IDLE",
    "task_id": "abc123"
  }
}
```

## 調査・参考文献

### Webhook ベストプラクティス

1. **at-least-once vs at-most-once**:
   - Industry standard は at-least-once。受信側はべき等性を保証する。
   - 本実装はファイアアンドフォーゲット（失敗=警告のみ、リトライなし）= at-most-once。
   - 理由: シンプルな orchestrator 向けには at-most-once で十分。Enterprise グレードの
     at-least-once（ジャーナル＋リトライ）は future iteration に回す。
   - Hookdeck "Webhooks at Scale" https://hookdeck.com/blog/webhooks-at-scale
   - Medium "Webhooks done right" https://medium.com/prospa-technology/webhooks-done-right-676d4e74578a

2. **タイムスタンプ形式**:
   - ISO 8601 UTC が REST API の業界標準。
   - GitHub / Stripe / Azure Storage すべて ISO 8601 Z suffix を採用。
   - 参照: OpenStack API-WG https://specs.openstack.org/openstack/api-wg/guidelines/time.html
   - 参照: nickb.dev "Unix time vs ISO-8601" https://nickb.dev/blog/designing-a-rest-api-unix-time-vs-iso-8601/

3. **asyncio fire-and-forget**:
   - `asyncio.create_task()` で background task を作成し、タスク参照を集合で保持して
     done callback で削除するパターン（CPython issue #104091 参照）。
   - 既存 `WebhookManager._send()` はこれを実装済み。
   - Python asyncio docs: https://docs.python.org/3/library/asyncio-task.html
   - DEV Community "Mastering Python Async Patterns 2026": https://dev.to/shehzan/mastering-python-async-patterns-a-complete-guide-to-asyncio-in-2026-10o6

## レビュー指摘

### 設計レビュー（自己レビュー）

**指摘1: `started_at` は UTC タイムゾーン付きで記録されるか確認**
> 対応: `_record_start_time()` で `datetime.now(tz=timezone.utc)` を使用しており、
> 常に UTC タイムゾーン付き datetime が記録される。テスト `test_record_start_time_sets_utc_datetime` で確認。

**指摘2: `uptime_s` は property で計算されるが、list_all() 呼び出し時に計算されるため時刻がズレる可能性**
> 対応: list_all() が呼ばれた瞬間の `agent.uptime_s` を記録する。これは仕様通り
> （スナップショット時点の uptime）。継続的な精度が必要なら SSE ストリームを使う
> べきだが、REST API の single-shot query としては適切。

**指摘3: Webhook の agent_status 配信は STATUS メッセージ全体を broadcast=True で受信するため
他の STATUS イベント（context_warning 等）との混在に注意**
> 対応: `_route_loop` 内で `event_name in {"agent_busy", "agent_idle", "agent_error"}` で
> 厳密にフィルタリングしているため、他のイベントには影響しない。

**指摘4: static webhook の secret は YAML に平文記載になる点のセキュリティ考慮**
> 対応: 既存の `api_key` フィールドも YAML 平文なので一貫性あり。
> 本番環境では環境変数展開や vault integration が望ましいが、v1.0.21 スコープ外。
> DESIGN.md §11 に「YAML secret を環境変数から読む」タスクを追加候補として記録。
