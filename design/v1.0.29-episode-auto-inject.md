# v1.0.29 Design: Episode Auto-Record + Auto-Inject on Task Dispatch

## Problem Statement

v1.0.28 implemented the `EpisodeStore` REST API (`GET/POST/DELETE /agents/{id}/memory`) with JSONL persistence. However, agents must manually call `POST /agents/{id}/memory` after each task and `GET /agents/{id}/memory` before each task to benefit from episodic memory. This is error-prone and requires agents to remember to do so.

**Goal**: Automate both directions:
1. **Auto-record**: When `POST /agents/{id}/task-complete` is called, automatically save an episode to `EpisodeStore` with the task summary.
2. **Auto-inject**: When the orchestrator dispatches a task to an agent (via `send_keys`), prepend the last N episodes to the prompt so the agent has context from prior tasks.

## Proposed Solution

### Configuration
Add to `OrchestratorConfig` (in `config.py`):
```python
memory_auto_record: bool = True     # auto-save episode on task-complete
memory_inject_count: int = 5        # N episodes to inject on dispatch (0 = disabled)
```

### Auto-Record (task-complete → EpisodeStore)
In `web/app.py`, `agent_task_complete` endpoint:
- After updating agent status to IDLE
- If `memory_auto_record` is True and config has an `episode_store`
- Create an `EpisodeCreate(summary=output, outcome="success", lessons="")` and append to store

### Auto-Inject (dispatch → prompt prefix)
In `agents/claude_code.py`, `_dispatch` method:
- Before `send_keys(prompt)`
- If `memory_inject_count > 0` and `episode_store` is available
- Fetch last N episodes from store
- Prepend formatted section to prompt:
  ```
  ## 過去のタスク経験 (直近{N}件)

  1. [2026-03-09T10:00:00] summary: "implemented X" | outcome: success
  2. [2026-03-09T09:00:00] summary: "reviewed Y" | outcome: success

  ---

  {original_prompt}
  ```

### EpisodeStore access from dispatch
The `episode_store` needs to be passed to `ClaudeCodeAgent`. Currently `Orchestrator` holds the dispatch logic. We need to thread `episode_store` + `memory_inject_count` through to the agent.

**Implementation path**:
1. Add `memory_auto_record: bool` and `memory_inject_count: int` to `OrchestratorConfig`
2. Pass `episode_store` and `memory_inject_count` to `Orchestrator.__init__`
3. In `Orchestrator._dispatch_task`, inject episodes into prompt before calling `agent.run(task)`
4. In `web/app.py` `agent_task_complete`, auto-record episode to `episode_store`

This keeps `ClaudeCodeAgent` unaware of `EpisodeStore` — the orchestrator layer handles injection.

## Research

### Query 1: episodic memory auto-injection LLM agent task dispatch prompt engineering 2025

**Key findings**:
- **PlugMem** (arXiv:2603.03296, 2025): Task-agnostic plugin memory module for LLM agents. Stores episodic memory as structured logs; injects relevant memories into prompt at task start. Shows 23% improvement over no-memory baselines.
- **A-MEM** (arXiv:2502.12110, 2025): Agentic memory system that dynamically organizes memories via Zettelkasten method. Each new memory generates structured attributes; injection uses keyword-based retrieval.
- Session summaries automatically injected into orchestration prompt templates become part of agent's system instructions — enabling behavior to evolve based on accumulated context.

**References**:
- "PlugMem: A Task-Agnostic Plugin Memory Module for LLM Agents", arXiv:2603.03296, 2025. https://arxiv.org/html/2603.03296
- "A-MEM: Agentic Memory for LLM Agents", arXiv:2502.12110, 2025. https://arxiv.org/abs/2502.12110

### Query 2: MIRIX episodic memory automatic recording task completion hook 2025

**Key findings**:
- **MIRIX** (Wang & Chen, arXiv:2507.07957): Episodic memory entries include `{event_type, summary, details, actor, timestamp}`. Active Retrieval mechanism: autonomously infers topic on user input → retrieves from all memory components → tags retrieved data for contextual injection into system prompt.
- Memory updates triggered automatically after task batches (not manual). The "Memory Manager" agent decides what to store; injection is automatic.
- 35% accuracy improvement over RAG baselines on ScreenshotVQA.

**References**:
- Wang & Chen, "MIRIX: Multi-Agent Memory System for LLM-Based Agents", arXiv:2507.07957, July 2025. https://arxiv.org/abs/2507.07957
- MIRIX Documentation, Multi-Agent System. https://docs.mirix.io/architecture/multi-agent-system/

### Query 3: LLM agent persistent memory cross-task automatic injection context engineering 2025

**Key findings**:
- **Design Patterns for Long-Term Memory in LLM-Powered Architectures** (Serokell Blog, 2025): "Session summaries are automatically injected into orchestration prompt templates, becoming part of the agent's system instructions in subsequent sessions" — canonical pattern for cross-task memory.
- **Memory Management for Long-Running Low-Code Agents** (arXiv:2509.25250): Contextual consistency via automatic memory injection prevents context drift over long task sequences.
- Static heuristic memory pipelines are a known limitation; automatic hooks tied to task lifecycle events solve this.

**References**:
- "Design Patterns for Long-Term Memory in LLM-Powered Architectures", Serokell Blog, 2025. https://serokell.io/blog/design-patterns-for-long-term-memory-in-llm-powered-architectures
- "Memory Management and Contextual Consistency for Long-Running Low-Code Agents", arXiv:2509.25250, 2025. https://arxiv.org/pdf/2509.25250

## レビュー指摘

**レビュー結果 (v1.0.29 post-implementation)**:

1. `dataclasses.replace(task, ...)` は既存フィールドを全て保持するので安全。ただし `submitted_at` と `trace_id` も保持されるため、注入後タスクが再エンキューされた場合に重複 trace_id が発生しうる。
   > 対応: 現実装では注入はディスパッチ直前の一回のみ (re-queue されない)。重複 trace_id は問題にならない。

2. 注入エラー (`BLE001` の silent catch) がデバッグを困難にする可能性がある。
   > 対応: `logger.warning` で記録しており、デバッグ可能。Silent fail はサービス継続性のため意図的。

3. `task._task_started_prompt[task.id]` は注入前の原文プロンプトを保存 (dispatch前に記録)。注入後のプロンプトが履歴に残らないが、これは意図的か？
   > 対応: `_task_started_prompt` は元プロンプトを保存する設計で正しい。エピソードインジェクションは一時的なコンテキスト付与であり、履歴には元プロンプトのみ保存が適切。

4. `output[:500]` の 500 文字上限はなぜ 500 か？
   > 対応: エピソードを compact に保つためのヒューリスティック。JSONL ファイルサイズと可読性のバランス。設定可能にすることも検討したが、シンプルさを優先。
