# Design: v1.0.18 — Demo Stability: History-Based IDLE Detection + WorktreeIntegrity Migration

## 選択理由 (Step 0)

本イテレーションでは以下の2つの目標を達成する:

1. **history-based IDLE detection** — デモスクリプトで使われている `wait_for_agent_idle()` はファイルアーティファクトの有無またはエージェントの IDLE ステータスのポーリングに依存している。これは信頼性が低い（エージェントが新しいタスクを受け取った場合に IDLE に戻ってしまう、アーティファクトが書かれない場合の偽成功など）。`GET /agents/{id}/history` エンドポイントは `task_id` を使って特定タスクの完了を確実に検出できる。

2. **worktree_integrity.py → infrastructure/** — v1.0.16/1.0.17 で worktree.py, process_port.py, messaging.py を infrastructure/ に移行した。`worktree_integrity.py` はインフラ層の関心事（git コマンド実行）であり、同様に移行すべき。

選択しなかった候補: application layer のさらなる整理（前イテレーションからの継続で依存関係が複雑なため次回に延期）

## 調査記録 (Step 1)

### Query 1: Async task completion polling patterns REST API

**Source**: Microsoft Azure Architecture Center — "Asynchronous Request-Reply Pattern"
URL: https://learn.microsoft.com/en-us/azure/architecture/patterns/asynchronous-request-reply
Year: 2024

**Findings**:
- HTTP 202 Accepted → Location header → polling for status
- Client polls status endpoint with exponential backoff
- When `finished_at` is set (non-null), the task is complete
- Status endpoint returns the resource + redirect (303) on completion

**Source**: adidas API Guidelines — "Polling"
URL: https://adidas.gitbook.io/api-guidelines/rest-api-guidelines/execution/long-running-tasks/polling

**Findings**:
- `Retry-After` header to communicate ideal poll interval
- Task record with `status` field: `in_progress` → `completed` / `failed`
- Poll at gradually increasing intervals (exponential backoff with jitter)

### Query 2: Polling vs webhooks task completion detection

**Source**: hookdeck.com — "When to Use Webhooks, WebSocket, Pub/Sub, and Polling"
URL: https://hookdeck.com/webhooks/guides/when-to-use-webhooks
Year: 2025

**Findings**:
- Polling: simple, no server-side setup. Ideal when control lies with the client (demo scripts).
- Webhooks: require a receiver. Not feasible in demo scripts (no HTTP server in demo).
- Hybrid: use polling with task_id-based termination condition (not IDLE status), which is immune to the race condition where the agent goes IDLE then receives another task.

**Source**: Zapier analysis (cited in search results) — 98.5% of polls are "wasted" (no new data).
Implication: polling interval should be 5–10s for long-running agent tasks, not 1s.

### Query 3: Strangler Fig pattern for module migration

**Source**: Martin Fowler, "StranglerFigApplication" (2004) — coined the term.
Referenced in: https://www.gocodeo.com/post/how-the-strangler-fig-pattern-enables-safe-and-gradual-refactoring
- Pattern: new module lives at new path; old path becomes a shim that re-exports.
- This preserves backward compatibility while redirecting imports incrementally.
- Shim is eventually removed when all callers updated.

**Source**: Shopify Engineering — "Refactoring Legacy Code with the Strangler Fig Pattern"
URL: https://shopify.engineering/refactoring-legacy-code-strangler-fig-pattern

**Findings** for Python:
```python
# Old path: worktree_integrity.py (shim)
from tmux_orchestrator.infrastructure.worktree_integrity import *  # noqa: F401, F403
from tmux_orchestrator.infrastructure.worktree_integrity import WorktreeStatus, WorktreeIntegrityChecker
```

### Query 4: Python exponential backoff polling

**Source**: PyPI backoff library — https://pypi.org/project/backoff/
**Source**: Tenacity — https://tenacity.readthedocs.io/

**Findings**:
- For synchronous demo scripts (no asyncio in demo.py), simple `time.sleep(poll_interval)` loop is preferred.
- Exponential backoff formula: `min(poll_interval * 2**attempt, max_interval)`
- For the demo_client helper: fixed interval polling is sufficient (agents take minutes, not ms)
- Key termination condition: `task_id` appears in history with `finished_at` is not None

## 設計決定

### A. `wait_for_agent_done` の配置場所

**選択: `src/tmux_orchestrator/testing/demo_client.py`**

理由:
- デモスクリプトは `PYTHONPATH=src` で実行するため、`tmux_orchestrator.testing.demo_client` が直接 import 可能
- `~/Demonstration/demo_utils.py` はプロジェクトの依存関係管理外（バージョン管理されない）
- `testing/` サブパッケージにより、将来的にテスト用モックや fixture も同梱できる
- `src/` 配下に置くことで `uv run pytest` でも import できる
- パッケージとして配布した場合でも optional extra として提供可能

`~/Demonstration/demo_utils.py` を選ばない理由:
- バージョン管理されないため、demo スクリプト間の一貫性が保てない
- `PYTHONPATH` の追加設定が必要
- 将来的な API 変更への追従が難しい

### B. `wait_for_agent_done` の設計

```python
def wait_for_agent_done(
    base_url: str,
    agent_id: str,
    task_id: str,
    *,
    api_key: str = "",
    timeout: float = 900,
    poll_interval: float = 5.0,
) -> dict:
    """Poll GET /agents/{agent_id}/history until task_id appears with finished_at set.

    Termination condition: task_id in history AND record["finished_at"] is not None.
    This is immune to the IDLE-race condition (agent goes IDLE then gets another task).

    Returns the history record dict.
    Raises TimeoutError on timeout.
    Raises RuntimeError if agent not found (404).
    """
```

**Termination logic**:
1. Poll `GET /agents/{agent_id}/history?limit=200`
2. Search for record where `record["task_id"] == task_id`
3. If found AND `record["finished_at"] is not None` → return record
4. If not found → agent hasn't started or hasn't completed yet → continue polling
5. On timeout → raise `TimeoutError`

**Note on history limit**: The default `limit=50` may not be enough if many tasks have been dispatched. Use `limit=200` (the server-side cap).

## レビュー指摘

### レビュー指摘 1: `wait_for_agent_done` の `task_id` 型不一致リスク
> エンドポイントから返される `task_id` の型が UUID 文字列の場合、デモスクリプトが `task1 = api("POST", "/tasks", ...)` で受け取る `task_id` の形式と一致しているか確認が必要。`task1.get("task_id", task1.get("id", ""))` というパターンでは "id" と "task_id" が混在している。

> 対応: `wait_for_agent_done` は文字列比較 (`record["task_id"] == task_id`) で実装する。デモスクリプトでは `task_id = task1.get("task_id", task1.get("id", ""))` の両方のキーを試してから渡す。これは既存のデモパターンと一致する。

### レビュー指摘 2: `testing/` パッケージがインストール成果物に含まれる
> `src/tmux_orchestrator/testing/` は通常ユーザーに配布するパッケージに含まれてしまう。production install に test helper が混入する。

> 対応: `testing/` は `tmux-orchestrator` パッケージの一部として配布するが、demo helper という意味合いから `testing.demo_client` を `testing/` に置くことはむしろ適切（pytest fixtures や test helpers と同じ慣習）。将来的に `optional-dependencies` の `dev` extra に分離可能。今回はリスクが低いためそのまま進める。

### レビュー指摘 3: `GET /agents/{id}/history` の `task_id` フィールドの存在保証
> 旧バージョンの history レコードには `task_id` フィールドがない可能性がある。`KeyError` を防ぐ必要がある。

> 対応: `record.get("task_id") == task_id` を使用して `KeyError` を回避する。

### レビュー指摘 4: `worktree_integrity.py` が `web/app.py` 以外で import されているか確認
> shim を書く前に、実際にどのファイルが `worktree_integrity` を import しているか確認が必要。

> 対応: 実装前に `grep -r "worktree_integrity"` で全 import 箇所を確認してから shim を作成する。
