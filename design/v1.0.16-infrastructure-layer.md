# v1.0.16 — infrastructure/ 層の明示化

## 目的

Clean Architecture の dependency rule を強制する。`domain/` と `application/` が
`infrastructure/` に依存しないことを AST ベースのテストで保証する。

---

## infrastructure/ 層に属するもの

Hexagonal Architecture（Cockburn 2005、書籍版 2024）では、外部システムとの通信を担う
コンポーネントを「アダプター」と呼ぶ。これらが `infrastructure/` 層に属する。

| カテゴリ | 対象ファイル | 理由 |
|---|---|---|
| tmux バインディング | `tmux_interface.py` | `libtmux` に直接依存。外部プロセス管理 |
| Claude 信頼設定 | `trust.py` | `~/.claude.json` という外部ファイルシステムの操作 |
| (将来候補) | `worktree.py` | `git` サブプロセスを直接呼ぶ filesystem adapter |
| (将来候補) | `messaging.py` | ファイルシステムベースのメールボックス |

### 今回のスコープ

- `tmux_interface.py` → `infrastructure/tmux.py`（shim を旧パスに残す）
- `trust.py` → `infrastructure/claude_trust.py`（shim を旧パスに残す）

---

## 依存方向のルール

```
domain/ ←── application/ ←── infrastructure/ ←── (外部ライブラリ)
```

- `domain/` は何にも依存しない（stdlib のみ可）
- `application/` は `domain/` のみに依存（`infrastructure/` 不可）
- `infrastructure/` は `domain/` と `application/` の両方に依存可
- この逆方向の依存は **禁止**

**参考文献**:
- Martin, Robert C. "Clean Architecture" (2017) Ch. 22 — The Clean Architecture
  - 「ソースコードの依存関係は常に内側（より高レベルのポリシー）を指し示さなければならない」
- Cockburn, Alistair & Garrido de Paz, Juan Manuel. "Hexagonal Architecture Explained" (2024)
  - Ports and Adapters: domain が外部アダプターを知らない設計
  - URL: https://blog.szymonmiks.pl/p/hexagonal-architecture-in-python/
- Sourcery AI, "Maintain A Clean Architecture With Dependency Rules" (2022)
  - AST/import ベースで依存ルールを検査するアプローチ
  - URL: https://www.sourcery.ai/blog/dependency-rules
- Miks, Szymon. "Hexagonal Architecture in Python" (blog.szymonmiks.pl)
  - Infrastructure = output adapters (DB, external APIs, filesystem)
  - URL: https://blog.szymonmiks.pl/p/hexagonal-architecture-in-python/

## Strangler Fig パターン（後方互換 shim）

既存コードが `from tmux_orchestrator.tmux_interface import TmuxInterface` を使っている。
これを壊さないため、旧パスに shim モジュールを残す:

```python
# tmux_interface.py (shim)
"""Backward-compatibility shim. Import from infrastructure.tmux instead."""
from tmux_orchestrator.infrastructure.tmux import *  # noqa: F401,F403
from tmux_orchestrator.infrastructure.tmux import TmuxInterface, PaneOutputEvent, POLL_INTERVAL
```

これは Strangler Fig パターンの応用：新しい構造に移行しつつ、旧インターフェースを維持する。

参考: Fowler, Martin. "Strangler Fig Application" (martinfowler.com, 2004)
     https://en.wikipedia.org/wiki/Strangler_fig_pattern

## レビュー指摘

### 指摘 1: shim の `libtmux`・`time` 再エクスポートと循環依存リスク
`tmux_interface.py` shim が `import libtmux` と `import time` を直接行う。
これは既存テストの `@patch("tmux_orchestrator.tmux_interface.libtmux.Server")` を
維持するために必要だが、shim が外部ライブラリに直接依存する点は設計上やや曖昧。

> 対応: `libtmux` は stdlib ではなく外部ライブラリだが、shim はテスト互換性のため再エクスポートする。
> 将来的にテスト側を `@patch("tmux_orchestrator.infrastructure.tmux.libtmux.Server")` に
> 移行すれば shim から `libtmux` の直接 import を除去できる。現時点では互換性優先。

### 指摘 2: `worktree.py` / `messaging.py` は今回スコープ外だが将来候補に明記済
`worktree.py` (git サブプロセス) と `messaging.py` (ファイルシステム) も infrastructure 層に属するが、
今回は移行しない。これにより `infrastructure/` が部分的な状態になる。

> 対応: DESIGN.md §11 に次回 infrastructure 拡張候補として追記する。部分的な状態は
> Strangler Fig パターンの正常な過渡期であり、既存テストが全通過している限り問題ない。

### 指摘 3: AST テストの網羅性 — shim ファイル自体が infrastructure を import する
`tmux_interface.py` shim は `from tmux_orchestrator.infrastructure.tmux import ...` を行う。
purity テストの対象が `domain/` と `application/` のみなので問題ないが、
shim が `infrastructure` を import することは設計意図通りである（shim は infrastructure 層の一部）。

> 対応: shim は `domain/` でも `application/` でもなく、`src/tmux_orchestrator/` のルート層に置かれる。
> AST テストの対象外であり、設計意図と整合している。

### 指摘 4: `infrastructure/__init__.py` が `claude_trust._atomic_write_json` を re-export しない
内部関数 `_atomic_write_json` はパブリック API ではないため、`__init__.py` に含まれていない。
`trust.py` shim は含むが、`infrastructure/__init__.py` では省略している。

> 対応: `_atomic_write_json` はアンダースコア付きのプライベート関数のため、`__init__.py` の
> `__all__` から除外することは正しい。shim がテスト用に re-export しているのは既存コードとの
> 互換性維持のため（`test_trust_dialog.py` が直接使用する可能性）。問題なし。
