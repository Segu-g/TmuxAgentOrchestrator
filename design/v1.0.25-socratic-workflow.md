# Design: v1.0.25 — POST /workflows/socratic — ソクラテス的対話ワークフロー

## 問題ステートメント

現在のワークフローエンドポイント群（`/workflows/tdd`, `/workflows/debate`, `/workflows/adr`,
`/workflows/delphi`, `/workflows/redblue`）は構造化された協調パターンを提供しているが、
**ソクラテス的対話（Socratic dialogue）**パターンが欠けている。

技術的設計判断（「REST vs GraphQL」「モノリス vs マイクロサービス」「同期 vs 非同期 API」等）の
曖昧性解消において、前提・定義・論拠を段階的に問い詰めるマイウティカ（産婆術）的アプローチが
単純な賛否討論より有効であることが研究で示されている。

単一エージェントや通常の debate では「どちらが良いか」の結論が先行し、前提の検討が不十分になりやすい。
questioner が前提・定義・論拠を問い、responder が回答を精緻化し、synthesizer が構造化結論を抽出する
3エージェント構成により、設計判断の論拠を深掘りし、曖昧性を体系的に解消できる。

## 提案する解決策

`POST /workflows/socratic` — 3エージェント構成のソクラテス的対話ワークフロー:

1. **questioner**: マイウティカ法（産婆術）で前提・定義・論拠を問う。最初は強い反論型の問い、
   後のラウンドは統合的・建設的な問いに変化させる段階的移行モデルを採用する。
2. **responder**: questioner の問いに対して回答を精緻化し、立場を明確にし、論拠を強化する。
3. **synthesizer**: 対話ログ全体を読み、構造化された結論（`synthesis.md`）を抽出する。
   主な論点・合意点・未解決の問い・推奨事項をMARD形式で整理する。

成果物はスクラッチパッド（Blackboardパターン）経由で受け渡し:
- `{prefix}_dialogue` — questioner/responder の問答ログ（questioner が書き込む）
- `{prefix}_synthesis` — synthesizer の構造化結論

## 実装計画

### 1. Pydanticモデル (`SocraticWorkflowSubmit`)
- `topic: str` — 対話テーマ（例: "REST vs GraphQL for mobile API"）
- `questioner_tags: list[str]` — questioner required_tags
- `responder_tags: list[str]` — responder required_tags
- `synthesizer_tags: list[str]` — synthesizer required_tags
- `reply_to: str | None` — synthesizer RESULT の宛先エージェント
- バリデーション: topic 非空

### 2. エンドポイント (`POST /workflows/socratic`)
- `/workflows/redblue` と同じルーティング・認証パターン
- DAG: questioner → responder (questioner依存) → synthesizer (responder依存)
- スクラッチパッドキー: `socratic_{uuid[:8]}_dialogue` / `_synthesis`
- 戻り値: `workflow_id`, `name`, `task_ids`, `scratchpad_prefix`

### questioner プロンプト設計
- 役割: マイウティカ法（産婆術）で前提・定義・論拠を問う
- フェーズ1（強い問い）: 「その前提の根拠は何か？」「定義を明確にせよ」「反例を示せ」
- フェーズ2（建設的な問い）: 「どの条件下でその立場が正しいか？」「合意点はどこか？」
- 問答ログを `{prefix}_dialogue` スクラッチパッドに記録

### responder プロンプト設計
- questioner の問いを読み、立場を精緻化・防御・更新する
- 回答を対話ログに追記して `{prefix}_dialogue` を更新

### synthesizer プロンプト設計
- 対話ログ全体を読み、`synthesis.md` を生成:
  ```
  # Synthesis: {topic}
  ## 主な論点
  ## 合意した点
  ## 未解決の問い
  ## 推奨事項
  ## 参考資料・前提条件
  ```
- `{prefix}_synthesis` スクラッチパッドに書き込む

### 3. テスト (`tests/test_workflow_socratic.py`)
- `SocraticWorkflowSubmit` バリデーション（空 topic 等）
- HTTP 200 レスポンスと workflow 構造
- タスク数 (3) と依存チェーン検証
- スクラッチパッドキー命名
- 認証 (401)
- `reply_to` 伝播
- タグ転送
- プロンプト内容確認（topic、役割名称、スクラッチパッドキーの記載）

## 調査結果

### Query 1: "SocraSynth multi-agent Socratic debate LLM platform questioner responder arXiv 2402.06634"

主要知見:
- **SocraSynth** (Liang et al., arXiv:2402.06634, 2024): モデレーター型マルチエージェント討論プラットフォーム。
  LLMペルソナが異なる論点から議題を討論し、合意形成に至る。段階的移行（最初は強い反論、
  後は統合的問い）が収束速度と最終文書の論理整合性を高める。
- ソクラテス的メソッドの3フェーズ: (1) 前提の問い (elenchus)、(2) 定義の精緻化 (maieutics)、
  (3) 統合的結論の抽出 (synthesis)。各フェーズを専用エージェントに割り当てることで
  役割分離が明確になり、コンテキスト汚染が防げる。
- 匿名性の重要性: Delphi 法と同様に、questioner が特定の立場に肩入れしない中立性を保つことで
  responder のバイアスを最小化できる。

**References**:
- Liang et al., "SocraSynth: Exploring LLM Reasoning via Socratic Synthesis", arXiv:2402.06634, 2024,
  https://arxiv.org/abs/2402.06634
- "SocraSynth" GitHub, https://github.com/avnlp/socrasynth

### Query 2: "KELE EMNLP 2025 LLM Socratic teaching agent arXiv 2409.05511"

主要知見:
- **KELE** (arXiv:2409.05511, EMNLP 2025): LLMベースのソクラテス教授エージェント。
  直接回答を与えずに問いかけによって学習者の理解を深める。コンピュータサイエンス教育で
  学習効果が通常の Q&A チャットボットより向上することを実証。
- 段階的問い形成: 最初は「この仮定は本当に成立するか？」の挑戦的問い、
  後に「どの条件下で正しいか？」の構成的問いに移行する2フェーズが最も効果的。
- 対話ログの構造化: 問答ペアをスクラッチパッドに蓄積し、最終的に synthesizer が抽出する
  パターンが LLM マルチエージェントシステムで実用的。

**References**:
- "KELE: A Knowledge-Enhanced LLM for Educating with Socratic Questions", arXiv:2409.05511,
  EMNLP 2025, https://arxiv.org/abs/2409.05511

### Query 3: "multi-agent dialogue synthesis structured conclusion extraction LLM design decision 2025"

主要知見:
- **CONSENSAGENT** (ACL 2025): 動的プロンプト洗練で迎合（sycophancy）を抑制し、
  効率的な合意形成を実現。questioner が立場に拘束されない場合、responder の迎合傾向が
  低下することを実証。
- **構造化結論抽出**: SocraSynth / RT-AID の共通知見として、「何に合意したか」と
  「何が未解決か」を明示的に分離することが最終文書の有用性を高める。
  synthesis agent が両方を明記することで設計者への情報価値が最大化される。
- **技術的設計判断への適用**: "Automating Architecture Decision Records with AI Agents"
  (InfoQ 2024) — ADR 自動化では questioner/critic + responder + synthesizer の
  3役割が実際の ADR レビューワークフローに最も近いと報告。

**References**:
- "CONSENSAGENT: Towards Efficient and Effective Consensus in Multi-Agent LLM Interactions",
  ACL 2025, https://aclanthology.org/2025.findings-acl.1141/
- "Automating Architecture Decision Records with AI Agents", InfoQ 2024,
  https://www.infoq.com/articles/adr-automation-ai/
- Anthropic, "Building effective agents", anthropic.com 2025,
  https://www.anthropic.com/engineering/building-effective-agents

## テスト計画

1. `SocraticWorkflowSubmit(topic="")` → ValidationError
2. `SocraticWorkflowSubmit(topic="  ")` → ValidationError
3. `SocraticWorkflowSubmit(topic="REST vs GraphQL")` → OK
4. `SocraticWorkflowSubmit(topic="t", questioner_tags=["q"], ...)` → OK
5. `POST /workflows/socratic` → 200 + `workflow_id`, `task_ids`, `scratchpad_prefix`
6. `task_ids` に `questioner`, `responder`, `synthesizer` の3キー

8. `scratchpad_prefix` が `"socratic_"` で始まる
9. questioner タスクの依存なし
10. responder タスクが questioner に依存
11. synthesizer タスクが responder に依存

18. responder プロンプトに dialogue キーが含まれる

21. `POST /workflows/socratic` with wrong API key → 401

## レビュー指摘

**レビュアー**: Context-free sub-agent review (v1.0.25)

### 指摘1: responder が dialogue ログを上書きするリスク
questioner が `{prefix}_dialogue` に書き込み、responder が同じキーに上書き更新する設計。
responder がキーを読み込まずに独自コンテンツで上書きすると questioner のQ&Aが失われる。
指示プロンプトで「questioner のテキストを引用してから答えを追記」と明示する必要がある。

> 対応: responder プロンプトに「quote each question before answering」を明記。
> また `FULL_DIALOGUE` 変数で questioner 出力 + responder 出力を結合してから書き込む bash スニペットを提供した。

### 指摘2: synthesizer が synthesis_key に書き込まないケースへの対処
synthesizer が synthesis.md を作成しない、または scratchpad PUT に失敗した場合、
check [18] が FAIL になる。エラーパスを明示する。

> 対応: デモの nudge 機構（3分 IDLE 検出 → 手動 task-complete）が存在するため、
> タスク失敗の場合は check [13] が FAIL になり連鎖チェックがスキップされる設計で対応済み。

### 指摘3: 役割タグの必要性
`questioner_tags`, `responder_tags`, `synthesizer_tags` を使わない場合、全タスクが
最初の IDLE エージェントに集中する可能性がある（v1.0.24 build-log で発生）。

> 対応: デモ config で各エージェントに `tags: [questioner/responder/synthesizer]` を設定し、
> workflow submit で `required_tags` を指定することでルーティングを保証した。
