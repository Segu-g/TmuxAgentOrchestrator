# v1.0.28 — MIRIX型エピソード記憶ストア

## 問題定義

現行の `/summarize` → `NOTES.md` ではタスク間で過去エピソードが上書き消滅する。
エージェントが同じ種類のタスクを繰り返す場合、前回の実行結果 (何が失敗したか、
どのアプローチが有効だったか) を参照できない。

## 提案する解決策

タスク完了ごとに `{task_id, summary, outcome, lessons, timestamp}` を
JSONL ファイルに追記する **エピソード記憶ストア** を追加する。

### REST API

| メソッド | エンドポイント | 説明 |
|----------|--------------|------|
| `GET /agents/{id}/memory` | エピソード一覧 (新しい順、limit クエリパラメータ対応) |
| `POST /agents/{id}/memory` | エピソード追加 |
| `DELETE /agents/{id}/memory/{episode_id}` | エピソード削除 |

### エピソードスキーマ (Pydantic v2)

```python
class EpisodeCreate(BaseModel):
    summary: str           # 1–2 文のタスク要約
    outcome: str           # "success" | "failure" | "partial"
    lessons: str = ""      # 次回活かせる知見 (自由記述)
    task_id: str | None = None

class Episode(EpisodeCreate):
    id: str                # UUID
    agent_id: str
    created_at: str        # ISO 8601
```

### 永続化

`{mailbox_dir}/{session_name}/{agent_id}/episodes.jsonl`
(エージェントのメールボックスと同じ場所に配置)

### タスク完了フック連携

`result_store.py` の `record_result()` 呼び出し後に
`EpisodeStore.append()` を自動呼び出しする。
`TaskResultPayload.summary` フィールドを新設し、
エージェントが `/task-complete <summary>` で渡した要約を格納する。

## テスト計画

- `EpisodeStore.append()`: JSONL に行が追記されること
- `EpisodeStore.list()`: 新しい順にソート、limit 対応
- `EpisodeStore.delete()`: 指定 ID の行が削除されること
- `GET /agents/{id}/memory`: 一覧返却、unknown agent は 404
- `POST /agents/{id}/memory`: エピソード作成、レスポンスに id が含まれること
- `DELETE /agents/{id}/memory/{episode_id}`: 削除成功 / 存在しない ID は 404

## 研究調査結果

### Query 1: MIRIX multi-agent memory system episodic memory arXiv 2507.07957 2025

**主要知見**:
- MIRIX (Wang & Chen, 2025): 6種類のメモリタイプ (Core/Episodic/Semantic/Procedural/Resource/Knowledge Vault)
- エピソードメモリ: タイムスタンプ付き状況的経験を記録し特定イベントを想起可能
- ScreenshotVQA ベンチマーク: RAG ベースラインより **35% 精度向上**、ストレージは 99.9% 削減
- LOCOMO (長期会話ベンチマーク): SOTA 85.4%
- エピソード単位: `{summary, details}` フィールド構造

**References**:
- Wang & Chen, "MIRIX: Multi-Agent Memory System for LLM-Based Agents", arXiv:2507.07957, 2025, https://arxiv.org/abs/2507.07957

### Query 2: episodic memory LLM agents JSONL persistence task summary retrieval 2025

**主要知見**:
- 永続化パラダイム: 累積追記 (complete historical appending) が長期記憶に最適
- エピソード→セマンティック昇格: 成功したエピソードは background process が抽象化してセマンティックメモリに昇格
- "Position: Episodic Memory is the Missing Piece for Long-Term LLM Agents" (arXiv:2502.06975, 2025): 単発学習・コンテキスト対応検索・知識統合を支える記憶基盤が必要
- JSONL 追記形式: 軽量でリアルタイム追記に適し、Redis/SQLite の依存を回避できる

**References**:
- "Position: Episodic Memory is the Missing Piece for Long-Term LLM Agents", arXiv:2502.06975, 2025, https://arxiv.org/pdf/2502.06975
- "Memory in the Age of AI Agents", arXiv:2512.13564, 2025, https://arxiv.org/abs/2512.13564

### Query 3: agent task history REST API design best practices LLM orchestration 2025

**主要知見**:
- LangChain/CrewAI の memory 設計: read/write/reflection の3操作が標準
- 「すべての重要な決定について生の入力・プロンプト・最終出力・決定パスを含む詳細なイミュータブルログエントリを書く」 — for both debugging and auditability
- LLM-first API 設計: OpenAPI スペック + system prompt の組み合わせで agent が REST を直接呼べる構造が推奨

**References**:
- "LLM Orchestration in 2025: Frameworks + Best Practices", orq.ai Blog, 2025, https://orq.ai/blog/llm-orchestration
- "AI Agent Orchestration Patterns", Azure Architecture Center, Microsoft Learn, 2025, https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns

## レビュー指摘

### レビュー (インラインレビュー — 2026-03-09)

**指摘1**: `EpisodeStore.delete()` は全行を読み込んでから再書き込みするため、エピソード数が多い場合に O(N) になる。SQLite など真のランダムアクセスストレージへの移行を検討すべき。
> 対応: 設計上の制約として受け入れる。エピソード数は通常 100 未満（1エージェントが 1 タスクで 1 エピソード）であり、O(N) のコストは許容範囲。将来のチェックポイント SQLite 基盤との統合時に解消予定（`DESIGN.md §11`）。

**指摘2**: `has_agent()` はファイルの存在のみ確認しており、エージェントがレジストリに登録されているかどうかを確認しない。dead agent のファイルが残る可能性がある。
> 対応: `has_agent()` はユーティリティメソッドであり、REST エンドポイントでは別途 `orchestrator.get_agent()` で存在確認している。dead agent ファイルの自動クリーンアップは worktree 削除フック（v1.0.0）に委ねる。

**指摘3**: `outcome` フィールドの Literal 型 (`"success" | "failure" | "partial"`) は `EpisodeStore` クラスでは検証されず、スキーマ検証は Pydantic レイヤーのみ。JSONL に直接書き込む内部呼び出しでは任意の文字列が入り得る。
> 対応: `EpisodeStore.append()` に outcome バリデーションを追加することは検討に値するが、現時点では REST API 経由の全呼び出しが Pydantic で検証される。内部直接呼び出しは存在しない（将来の自動記録フックでは Pydantic モデルを経由させる）。

**指摘4**: デモの writer_prompt に bash インジェクション脆弱性の可能性がある (`sed 's/\"/\\\\\"/g'` でコード内のバックスラッシュが問題になりうる)。
> 対応: デモはセキュリティ評価対象ではなく、制御された環境で実行される。merge_sort.py のコード内容はバックスラッシュを含まないため実害なし。本番 API 経由では `PUT /scratchpad` が JSON スキーマで保護される。
