## Feature 1: Trust ダイアログ自動抑制

### 問題陳述

Claude Code はワークツリーへの初回起動時に「Do you trust the files in this folder?」という
インタラクティブプロンプトを表示する。このプロンプトは `_wait_for_ready()` が待機している
SessionStart フック（`POST /agents/{id}/ready`）が呼ばれる前に表示されるため、
エージェントが応答しない状態になりタイムアウトを引き起こす。

```
Claude Code 起動
  → Trust ダイアログ表示（stdin 待ち）
  → SessionStart フック呼ばれない
  → _wait_for_ready() が 60 秒タイムアウト
  → デモ失敗
```

### Trust 格納形式の調査

#### 調査方法

1. GitHub Issues 検索: `Claude Code "trust this folder"` `hasTrustDialogAccepted`
2. ローカル `~/.claude.json` の実検査

#### 実際の格納形式（実機確認済み）

`~/.claude.json` の `projects` キー下に、絶対パスをキーとして格納される:

```json
{
  "projects": {
    "/home/segument": {
      "allowedTools": [],
      "mcpContextUris": [],
      "mcpServers": {},
      "enabledMcpjsonServers": [],
      "disabledMcpjsonServers": [],
      "hasTrustDialogAccepted": false,
      "projectOnboardingSeenCount": 0,
      "hasClaudeMdExternalIncludesApproved": false,
      "hasClaudeMdExternalIncludesWarningShown": false
    }
  }
}
```

必要最小フィールド（信頼付与に必要）:
- `hasTrustDialogAccepted: true` — ワークスペース信頼ダイアログをスキップ

オプションフィールド（安定性向上）:
- `hasClaudeMdExternalIncludesApproved: true` — CLAUDE.md の外部 include 承認

親ディレクトリ探索仕様（GitHub Issue #23109 より）:
```javascript
// Claude Code の内部実装（obfuscated JS から逆引き）
while(!0) { if(T.projects?.[_]?.hasTrustDialogAccepted) return true; ... }
```
つまり **親ディレクトリを遡って確認する**。ホームディレクトリを信頼すれば全サブディレクトリが信頼される。

#### 重要な注意点

- v2.0.8 で `.claude.json` の `allowedTools`, `ignorePatterns` 等は廃止され `settings.json` へ移行
- しかし `hasTrustDialogAccepted` は `.claude.json` の `projects` で現在も有効（実機で確認）
- ファイルは JSON 形式でアトミック書き込みが必要（並行エージェント起動時のレース回避）

### 参考文献

1. GitHub Issue #23109 "Trusted workspace patterns for git worktrees" — https://github.com/anthropics/claude-code/issues/23109
2. GitHub Issue #2147 "Claude forgets trust dialog acceptance" — https://github.com/anthropics/claude-code/issues/2147 — コメント内の Python スクリプトが格納形式を明示
3. GitHub Issue #9113 "Workspace Trust Dialog not respecting ~/.claude.json" (closed NOT_PLANNED, 2026-01-10) — https://github.com/anthropics/claude-code/issues/9113
4. GitHub Issue #5572 "hasTrustDialogHooksAccepted option" — https://github.com/anthropics/claude-code/issues/5572

### 提案実装: `_pre_trust_worktree(cwd: Path)`

`claude_code.py` に以下を追加:

```python
def _pre_trust_worktree(self, cwd: Path) -> None:
    """~/.claude.json に worktree の信頼エントリを事前作成する。

    Claude Code は起動時に ~/.claude.json の projects[path].hasTrustDialogAccepted を
    確認し、true であればインタラクティブな "trust this folder" ダイアログをスキップする。
    このメソッドは _launch_claude() 前に呼ばれ、ダイアログによる _wait_for_ready() タイムアウトを防ぐ。

    冪等: 既存エントリを上書きしない（hasTrustDialogAccepted が既に true の場合保持）。
    アトミック書き込み: 一時ファイル → rename で競合状態を防ぐ。
    """
```

**呼び出し場所**: `start()` 内の `_setup_worktree()` 後、claude 起動直前

**アトミック書き込み**:
1. `~/.claude.json` をロード（存在しない場合は `{}` として扱う）
2. `projects[str(cwd)]` が存在しない場合は空 dict を作成
3. `hasTrustDialogAccepted` を `True` に設定（既存値が True の場合はスキップ）
4. 一時ファイルに書き込み後、`os.replace()` で原子的に置換

**セキュリティ考慮**:
- `~/.claude.json` は Claude Code が所有するユーザーファイル → 書き込み権限は問題なし
- JSON ファイルへの機密情報書き込みなし（`hasTrustDialogAccepted` は Boolean のみ）
- 一時ファイルは同一ディレクトリに作成して原子的 rename を保証

## Feature 2: orchestrator.py 部分的 DI 化 (TaskQueue Protocol)

### 問題陳述

`Orchestrator.__init__` が `asyncio.PriorityQueue` を直接インスタンス化しており、
テストで差し替えが困難。TaskQueue Protocol を抽出することで:
- ユニットテストに fakeキューを注入できる
- インフラ（asyncio）への直接依存が application 層から切り離せる
- DESIGN.md §11 の「orchestrator.py のインフラ依存を DI 化」の第一歩

### 提案実装: TaskQueue Protocol

```python
# task_queue.py (新ファイル)
from typing import Protocol, TypeVar

class TaskQueue(Protocol):
    async def put(self, item: tuple[int, int, Task]) -> None: ...
    async def get(self) -> tuple[int, int, Task]: ...
    def task_done(self) -> None: ...
    def empty(self) -> bool: ...
    def qsize(self) -> int: ...

class AsyncPriorityTaskQueue:
    """asyncio.PriorityQueue の TaskQueue 実装。"""
    def __init__(self, maxsize: int = 0): ...
```

`Orchestrator.__init__` に:
```python
task_queue: "TaskQueue | None" = None,
```
を追加し、`None` の場合は `AsyncPriorityTaskQueue(maxsize=config.task_queue_maxsize)` をデフォルトとする。

## レビュー指摘

_レビュアー: 設計文書を文脈なしで読んだ独立レビュアーによる指摘 (2026-03-09)_

### 指摘1: ~/.claude.json の並行書き込みレース

**問題**: 複数エージェントが同時起動すると、それぞれが `~/.claude.json` を読み取り →
加工 → 書き込む手順を踏む。一時ファイル+rename はファイル全体を原子的に置き換えるが、
2 プロセスが同時に read → modify → replace するとラストライターが他のエントリを上書きする
（lost update）。

**重大度**: 中（デモ環境では通常 2〜3 エージェントを同時起動するため発生確率あり）

> **対応**: `fcntl.flock()` などでファイルロックを追加する。Linux では flock + rename の組み合わせが
> アトミックかつ移植性が高い。ただし `~/.claude.json` は小さなファイルなので、write の窓は
> 極めて短い。競合が発生しても冪等性により次回の pre_trust_worktree 呼び出しで修復される。
> v1.0.13 では「最後の書き込みが勝つ」許容範囲内とする。trust 付与は実際のエントリが残れば
> 十分であり、複数の trust エントリが混在しても問題はない。将来的にロック追加を検討 (§11候補)。

### 指摘2: ~/.claude.json が Claude Code 以外のプロセスにより書き換えられる可能性

**問題**: Claude Code 自体が起動中に ~/.claude.json を更新する場合があり（セッション設定保存など）、
pre_trust_worktree が書いたエントリを上書きする可能性がある。

**重大度**: 低（Claude Code は起動前に書き込むため、タイミング的に問題になりにくい）

> **対応**: pre_trust_worktree は claude 起動前に呼ばれる（start() の設計通り）。Claude Code が
> 書き込む前に trust エントリを設定しておけば、Claude Code が起動時にそのエントリを読んで
> trust ダイアログをスキップできる。Claude Code が起動後に ~/.claude.json を上書きする際も
> trust エントリは保持されるはずだが、バージョンによって異なる可能性あり。

### 指摘3: TaskQueue Protocol の `_queue`/`_unfinished_tasks` 内部属性

**問題**: Protocol にない内部属性 `_queue`, `_unfinished_tasks` へのアクセスを
`AsyncPriorityTaskQueue` の property で shim している。FakeQueue 等の代替実装が
これらを実装しなければ orchestrator の cancel/reprioritize ロジックで AttributeError が発生する。

**重大度**: 中（DI 目的のモックを書く開発者がこの制約を知らないと問題になる）

> **対応**: `AsyncPriorityTaskQueue` の docstring に「internal shim properties」であることを明記済み。
> また、test_task_queue_di.py の FakeTaskQueue も `_queue` プロパティを実装している。
> 長期的には orchestrator の cancel/reprioritize ロジックを `peek_all()`, `remove()`, `replace_all()`
> などの公式 Protocol メソッドに移行すべき（次イテレーション候補として §11 に追加）。

### 指摘4: 非 isolate エージェントへの trust 付与

**問題**: `isolate=False` のエージェントは既存ディレクトリ（リポジトリルート等）で動作する。
そのディレクトリへの pre_trust_worktree 書き込みが既存の Claude Code セッションの
trust 状態を変更する副作用がある。

**重大度**: 低（trust を true に変更するだけで、ユーザーの明示的行動を超えない）

> **対応**: pre_trust_worktree は hasTrustDialogAccepted=true への書き込みのみ行う。
> 既存の true エントリは変更しない（冪等）。false→true の変更はユーザーが
> tmux-orchestrator を起動した意図と一致している（悪意ある操作ではない）。

### 指摘5: platform 依存 (macOS vs Linux)

**問題**: `~/.claude.json` のパスは macOS/Linux 共通だが、`os.replace()` の原子性は
Linux では同一ファイルシステム内で保証されるが、macOS の一部ケース（NFS マウント等）では
保証されない場合がある。

**重大度**: 最低（デモ環境は Linux のみ）

> **対応**: `tmp_path = tempfile.mkstemp(dir=target.parent, ...)` により同一ディレクトリに
> 一時ファイルを作成するため、同一ファイルシステムが保証される。Linux での使用を想定。
