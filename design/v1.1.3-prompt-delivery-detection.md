## Part A: File-Existence Based Paste Detection

### Problem

v1.1.2 introduced the `UserPromptSubmit` hook pattern:
- `_dispatch_task()` writes the full prompt to `__task_prompt__<agent_id>__.txt`
- Sends short trigger `__TASK__` via `send_keys` (no paste-preview risk)
- The hook reads the file, deletes it, and returns `additionalContext`

**Gap**: If Claude CLI shows a paste-preview confirmation dialog before the hook fires
(e.g., on older terminal configurations), the `__TASK__` trigger is queued but the
hook is never called. The prompt file remains on disk indefinitely, and the task hangs.

### Solution: File-Existence Polling

The hook's `unlink()` call creates an observable side-effect: the prompt file disappears
when and only when the hook fires. By polling `prompt_file.exists()` after sending the
trigger, we can detect whether the hook fired:

```
send_keys("__TASK__")
  → poll 30 × 100ms
    → file gone? → hook fired → success
    → still present after 3s? → paste-preview blocking
      → send_keys("", enter=True)
      → poll 30 × 100ms again
        → file gone? → success
        → still present? → warn and continue (completion strategy handles timeout)
```

### Implementation

In `ClaudeCodeAgent._dispatch_task()`, after `process.send_keys(keys_to_send)`:

```python
# After sending trigger, wait up to 3s for the prompt file to be consumed
for _ in range(30):  # 30 × 100ms = 3s
    await asyncio.sleep(0.1)
    if not prompt_file.exists():
        break  # UserPromptSubmit fired, prompt delivered
else:
    # File still exists → paste-preview blocking → send Enter
    logger.debug(
        "Agent %s: prompt file still present after 3s — sending Enter to dismiss paste-preview",
        self.id,
    )
    await loop.run_in_executor(None, self.process.send_keys, "")
    # Wait again for file to be consumed
    for _ in range(30):
        await asyncio.sleep(0.1)
        if not prompt_file.exists():
            break
    else:
        logger.warning(
            "Agent %s: prompt file still present after Enter retry — hook may not have fired",
            self.id,
        )
```

This only runs when `_cwd is not None` (the file-based injection path).
The fallback path (`_cwd is None`) is unaffected.

### Why Not Watchdog/inotify?

- `watchdog` and `aionotify` add external dependencies for a feature we can handle with
  a simple 100ms poll loop
- 100ms granularity is sufficient; the hook fires in <50ms on typical hardware
- `Path.exists()` is a single syscall (`stat()`), negligible overhead

## Part B: Context 4-Strategy Guide

### Problem

§11 层4 "コンテキスト4戦略ガイド" is an unimplemented candidate. Agents currently receive
general instructions in CLAUDE.md but no structured framework for managing their context window.
Without this guidance, agents tend to let context grow unbounded until `/summarize` is forced by
the ContextMonitor.

### Solution: Add Context Engineering Strategies Section to CLAUDE.md

The `_write_agent_claude_md()` method generates a per-agent CLAUDE.md at startup.
Adding a "## Context Engineering Strategies" section with the four strategies provides
structured guidance at zero runtime cost.

### The Four Strategies (from Zilliz / LangChain / R. Lance Martin)

| Strategy | When | How |
|----------|------|-----|
| **Write** | Produce output for future reference | NOTES.md, PLAN.md, scratchpad PUT |
| **Select** | Pull relevant context in | `context_files`, read NOTES.md at task start |
| **Compress** | Reduce token count | `/summarize` when context > 60% |
| **Isolate** | Split into sub-contexts | `/spawn-subagent` + worktree per sub-task |

### Role-Based Recommendations

- **Worker agents**: Write → Select → Compress sequence
- **Director agents**: Write (PLAN.md) → Select (worker results) → Isolate (spawn sub-agents)

### Research References

1. **Zilliz "Context Engineering Strategies for AI Agents"** (https://zilliz.com/blog/context-engineering-for-ai-agents):
   Defines Write/Select/Compress/Isolate as the four fundamental strategies.

   Establishes the four-strategy framework as industry standard (2025).

3. **R. Lance Martin "Context Engineering for Agents"** (https://rlancemartin.github.io/2025/06/23/context_engineering/):

### 指摘 1: `_wait_for_prompt_file_consumed` の `loop` 変数未使用

`_wait_for_prompt_file_consumed` 内で `send_keys` を呼ぶ際、`loop.run_in_executor` を使っているが、
`_dispatch_task` から `loop = asyncio.get_running_loop()` を引き回していない — `_wait_for_prompt_file_consumed` は
スタンドアロンの `async def` なので内部で `asyncio.get_running_loop()` を再度取得する必要がある。

> 対応: 実装コードを確認したところ、`_wait_for_prompt_file_consumed` では `loop = asyncio.get_running_loop()` を
> `_dispatch_task` から受け取るのではなく、内部で `asyncio.get_running_loop()` を呼んでいる。
> `send_keys` 呼び出しを `loop.run_in_executor(None, self.process.send_keys, "")` で実装しており、
> この `loop` は `_dispatch_task` のスコープ変数。`_wait_for_prompt_file_consumed` は `async def` なので
> 同一イベントループ内で実行される。対応済み — ただし、より明確にするため `_wait_for_prompt_file_consumed`
> 内部で独立して `loop = asyncio.get_running_loop()` を呼ぶように修正する。
