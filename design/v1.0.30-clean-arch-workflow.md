# v1.0.30 ‚Äî `POST /workflows/clean-arch` ‚Äî 4-Agent Clean Architecture Pipeline

## Problem Statement

The TmuxAgentOrchestrator has a growing library of multi-agent workflow templates
(tdd, fulldev, pair, debate, adr, delphi, redblue, socratic). However, the most
architecturally expressive workflow ‚Äî a Clean Architecture decomposition pipeline ‚Äî
has been deferred since v0.25.0 because of dependency on role templates and
sequential pipeline infrastructure. Both prerequisites are now satisfied (v1.0.27).

The `POST /workflows/clean-arch` endpoint should enable users to take a software
feature description and automatically decompose it into four architecture layers
using four sequential agents, each specialising in one layer and handing off to
the next via the shared scratchpad (Blackboard pattern).

## Solution

Implement `POST /workflows/clean-arch` as a 4-agent sequential pipeline:

```
domain-designer ‚Üí usecase-designer ‚Üí adapter-designer ‚Üí framework-designer
```

### Layer Responsibilities

1. **domain-designer**: Defines domain entities, value objects, aggregates,
   and domain events. Writes `DOMAIN.md`. Does NOT import any framework.

2. **usecase-designer**: Reads `DOMAIN.md` from scratchpad. Defines application
   use cases (interactors), input/output DTOs, and port interfaces.
   Writes `USECASES.md`.

3. **adapter-designer**: Reads `DOMAIN.md` + `USECASES.md`. Defines concrete
   adapters (repositories, presenters, controllers). Writes `ADAPTERS.md`.

4. **framework-designer**: Reads all previous layers. Writes the final
   `ARCHITECTURE.md` synthesising all layers plus specifying framework
   wiring (FastAPI/SQLite/CLI entry points). Writes executable skeleton
   Python files demonstrating the structure.

### Scratchpad Keys

- `{prefix}_domain`   ‚Äî domain-designer's DOMAIN.md
- `{prefix}_usecases` ‚Äî usecase-designer's USECASES.md
- `{prefix}_adapters` ‚Äî adapter-designer's ADAPTERS.md
- `{prefix}_arch`     ‚Äî framework-designer's ARCHITECTURE.md

### Response Structure

```json
{
  "workflow_id": "<uuid>",
  "name": "clean-arch/<feature>",
  "task_ids": {
    "domain_designer": "<task-id>",
    "usecase_designer": "<task-id>",
    "adapter_designer": "<task-id>",
    "framework_designer": "<task-id>"
  },
  "scratchpad_prefix": "cleanarch_<run-id[:8]>"
}
```

   communication via shared outputs (not direct P2P messaging) is the core
   handoff mechanism ‚Äî equivalent to our scratchpad Blackboard pattern.
   https://arxiv.org/abs/2507.19902

   boundaries map directly to AI agent context constraints. Per-layer agent
   separation reduces cognitive load and prevents boundary violations.
   https://notes.muthu.co/2025/11/the-architecture-is-the-prompt-guiding-ai-with-hexagonal-design/

**Reviewer**: v1.0.30 clean-arch workflow implementation review.

**Observations:**

1. **Model validation**: `CleanArchWorkflowSubmit` validator correctly rejects empty/whitespace
   `feature` strings. The `from pydantic import field_validator` inside the class body is an
   unusual pattern (same as existing models in the codebase) ‚Äî works but unusual in Python.

2. **Scratchpad key naming**: `cleanarch_XXXXXXXX` prefix uses underscores throughout
   (consistent with `fulldev_`, `pair_` prefixes). No slashes, which is correct.

3. **Dependency chain**: `domain ‚Üí usecase ‚Üí adapter ‚Üí framework` exactly mirrors Clean
   Architecture's dependency rule. Good.

4. **Prompt content**: framework-designer's prompt includes `ARCHITECTURE.md` synthesis +
   `main.py` skeleton instructions. This is more complex than other workflows but appropriate
   for the outermost layer synthesising all previous.

5. **`reply_to` routing**: Correctly passes `reply_to` only to the final task (framework_designer),
   consistent with how `fulldev` and `pair` handle it.

6. **Test coverage**: 37 tests covering model validation, HTTP auth, response structure,
   dependency chain, reply_to propagation, tag routing, prompt content, scratchpad keys,
   workflow ID uniqueness, and OpenAPI registration. Comprehensive.

7. **Potential improvement**: The adapter-designer prompt says "Adapters must NOT import
   domain entities directly except through Port interfaces" ‚Äî this is slightly imprecise
   (adapters do implement Port interfaces that deal with domain entities). Future refinement
   could clarify. Not a blocker.
