# v1.0.14 — orchestrator.py の完全 DI 化

## 目的

`orchestrator.py` のコンストラクタ内で直接インスタンス化されているインフラ依存を、
Protocol 経由で注入可能にする。これにより、ユニットテスト時に real tmux・real
ファイルシステムなしでオーケストレーターを構築できる。

---

## 今回のイテレーションで注入対象とする依存

### 優先度 A — テスタビリティ最大化 (今回実装)

**`ContextMonitor`** と **`DriftMonitor`** が最も重要。これらは `start()` / `stop()` を持つ
監視ループで、tmux ペインへの実際のアクセスを要求する。偽実装を注入できれば、
オーケストレーターの start/stop ライフサイクルを tmux なしでテストできる。

注入方法: `Monitor` Protocol を新設。`start()` / `stop()` を定義する。

```python
class Monitor(Protocol):
    def start(self) -> None: ...
    def stop(self) -> None: ...
```

`Orchestrator.__init__` のシグネチャ変更:

```python
def __init__(
    self,
    bus: Bus,
    tmux: "TmuxInterface",
    config: "OrchestratorConfig",
    worktree_manager: "WorktreeManager | None" = None,
    task_queue: "TaskQueue | None" = None,
    context_monitor: "Monitor | None" = None,   # NEW
    drift_monitor: "Monitor | None" = None,      # NEW
) -> None:
```

`None` の場合はデフォルト実装(`ContextMonitor`, `DriftMonitor`)を内部で生成する。
`factory.py` は変更なし — 既存の動作を壊さない。

### 優先度 B — 条件付き生成のため今回は据え置き

`AutoScaler`, `ResultStore`, `CheckpointStore`, `TelemetrySetup` は設定値に依存した
条件付き生成であり、現在のテストでもほとんど無効状態でテストされている。
コスト対効果が低いため今回のスコープ外とする。

### 優先度 C — 純粋コンポーネント (今回スコープ外)

`WorkflowManager`, `GroupManager`, `WebhookManager` は tmux / ファイルシステム依存がなく、
テスト時に特段の問題を起こさない。今回は据え置く。

## Protocol vs ABC の選択

**Protocol を採用する。**

理由:
1. **後付け適合 (retroactive conformance)** — 既存の `ContextMonitor`, `DriftMonitor` を
   変更せずに新 Protocol を満たせる。
2. **静的型チェッカー対応** — mypy が structural subtyping で検証。
3. **軽量** — ABC 継承階層なし。テスト用偽実装は2行で書ける。
4. **PEP 544** の推奨スタイル。

ABCの欠点: 既存実装クラスに `(ABC)` 継承を追加しなければならず、侵襲的変更になる。

## レビュー指摘

### DI 表面の適切さ

1. **指摘**: `Monitor` という単一 Protocol では不十分。`ContextMonitor` と `DriftMonitor` は
   `start()`/`stop()` 以外に異なるメソッド (`get_stats`, `all_stats` vs `get_drift_stats`,
   `all_drift_stats`) を持つ。単一 `Monitor` Protocol を型として使うと、Orchestrator の
   `get_agent_context_stats()` などが型エラーになる。
   > 対応: `ContextMonitorProtocol` と `DriftMonitorProtocol` に分割する。各 Protocol は
   > それぞれの公開メソッド全体をカバーする。

2. **指摘**: `Protocol` に `start()`/`stop()` 以外のメソッドも含めるなら、偽実装も
   それらを実装しなければならない。テストが複雑になる。
   > 対応: Null Object パターンで `NullContextMonitor` / `NullDriftMonitor` を同ファイルに
   > 定義。`start()` = noop, `stop()` = noop, `get_stats()` = `None`, `all_stats()` = `[]`,
   > `get_drift_stats()` = `None`, `all_drift_stats()` = `[]`。
   > 単純で明示的、テストコードで毎回書く必要がない。

3. **指摘**: デフォルト `None` → 内部生成パターンは型安全性が下がる
   (`Optional[ContextMonitorProtocol]` → 内部で非 None 保証だが型は Optional のまま)。
   > 対応: インスタンス変数の型アノテーションを `ContextMonitorProtocol` に変更。
   > `__init__` の引数のみ `| None` にして、内部では None 時にデフォルト生成してから
   > 非 Optional として保持する。

4. **指摘**: `factory.py` で `ContextMonitor` / `DriftMonitor` を直接渡すことも可能にすると
   テストデモが書きやすくなる。
   > 対応: factory.py は変更しない（デフォルト生成で十分）。必要な場合のみ注入するのが
   > 実際のユースケース（ユニットテスト）。

### Protocol vs ABC の選択

5. **指摘**: 設計書に Protocol 採用の根拠があるのは良い。追記：ABCを使っても型安全だが
   既存実装クラスに侵襲的変更が必要。Protocol が明らかに優位。
   > 対応: 設計書の判断を支持。変更なし。

### スコープ

6. **指摘**: `WorkflowManager`, `GroupManager`, `WebhookManager` を今回スコープ外にする判断
   は妥当。これらのインスタンス生成は副作用が小さくテストに支障がない。
   > 対応: 判断維持。今後 DESIGN.md §11 に候補として追加する。
