# v1.0.26 — Stop Hook 不発火の根本原因調査・修正

## Step 0 — 選定理由

**選択: Stop Hook 不発火の根本原因調査・修正 (v1.0.26)**

v1.0.23 以降、デモで Stop Hook が間欠的に不発火となり、エージェントが watchdog タイムアウトまで `❯` プロンプトで停止し続ける問題が継続している。v1.0.24 と v1.0.25 では「手動 nudge」ワークアラウンドで回避していたが、根本原因を修正しないと信頼性の低いデモが続く。本イテレーションでこの問題を確実に解決する。

**選択しなかった候補:**
- チェックポイント永続化 (SQLite): 実装規模が大きく本イテレーションには不適。
- `ProcessPort` 抽象インターフェース: テスト基盤の全面改修が必要で規模が大きい。
- OpenTelemetry GenAI Semantic Conventions: 外部インフラ依存が増えデモが複雑化する。

## Step 1 — 調査結果 (Research)

### Query 1: "Claude Code hooks settings.local.json Stop hook resolution path 2025"

**主要知見**:
- Claude Code のフック設定ファイル解決順: `~/.claude/settings.json` (グローバル) → `.claude/settings.json` (プロジェクト) → `.claude/settings.local.json` (ローカル上書き) → マネージドポリシー → プラグイン `hooks/hooks.json`
- **重要**: フック設定はセッション起動時にスナップショットが取られ、セッション中は変更が即反映されない
- 設定ファイルが外部から変更された場合、Claude Code は警告を出し、`/hooks` メニューでのレビューを要求する

**References**:
- "Hooks reference - Claude Code Docs", https://code.claude.com/docs/en/hooks (2025)
- GitHub disler, "claude-code-hooks-mastery", https://github.com/disler/claude-code-hooks-mastery (2025)
- "[BUG] Hooks not loading from settings.json", anthropics/claude-code Issue #11544, https://github.com/anthropics/claude-code/issues/11544

### Query 2: "Claude Code hooks Stop hook not firing intermittent worktree configuration 2025"

**主要知見**:
- **Stop Hook が「silent tool stop」でも不発火**: Claude がツール結果 (Exit code 1 等) を受け取った後、テキスト出力なしに停止する場合、Stop hook が発火しない既知のバグ (Issue #29881)
- **ワークツリー環境での Stop Hook 障害**: git worktree 内で Claude Code を実行すると、トランスクリプトファイルが見つからないために Stop hook が失敗する (claude-mem Issue #1234/#1276)
- **WorktreeCreate/Remove フックが呼ばれない**: Claude Desktop でワークツリー作成時に WorktreeCreate フックが呼ばれないバグ (Issue #29716)
- `ConfigChange` は設定ファイル変更時に発火するが、これはフックの変更が適用されるか「ブロック」できるだけで、スナップショット更新の自動承認にはならない

**References**:
- "Stop hook not fired when Claude stalls mid-turn after tool result", anthropics/claude-code Issue #29881, https://github.com/anthropics/claude-code/issues/29881 (2025)
- "Stop hook fails when Claude Code runs in a git worktree", thedotmack/claude-mem Issue #1276, https://github.com/thedotmack/claude-mem/issues/1276
- "Stop hook crashes in git worktrees: transcript path not found", thedotmack/claude-mem Issue #1234, https://github.com/thedotmack/claude-mem/issues/1234
- "WorktreeCreate / Remove hooks not called in Claude Desktop", anthropics/claude-code Issue #29716, https://github.com/anthropics/claude-code/issues/29716

### Query 3: "Claude Code 'settings.local.json' hooks snapshot startup session reading order project user 2025"

**主要知見**:
- **公式ドキュメント明言 (Line 431 of hooks reference)**:
  > "Direct edits to hooks in settings files don't take effect immediately. Claude Code captures a snapshot of hooks at startup and uses it throughout the session. This prevents malicious or accidental hook modifications from taking effect mid-session without your review. If hooks are modified externally, Claude Code warns you and requires review in the `/hooks` menu before changes apply."
- `ConfigChange` フックは `source` フィールドとして `local_settings` を持ち、設定ファイル変更時に発火するが、フックの変更が「自動的に適用される」わけではない
- **設定ファイルへの動的書き込みは機能しない**: セッション起動後に `settings.local.json` を書き込んでも、そのセッションのフックスナップショットには影響しない

- "A developer's guide to settings.json in Claude Code (2025)", https://www.eesel.ai/blog/settings-json-claude-code

## 根本原因分析

### 確定した根本原因: フックスナップショット機構

```
現状のフロー (壊れている):
1. ClaudeCodeAgent.start() → claude プロセス起動
2. claude が settings.local.json をスナップショット → フックなし
3. タスク dispatch
4. NudgingStrategy.on_task_dispatch() → settings.local.json に Stop hook を書き込む  ← ここが問題
5. claude は既にスナップショット済みなのでファイル変更を無視
6. Stop hook が発火しない
```

**公式ドキュメントの確認**: Claude Code はセッション起動時にフックのスナップショットを取り、セッション中は外部からの変更を無視する。`on_task_dispatch()` でファイルを書き込んでも、既に起動している claude プロセスはそれを読まない。

### 仮説の検証

| 仮説 | 状態 | 根拠 |
|------|------|------|
| 1. Wrong cwd | 棄却 | `_cwd` は `_setup_worktree()` の戻り値で正しい。`on_task_dispatch(self._cwd, task.id)` の呼び出しは問題なし |
| 2. settings.local.json が間違ったディレクトリに書かれる | 棄却 | `_write_settings(cwd, ...)` は `cwd / ".claude" / "settings.local.json"` に正しく書く |
| 3. **Claude Code が worktree の settings.local.json を読まない** | **確定** | スナップショット機構により、セッション起動後の変更は無視される |
| 4. Env var 不足 | 部分的に関係あり | `TMUX_ORCHESTRATOR_API_KEY` は tmux 環境変数として設定されているが、hook が発火しない問題の主因ではない |
| 5. Hook format 変更 | 棄却 | `type: "command"` + `hooks` ネスト構造は現行ドキュメントの形式と一致する |

## 修正方針

### Fix: Stop hook を `on_start()` で書き込む (セッション起動前)

現状の `on_task_dispatch()` での書き込みを、`on_start()` での書き込みに移動する。

**問題**: task_id が `on_start()` 時点では不明。
**解決策**: task_id なしの URL をセッション起動前に書き込む。エンドポイントは既に task_id なしの呼び出しを許容している (app.py line 1534: "Calls without a task_id...are accepted as before")。

```python
# 修正後:
def on_start(self, cwd: Path) -> None:
    # セッション起動前に Stop hook を書き込む (スナップショット前)
    url = f"{self._web_base_url}/agents/{self._agent_id}/task-complete"
    _write_settings(cwd, _build_stop_hook_settings(url))

def on_task_dispatch(self, cwd: Path, task_id: str) -> None:
    # task_id による URL 更新は不要 (スナップショット後は無視される)
    # ログだけ記録
    pass
```

**task_id スコーピングの代替**: `on_task_dispatch()` での URL 上書きはもはや機能しないため、stale hook 防止は別の方法で行う:
- エンドポイント側で `agent._current_task` が None の場合はスキップ (既に実装済み、409 返却)
- `stop_hook_active=True` の場合はスキップ (既に実装済み)
- task_id のミスマッチ検出は Stop hook からは URL で渡せないため削除 (ただし explicit `/task-complete` からの task_id は引き続き有効)

### また、on_stop() の変更

`on_stop()` は引き続き `settings.local.json` を削除する (エージェント停止後に stale hook が残らないよう)。

## レビュー指摘

### レビュー結果 (2026-03-09, context-free review)

**指摘1**: `on_task_dispatch()` を完全な no-op にすることで、task_id スコーピングが失われる。stale hook 防止が弱体化しないか？

> 対応: `stop_hook_active` フラグによる区別で十分。Stop hook からの呼び出しは常に `stop_hook_active` キーを含み、これにより explicit `/task-complete` と区別される。task_id なしのコールは endpoint 側で `agent._current_task` の null チェックで処理済み（409 返却）。意図的な degradation として受容可能。

**指摘2**: `on_start()` で書かれた URL に task_id がないため、もし複数タスクが同じエージェントに連続して dispatch された場合、旧タスクの Stop hook が新タスクに対して発火する可能性があるか？

> 対応: これは問題にならない。Stop hook が発火するのは Claude が response turn を終えた時のみ。agent は一度に1タスクしか実行しない（`_run_loop` がキューから順次 dequeue）。新タスクが始まる前に旧タスクは完了しているため、クロスタスク汚染は起きない。

**指摘3**: デモで "task_id=unknown" ログが出ている。これは混乱を招かないか？

> 対応: ログレベルを INFO から DEBUG に下げることを検討すべきだが、本イテレーションでは機能修正が優先。build-log の次への課題として記録済み。
