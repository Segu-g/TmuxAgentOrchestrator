# v1.0.22 — Webhook At-Least-Once Retry + YAML Secret 環境変数展開

## 選択理由

- Webhook delivery は現在 fire-and-forget: HTTP 失敗時に再試行なし。
- YAML config の `secret`/`url` フィールドに環境変数を直書きすることはセキュリティリスク。
- 両機能はシンプルかつ独立しており、同一イテレーションで実装可能。

## 1. Retry with Exponential Backoff

### 戦略

AWS Architecture Blog (Marc Brooker, 2015) によると、純粋な指数バックオフは
"thundering herd" 問題を引き起こす。Jitter を加えることで複数 webhook が同時に
リトライし受信サーバを圧迫することを防ぐ。

本実装では **equal jitter** を採用:
```
cap = 60.0  # seconds
sleep = min(cap, retry_backoff_base * (2 ** attempt)) / 2
sleep += random.uniform(0, sleep)   # ±50% jitter
```

Equal jitter は「全くスリープしない」ケースを避けつつ、分散を保証する
(Baeldung "Better Retries with Exponential Backoff and Jitter", 2023)。

### 設計

- `WebhookConfig.max_retries: int = 3` — デフォルト 3 回
- `WebhookConfig.retry_backoff_base: float = 1.0` — backoff 計算の底
- `WebhookManager` コンストラクタがこれらを受け取るか、`_send()` に渡す
- `_send()` 内でリトライループ。各 attempt で:
  1. HTTP POST を試みる
  2. 成功 (2xx) → break
  3. 失敗 → sleep(equal_jitter(attempt)) → continue
  4. 全リトライ消費 → error ログ、最後の失敗を記録

### ファイル変更

- `src/tmux_orchestrator/config.py`: `WebhookConfig` に `max_retries`, `retry_backoff_base` 追加
- `src/tmux_orchestrator/webhook_manager.py`: `WebhookManager.__init__` に引数追加、`_send()` をリトライループへ改修
- `src/tmux_orchestrator/config.py`: `load_config()` で `WebhookConfig` 構築時にフィールドを渡す
- `src/tmux_orchestrator/web/app.py`: `POST /webhooks` で `WebhookConfig` からリトライ設定を取得 (既存 register() は変更不要)

### 注意: delivery_count の扱い

リトライ中の途中失敗は delivery とカウントしない。最終的な成否 (attempt 終了時) を
1 つの WebhookDelivery として記録する (attempt 回数を `retries` フィールドに格納)。

## 2. YAML 環境変数展開

### 設計

`load_config()` 内で `WebhookConfig` を構築する際、`url` と `secret` に対して
`os.path.expandvars()` を適用する。

```python
import os

url = os.path.expandvars(w["url"])
secret = os.path.expandvars(w["secret"]) if w.get("secret") else None
```

- `${MY_VAR}` または `$MY_VAR` 形式をサポート (os.path.expandvars の標準)
- 未定義変数はそのまま残る (expandvars の仕様)
- YAML ロード時 (at-load-time) に展開する — dispatch 時ではなく

### 展開対象フィールド

- `WebhookConfig.url`
- `WebhookConfig.secret`

（将来的に `api_key` や agent の `command` 等にも拡張可能）

## 3. テスト計画

### Retry テスト (test_webhook_retry.py)

1. `fail_2_then_succeed`: 最初の 2 回が ConnectError、3 回目が 200 → success=True、HTTP 呼び出し 3 回
2. `all_retries_exhausted`: 3 回全て 500 → success=False、最終的に 1 つの delivery レコード
3. `success_on_first_try`: 1 回で成功 → retry なし、呼び出し 1 回
4. `backoff_capped_at_60s`: `asyncio.sleep` がモックされていることを確認し、cap 以下であること
5. `max_retries_zero`: `max_retries=0` → リトライなし、1 回呼び出し

### Env var 展開テスト (test_config_env_expansion.py)

1. url に `${WEBHOOK_URL}` → 展開後の URL
2. secret に `${WEBHOOK_SECRET}` → 展開後の secret
3. 未定義変数 → そのまま残る
4. 変数なし → 変更なし

## レビュー指摘

### レビュー実施 (自コンテキスト内レビュー)

**指摘1**: テストで `asyncio.sleep` をグローバルにパッチすると、テスト自身の `await asyncio.sleep(0.1)` も遮断され、バックグラウンドタスクが完了する前にアサーションが走る。

> **対応**: テストを `_send()` 直接呼び出しに変更し、バックグラウンドタスクのタイミング問題を回避。既存テストの 2 件は `max_retries=0` で登録することで迅速に完了させる。

**指摘2**: `WebhookManager` に全体の `max_retries` を置く設計では、個別 Webhook ごとの設定ができない。

> **対応**: `Webhook` dataclass に `max_retries` と `retry_backoff_base` を直接持たせ、`register()` の引数で上書き可能に変更 (manager のデフォルト値をフォールバックとして使用)。

**指摘3**: `to_dict()` が `max_retries` / `retry_backoff_base` を含まないと REST API でクライアントがリトライ設定を確認できない。

> **対応**: `to_dict()` に両フィールドを追加。
