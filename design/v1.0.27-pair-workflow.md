# Design: v1.0.27 — POST /workflows/pair — PairCoder (Navigator + Driver) Workflow

## Problem Statement

The TmuxAgentOrchestrator workflow suite (debate, delphi, redblue, socratic, adr, fulldev)
lacks a pair-programming workflow. The Navigator/Driver XP pattern is a well-researched
technique where one agent focuses on high-level planning (Navigator) while the other
focuses on implementation (Driver). This split improves code quality by enforcing explicit
planning before coding and providing clear handoff semantics via the scratchpad.

## Proposed Solution

Add `POST /workflows/pair` — a 2-agent sequential workflow:

1. **navigator**: receives the task description, writes `PLAN.md` (architecture, interfaces,
   acceptance criteria, step-by-step guide), stores it in the scratchpad.
2. **driver**: reads the navigator's plan, implements the code, writes tests, runs them,
   writes `driver_summary.md` with pass/fail status.

### Scratchpad Keys

- `{prefix}_plan`   — navigator's PLAN.md content
- `{prefix}_result` — driver's driver_summary.md content

### Route

```
POST /workflows/pair
```

### Request Body

```json
{
  "task": "implement a binary search function with full test coverage",
  "navigator_tags": ["navigator"],
  "driver_tags": ["driver"],
  "reply_to": null
}
```

### Response

```json
{
  "workflow_id": "...",
  "name": "pair/implement a binary search function with full",
  "task_ids": {"navigator": "...", "driver": "..."},
  "scratchpad_prefix": "pair_abc12345"
}
```

## Implementation Plan

1. **`PairWorkflowSubmit` model** — `task`, `navigator_tags`, `driver_tags`, `reply_to`
2. **`POST /workflows/pair` endpoint** — navigator prompt → driver prompt pipeline
3. **`tests/test_workflow_pair.py`** — unit tests for the endpoint
4. **OpenAPI snapshot update** — `UPDATE_SNAPSHOTS=1 pytest tests/test_openapi_schema.py`

## レビュー指摘

### レビュー指摘 1: 重複ヘルパー関数
`_ctx_snippet`, `_write_snippet`, `_read_snippet` が各ワークフローハンドラーで重複している。
> 対応: 既存のワークフロー (socratic, redblue, delphi) と同じパターンを採用。今後の共通化リファクタリング候補として§11に記録する。本イテレーションでは一貫性を優先する。

### レビュー指摘 2: driver の pytest コマンドが失敗した場合の扱い
`python -m pytest test_*.py -v 2>&1 | tee test_output.txt || true` の `|| true` でテスト失敗を握りつぶす可能性。
> 対応: `|| true` は意図的。テスト失敗をタスク失敗にしないのはワークフロー設計上正しい（Driver が `driver_summary.md` に FAIL を記録して完了するのが正常フロー）。`driver_summary.md` の `Status: PASS | FAIL` フィールドが実際の結果を伝える。

### レビュー指摘 3: Navigator tags と Driver tags が同一の場合の競合
同じタグを持つエージェントが 1 つしかない場合、navigator タスクが占有してしまい driver タスクが waiting になる。
> 対応: これは設計上の期待動作（`depends_on` チェーンにより navigator 完了後に driver が実行される）。ユーザーが同一エージェントを navigator と driver の両方に指定しても、エージェントが IDLE になった後に driver タスクが実行されるため問題ない。
