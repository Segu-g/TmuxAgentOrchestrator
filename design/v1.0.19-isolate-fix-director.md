# v1.0.19 設計記録 — isolate:false context file 競合修正 + Director Pattern 信頼性検証

## 選択理由

v1.0.18 デモで発見されたバグ修正が最優先：

1. **isolate:false 競合** — 複数エージェントが同じ cwd を共有する場合、`__orchestrator_context__.json` が
   最後に起動したエージェントで上書きされる。最初のエージェントが `/task-complete` を呼ぶと、
   ファイルに書かれた agent_id は別エージェントのものになっており、409 エラーを引き起こす。

2. **Director Pattern 信頼性検証** — Director エージェントが Stop hook ではなく REST エンドポイント
   `POST /agents/{id}/task-complete` を直接呼ぶパターン (ExplicitSignalStrategy) の E2E 検証が
   不十分だった。

---

## 1. Root Cause Analysis — Context File Race

### 問題の流れ

```
agent-a start() → _write_context_file(cwd) → writes {agent_id: "agent-a"} to cwd/__orchestrator_context__.json
agent-b start() → _write_context_file(cwd) → writes {agent_id: "agent-b"} to cwd/__orchestrator_context__.json  ← 上書き
...
agent-a /task-complete → reads __orchestrator_context__.json → agent_id = "agent-b" → POST /agents/agent-b/task-complete → 409 mismatch
```

### 根本原因

`_write_context_file()` は `base.py` で実装されており、常に同名ファイル
`__orchestrator_context__.json` に書き込む。複数エージェントが同一 cwd を使う場合
（`isolate: false`）、後から起動したエージェントが前のファイルを上書きする。

### 影響範囲

- `isolate: false` のエージェントが 2 台以上同時に動く場合のみ発生
- `isolate: true`（デフォルト）の場合は各エージェントが独自 worktree を持つため無問題
- 現在のデモはすべて `isolate: true` を使用しているため実害はなかった

## 2. 修正方針 — Per-Agent Context File Naming

### Option A: per-agent ファイル名（採用）

各エージェントが自分専用のコンテキストファイルを書く：

```
cwd/__orchestrator_context__agent-a__.json   # agent-a 専用
cwd/__orchestrator_context__agent-b__.json   # agent-b 専用
cwd/__orchestrator_context__.json            # 後方互換 (agent-a のコピー、または最後に書いたもの)
```

スラッシュコマンドは `$TMUX_ORCHESTRATOR_AGENT_ID` 環境変数で自分のファイルを特定できる：

```python
agent_id = os.environ.get("TMUX_ORCHESTRATOR_AGENT_ID", "")
ctx_path = Path(f"__orchestrator_context__{agent_id}__.json") if agent_id else None
if ctx_path is None or not ctx_path.exists():
    ctx_path = Path("__orchestrator_context__.json")
ctx = json.loads(ctx_path.read_text())
```

### Option B: per-agent サブディレクトリ（不採用）

```
cwd/.orchestrator/agent-a/context.json
cwd/.orchestrator/agent-b/context.json
```

問題：既存の slash command が `Path("__orchestrator_context__.json")` をハードコードしており、
変更箇所が多くなる。

### 後方互換性計画

- `isolate: true`（単一エージェント per cwd）: 新旧どちらのファイル名も存在するため完全互換
- `isolate: false`（共有 cwd）: 新フォーマットのみ完全動作。古いファイル名は最後の書き込みを保持するが
  スラッシュコマンドは per-agent ファイルを優先的に読む

### API キーファイルの同様修正

`__orchestrator_api_key__` も同様の競合が起きる可能性あり。
`__orchestrator_api_key__{agent_id}__` を書き、後方互換ファイルも維持する。

## 3. Director Pattern アーキテクチャ

### ExplicitSignalStrategy vs NudgingStrategy

| 特徴 | NudgingStrategy (WORKER) | ExplicitSignalStrategy (DIRECTOR) |
|---|---|---|
| 完了検出 | Stop hook → nudge → `/task-complete` slash command | `POST /agents/{id}/task-complete` REST直接呼び出し |
| タスク期間 | 単一 Claude レスポンスターン | 複数ターン（worker 待機中） |
| Stop hook | 各レスポンス後にナッジ送信 | タスク_id ミスマッチで自動スキップ |
| 依存 | agent_plugin/commands/task-complete.md | curl コマンド (CLAUDE.md に記載) |

### Director 完了フロー

```
Director CLAUDE.md に curl コマンド記載
  ↓
Director が全 worker 完了を確認 (GET /agents)
  ↓
Director が curl -X POST /agents/{director_id}/task-complete を実行
  ↓
agent_task_complete エンドポイント: body に stop_hook_active なし → 明示的完了
  ↓
handle_output() → RESULT publish → orchestrator が IDLE に遷移
```

## 4. 研究結果

### 検索クエリ 1: per-process config file naming Linux Unix 2024

- Linux/Unix では PID をファイル名に含めるパターン（`/run/nginx.pid`）が一般的
- SEI CERT C では PID ベースのファイル名は予測可能で攻撃可能と警告
- agent_id は UUIDv4 ベースであるため予測不可能 → セキュア

参照:
- SEI CERT C, "POS38-C. Beware of race conditions when using fork and file descriptors"
  https://wiki.sei.cmu.edu/confluence/display/c/POS38-C.+Beware+of+race+conditions+when+using+fork+and+file+descriptors

### 検索クエリ 2: concurrent file write race condition shared working directory naming

- TOCTOU 競合を避けるには原子的操作（`mkstemp`, `os.open` with O_CREAT|O_EXCL）を使う
- ファイルロックより per-process ファイル名のほうが単純で確実
- Apple Developer Library "Race Conditions and Secure File Operations" (2024):
  ファイル名の予測不可能性が race-free アクセスの基礎

参照:
- Apple Developer, "Race Conditions and Secure File Operations"
  https://developer.apple.com/library/archive/documentation/Security/Conceptual/SecureCodingGuide/Articles/RaceConditions.html
- dwheeler, "Avoid Race Conditions"
  https://dwheeler.com/secure-programs/Secure-Programs-HOWTO/avoid-race.html

### 検索クエリ 3: director worker pattern REST task completion explicit signal

- GigaSpaces Master-Worker Pattern: "manager dispatches tasks and tracks completion state"
  完了は worker からの明示的通知が best practice
  https://docs.gigaspaces.com/solution-hub/master-worker-pattern.html
- F Prime Manager/Worker Pattern (JPL): Director が REST API で worker 状態を polling するか、
  worker が completion event を送るかの 2 通り
  https://fprime.jpl.nasa.gov/devel/docs/user-manual/design-patterns/manager-worker/
- Camunda External Task Pattern: worker が "complete" エンドポイントを明示的に POST
  https://medium.com/@dashedsouvik/camunda-external-task-pattern-fd84a29d9d3e

## レビュー指摘

### 指摘 1: 後方互換ファイルの書き込みタイミング

後方互換ファイル `__orchestrator_context__.json` を毎回上書きすると、isolate:false で
複数エージェントが起動する場合、依然として最後の起動エージェントの内容が残る。

> 対応: 後方互換ファイルは **書かない** ことにする。代わりに、各 slash command が
> `$TMUX_ORCHESTRATOR_AGENT_ID` でまず per-agent ファイルを探し、
> 見つからない場合のみ従来ファイルにフォールバックする。
> isolate:true（単一エージェント per cwd）では per-agent ファイルのみ存在し正常動作。
> 旧バージョンとの互換性（ファイルが存在しない場合）も維持される。

### 指摘 2: API キーファイルの chmod 確認

per-agent API キーファイル `__orchestrator_api_key__{id}__` も chmod 600 で書くこと。

> 対応: `_write_api_key_file()` の既存実装は `os.open(flags, 0o600)` を使用しており、
> per-agent ファイル名変更後も同じコードパスを通るため自動的に対応される。

### 指摘 3: slash command の環境変数フォールバック

`$TMUX_ORCHESTRATOR_AGENT_ID` が設定されていない古い環境（env var が存在しない）でも
動作すること。

> 対応: `agent_id = os.environ.get("TMUX_ORCHESTRATOR_AGENT_ID", "")` → 空文字列の場合は
> per-agent ファイルを試みずに従来ファイルに直接フォールバックする。

### 指摘 4: Director デモの /task-complete 呼び出し確認

Director が curl で `POST /agents/{id}/task-complete` を呼ぶとき、body に
`stop_hook_active` キーを含めないこと（含むと nudge 扱いになる）。

> 対応: CLAUDE.md のサンプル curl コマンドは `{"output": "..."}` のみを送信する設計になっており、
> `stop_hook_active` は含まれない。`agent_task_complete` エンドポイントのロジックを確認済み。
