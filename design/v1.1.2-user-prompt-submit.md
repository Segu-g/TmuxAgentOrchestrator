# v1.1.2 — UserPromptSubmit フックによるタスクプロンプト注入

**作成日**: 2026-03-09

## 問題

v1.1.1 の `[Pasted text #1]` 修正は `TmuxInterface.send_keys()` 内のポーリングによる workaround。
長いプロンプトが `send_keys` を通過する限り paste-preview は発生しうる。根本解決は
長いプロンプトを `send_keys` に渡さない設計にすること。

## UserPromptSubmit フック仕様調査

### 公式ドキュメント (https://code.claude.com/docs/en/hooks)

`UserPromptSubmit` フックは以下の仕様を持つ:

**発火タイミング**: ユーザーがプロンプトを送信したとき、Claude 処理前

**stdin JSON**:
```json
{
  "session_id": "abc123",
  "transcript_path": "/Users/.../.claude/.../transcript.jsonl",
  "cwd": "/path/to/agent/workdir",
  "permission_mode": "default",
  "hook_event_name": "UserPromptSubmit",
  "prompt": "ユーザーが送信したテキスト"
}
```

**stdout の扱い (exit 0)**:
- plaintext stdout → Claude のコンテキストとして追加（プロンプトの置換ではない）
- JSON + `hookSpecificOutput.additionalContext` → より離散的にコンテキストとして追加

**exit code 2**: プロンプトをブロック・消去

**マッチャー**: 非対応。常に発火。

### 重要な制約

`UserPromptSubmit` の stdout は**プロンプトを置換しない**。元のプロンプトは保持されたまま、
stdout がコンテキストとして**追加される**。

egghead.io の記事は「stdout でプロンプトを書き換えられる」と述べるが、公式ドキュメントは
「added as context along with your prompt」と明記している。公式ドキュメントを優先する。

### stdout injection が実現できること

短いトリガー文字列をプロンプトとし、実際のタスク内容を `additionalContext` として渡すことができる:

```
ユーザープロンプト: "__TASK__"
additionalContext: "（実際の長いタスクテキスト）"
```

Claude はこの両方を受け取り、タスクを実行できる。

## 設計

### アーキテクチャ

```
ClaudeCodeAgent._dispatch_task(task)
  ↓ 1. プロンプトをファイルに書き込む
  write_text("__task_prompt__{agent_id}__.txt", task.prompt)
  ↓ 2. 短いトリガーのみ send_keys (paste-preview なし)
  process.send_keys("__TASK__")
  ↓ 3. UserPromptSubmit フック起動
  stdin: {"cwd": "/path/to/worktree", "prompt": "__TASK__", ...}
  ↓ 4. フックがファイルを読み込んでコンテキスト注入
  read("__task_prompt__{agent_id}__.txt") → delete → output additionalContext
  ↓ 5. Claude が受け取る
  prompt: "__TASK__"
  context: "（実際のタスクテキスト）"
```

### ファイル命名

- `__task_prompt__{agent_id}__.txt` — agent_id 付きで名前空間化
  - 非 isolate エージェント（複数エージェントが同じ cwd を共有）でも衝突しない
  - 例: `__task_prompt__worker-1__.txt`

### フックスクリプト (`user-prompt-submit.py`)

```python
#!/usr/bin/env python3
"""UserPromptSubmit hook: inject task prompt from __task_prompt__<id>__.txt."""
import json, sys, os
from pathlib import Path
import glob

data = json.load(sys.stdin)
cwd = Path(data.get("cwd", "."))

# Find __task_prompt__*.txt in cwd
pattern = str(cwd / "__task_prompt__*.txt")
matches = sorted(glob.glob(pattern))

if not matches:
    sys.exit(0)  # pass-through: no task prompt file

# Use the first (and normally only) match
prompt_file = Path(matches[0])
try:
    task_prompt = prompt_file.read_text()
    prompt_file.unlink()  # delete after reading (consume once)
except OSError:
    sys.exit(0)  # file disappeared between glob and read — ignore

# Output additionalContext so Claude receives the actual task
output = {
    "hookSpecificOutput": {
        "hookEventName": "UserPromptSubmit",
        "additionalContext": task_prompt,
    }
}
print(json.dumps(output))
sys.exit(0)
```

async def _dispatch_task(self, task: Task) -> None:
    ...
    # Write prompt to file for UserPromptSubmit hook injection
    if self._cwd is not None:

## フォールバック

- `self._cwd is None` の場合（cwd がない）: 従来通りプロンプトを直接 send_keys する
- v1.1.1 の paste-preview ポーリングはフォールバックとして維持する（フックが無効の場合に備える）

## テスト方針

### 単体テスト

1. **`test_user_prompt_submit_hook.py`**:
   - フックスクリプトが `__task_prompt__*.txt` を読み込み削除して additionalContext を出力する
   - ファイルが存在しない場合 exit 0 (pass-through)
   - 複数のファイルが存在する場合、最初のものを使用する
   - OSError（ファイルが消えた場合）は無視して exit 0

2. **`test_dispatch_task_prompt_file.py`**（または既存テストへの追加）:
   - `_dispatch_task()` が `__task_prompt__{agent_id}__.txt` を書き込む
   - `send_keys` にはトリガー文字列のみが渡される（プロンプト本文でない）
   - `cwd is None` の場合はフォールバック動作

### E2E デモ

- 1000文字超えの長いプロンプトを送信
- paste-preview (`[Pasted text #N]`) が発生しないことを確認
- 2+ エージェントによる並列タスク実行
