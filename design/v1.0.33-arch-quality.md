# v1.0.33 — Architecture Quality Hardening

## Problem

Three quality gaps identified in §11:

1. **`watchdog_poll` no cross-field validation**: Setting `watchdog_poll > task_timeout / 3`
   silently causes the watchdog to fire too infrequently, leading to tasks timing out much
   later than configured. Default of 10.0s was also too aggressive for production workloads.

2. **No Use Case Interactor layer**: FastAPI handlers (`web/app.py`) contain business
   logic directly — they call `orchestrator.submit_task(...)` and `orchestrator.cancel_task(...)`
   inline, mapping Pydantic models to domain calls and back. This violates Clean Architecture's
   Use Case layer and makes the logic unreachable from CLI/TUI without duplicating code.

3. **`AgentStatus` state machine not property-tested**: The `AgentStatus` enum has 5 states
   and multiple valid/invalid transitions, but no stateful Hypothesis tests verify that arbitrary
   transition sequences maintain invariants (non-negative task count, busy-iff-task-in-flight, etc.).

## Solution

### Feature 1: `watchdog_poll` validator + new default

Add `OrchestratorConfig.__post_init__` that enforces:
```
if watchdog_poll < task_timeout:
    assert watchdog_poll <= task_timeout / 3
```

Only enforces when `watchdog_poll < task_timeout` — when greater-or-equal the watchdog
is effectively disabled (a valid test pattern). Change default from 10.0 → 30.0 seconds.
Add `watchdog_poll` to `load_config()` (was missing).

### Feature 2: `application/use_cases.py`

Introduce `TaskService` protocol (PEP 544 structural subtyping), `SubmitTaskDTO`,
`SubmitTaskResult`, `SubmitTaskUseCase`, `CancelTaskDTO`, `CancelTaskResult`,
`CancelTaskUseCase`. FastAPI handlers can delegate to these use cases; the use cases
are independently testable without HTTP.

### Feature 3: `AgentStatus` Hypothesis stateful tests

`AgentStatusMachine` (Hypothesis `RuleBasedStateMachine`) models 11 state transitions
and 5 invariants:
- `status_is_valid_enum_member`
- `task_count_is_non_negative`
- `dispatched_ge_completed_plus_errors`
- `busy_iff_task_in_flight`
- `stopped_implies_no_tasks`

## Test Plan

- `test_watchdog_poll_validator.py`: 16 tests covering default value, valid configs,
  invalid configs, fail-fast behaviour
- `test_use_cases.py`: 28 tests covering SubmitTaskUseCase, CancelTaskUseCase,
  DTO-to-domain mapping, was_running detection, protocol structural check
- `test_agent_status_stateful.py`: 1 Hypothesis stateful test (200 examples × 20 steps)

## Reviewer Review

## レビュー指摘

1. **`SubmitTaskUseCase.execute`**: `metadata=dto.metadata or None` は空辞書 `{}` を `None` に変換するため、意図的に空 metadata を渡したい場合に副作用が生じる可能性がある。同様に `depends_on=dto.depends_on or None` は空リストを `None` にする。
2. **`CancelTaskUseCase`**: `agent_obj._current_task` (private field) へのアクセスは既存の `web/app.py` ハンドラーと同じパターンだが、`ProcessPort` 抽象化が完成すれば公開 API に移行すべき。
3. **`test_agent_status_stateful.py`**: `drain_while_busy` で `task_count` を即座にデクリメントするのは「タスクが drain 中に完了する」という暗黙的モデル — コメントで明示すべき。
4. **`__post_init__` の `task_timeout` 型**: `task_timeout: int` と `watchdog_poll: float` の比較は Python では問題ないが、`task_timeout=0` の場合 `max_watchdog_poll = 0/3 = 0` となり、`watchdog_poll=0.0` 以外はすべてエラーになる。ゼロ値は別途扱うべき。

> 対応:
> 1. 既存の `web/app.py` が同様の `or None` パターンを使用しており (行 1323-1326)、動作の一貫性を保つためこのまま維持。将来の `ProcessPort` 抽出時に DTO を厳密型にする。
> 2. 同上。`_current_task` アクセスは既存 handler と同じ技術的負債であり、本イテレーションのスコープ外。
> 3. コメントを `drain_while_busy` docstring で明示済み ("We model the moment the task finishes")。
> 4. `task_timeout=0` は実用上無効な設定（すべてのタスクが即タイムアウト）だが、バリデーターを追加して明示的に `task_timeout > 0` を要求する。
