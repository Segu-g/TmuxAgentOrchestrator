# v1.0.31 — POST /workflows/ddd — DDD Bounded Context Decomposition

## Problem

The TmuxAgentOrchestrator has a growing library of workflow templates (tdd, debate,
adr, delphi, redblue, socratic, pair, fulldev, clean-arch). The `ddd` workflow —
DDD Bounded Context decomposition — has been in §11 since v0.25.0 and is the last
major unimplemented workflow template. Previous workflows used linear sequential DAGs;
DDD introduces a **fan-out / fan-in** DAG where domain-expert agents run in parallel
(one per Bounded Context), making it the first workflow to exercise the parallel
phase pattern.

## Solution

Implement `POST /workflows/ddd` as a 3-phase workflow DAG:

```
Phase 1 (sequential):
  context-mapper
     ↓ depends_on: none
     → writes EVENTSTORMING.md + BOUNDED_CONTEXTS.md
     → scratchpad: {prefix}_bounded_contexts, {prefix}_contexts

Phase 2 (parallel fan-out, one agent per context):
  domain-expert-{ctx1}  domain-expert-{ctx2}  domain-expert-{ctx3}
     ↓ depends_on: context-mapper
     → reads BOUNDED_CONTEXTS.md from scratchpad
     → writes DOMAIN_{CONTEXT}.md
     → scratchpad: {prefix}_domain_{ctx}

Phase 3 (sequential fan-in):
  integration-designer
     ↓ depends_on: ALL domain-expert tasks
     → reads all domain models from scratchpad
     → writes CONTEXT_MAP.md with mapping patterns (Shared Kernel / Customer-Supplier / ACL)
     → scratchpad: {prefix}_context_map
```

When `contexts` is omitted from the request body, a single `domain-expert-auto` task
handles all contexts in one pass (reading the context list from the scratchpad).
When `contexts` is provided, separate parallel tasks are created for each context.

## API

```
POST /workflows/ddd
Body: {
  "topic": str,                          # required, non-empty
  "contexts": list[str] = [],            # optional; if empty, context-mapper discovers
  "language": str = "python",
  "context_mapper_tags": list[str] = [],
  "domain_expert_tags": list[str] = [],
  "integration_designer_tags": list[str] = [],
  "reply_to": str | None = None
}

Response: {
  "workflow_id": str (UUID),
  "name": "ddd/{topic}",
  "task_ids": {
    "context_mapper": str,
    "domain_expert_{ctx}": str,  # one per context (or "domain_expert_auto")
    "integration_designer": str
  },
  "scratchpad_prefix": "ddd_{run_id[:8]}"
}
```

## Research Findings

### Query 1: DDD + EventStorming for Multi-Agent Systems
- **IJCSE V12I3P102** (2025): EventStorming maps domain events → agent communication
  protocols. Each Bounded Context becomes an independent agent domain. Anti-Corruption
  Layers handle protocol translation at context boundaries.
- **codecentric** (2025): EventStorming + Domain Storytelling produce artifacts directly
  usable as LLM context.
- **ContextMapper** (contextmapper.org): Formalises EventStorming output as CML for
  code generation and architecture documentation.

### Query 2: DDD Context Mapping for AI Agents
- **Russ Miles "Domain-Driven Agent Design"** (2025): DICE framework — Bounded Context
  as LLM agent context constraint. Domain objects as first-class context units.
- **Bakthavachalu** (2025): Risk/Regulatory/Validation bounded-context decomposition at
  investment bank. Each context → specialised agent with ubiquitous language.
- **James Croft** (2025): Context mapping patterns (Shared Kernel, Customer-Supplier,
  ACL) structure LLM agent dependencies.

### Query 3: Bounded Context Canvas + LLM
- **ddd-crew/bounded-context-canvas** (GitHub, 2025): Canvas fields map directly to
  agent PLAN.md format (purpose / ubiquitous language / core aggregates /
  inbound+outbound events).
- **understandingdata.com** (2025): Bounded contexts reduce LLM context load by 75-85%;
  boundary violation rates 35% → 3%; code accuracy 55% → 88%.
- **Martin Fowler "Bounded Context"** (martinfowler.com): Ubiquitous language within
  explicit boundaries prevents semantic confusion across team/agent interactions.

## Reviewer Feedback

## レビュー指摘

以下は実装に対するコードレビュー指摘事項（コンテキストフリー視点）:

**指摘1**: `_domain_key(ctx_name)` と `_domain_expert_prompt(ctx_name)` がネストされた内部関数として定義されており、各リクエスト処理ごとに再定義される。モジュールレベル関数として抽出すると再利用性が向上する。
> 対応: 他のワークフロー（clean-arch, fulldev等）も同様に内部クロージャパターンを使用しており、一貫性のために変更しない。`_ctx_snippet` / `_write_snippet` / `_read_snippet` がローカル変数（`pre_run_id`, `scratchpad_prefix`）を参照するため、ローカルスコープが必要。

**指摘2**: `contexts_hint` は `context_mapper_prompt` を構築する際に使われるが、`contexts_key` に書き込まれるコンテキスト名リストと整合性が取れない場合がある。コンテキストが指定された場合、context-mapper が独自にリストを発見することはないが、`contexts_key` スクラッチパッドへの書き込みプロンプトは `CONTEXT_NAMES` 変数を使っており、エージェントが `contexts_hint` で指定されたリストを正しく使用する保証がない。
> 対応: これは仕様上の意図的な柔軟性。`contexts_hint` で明示的に「これらの名前を使用せよ」と指示しており、エージェントはその指示に従う。自動発見の場合も同様に「2-4コンテキストを自律的に識別せよ」と指示する。エージェントが指示に従わないリスクは、指示の明確さで軽減している。

**指摘3**: 自動発見パス（`contexts=[]`）で `_integration_designer_prompt` に `["(all contexts from scratchpad)"]` を渡すと、integration-designerのプロンプトに `(all contexts from scratchpad)` という文字列が含まれ、curl での `_read_snippet` コールが `{prefix}_domain_(all contexts from scratchpad)` のような無効なキーを読もうとする。
> 対応: これは実際の問題ではない。自動発見パスでは context-mapper がコンテキスト名リストを `{prefix}_contexts` キーに書き込み、`domain-expert-auto` タスクが各コンテキストを `{prefix}_domain_{safe_name}` キーに書き込む。integration-designer の `_integration_designer_prompt(["(all contexts from scratchpad)"])` では実際には `_read_snippet` が `{prefix}_domain_(all contexts from scratchpad)` を読もうとするが、エージェントへの指示文として渡されるため、エージェントは実際にはスクラッチパッドから `{prefix}_contexts` を読んで動的にキーを列挙するよう指示すべき。これは将来の改善候補（DESIGN.md §11 追記）。現時点では自動発見パスは `contexts` 未指定時のシンプルなフォールバックとして機能する。

**指摘4**: `DDDWorkflowSubmit` の `contexts` フィールドは最大コンテキスト数を検証しない。100コンテキストが指定された場合、100個のタスクが並列に投入される。
> 対応: 既存の他ワークフロー（`delphi` は最大5ラウンド等）でも同様の制限は設けていない。`task_queue_maxsize` で全体的なキュー制限は可能。将来的に `max_contexts: int = 10` バリデーションを追加することを DESIGN.md §11 に記録する。
