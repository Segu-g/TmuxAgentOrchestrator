# v1.0.17 — infrastructure/ 層の継続移行: worktree / process_port / messaging

## 概要

v1.0.16 で `infrastructure/tmux.py` と `infrastructure/claude_trust.py` を抽出した。
本イテレーションでは残る外部I/Oアダプターを同様に移行する。

---

## 移行対象ファイルの分析

### 1. `worktree.py` → `infrastructure/worktree.py`

**理由**: `WorktreeManager` は `git` コマンドを `subprocess` 経由で呼び出す外部プロセスアダプター。
ドメインロジック（Agentのライフサイクル状態）とは分離されるべき I/O 境界実装である。
Clean Architecture の「外部システムアダプター」定義に合致する。

**依存分析** (インポート先):
- `src/tmux_orchestrator/factory.py` — `from tmux_orchestrator.worktree import WorktreeManager`
- `src/tmux_orchestrator/agents/base.py` — TYPE_CHECKING 内（型ヒントのみ）
- `src/tmux_orchestrator/agents/claude_code.py` — TYPE_CHECKING 内（型ヒントのみ）
- `src/tmux_orchestrator/orchestrator.py` — TYPE_CHECKING 内（型ヒントのみ）
- `tests/test_worktree.py` — `from tmux_orchestrator.worktree import WorktreeManager`
- `tests/test_worktree_integrity.py` — `from tmux_orchestrator.worktree import WorktreeManager`

**移行方針**:
- 実装を `infrastructure/worktree.py` に移動
- 旧パス `tmux_orchestrator.worktree` は Strangler Fig shim として残す

### 2. `process_port.py` → `infrastructure/process_port.py`

**理由**: `TmuxProcessAdapter` は `libtmux.Pane` と `TmuxInterface` をラップする tmux アダプター。
`StdioProcessAdapter` はテスト用フェイクであり、テスト層のみに属するべきだが、
`ProcessPort` プロトコル（Port定義）は application/ 層向けのインターフェース。
現状は全部同一ファイルにある。

`TmuxProcessAdapter` は外部システム (tmux) に依存するため infrastructure/ に属する。
`ProcessPort` プロトコルと `StdioProcessAdapter` も同ファイルに留める
（分割コストよりも一体性を優先し、既存テストへの影響を最小化する）。

**依存分析** (インポート先):
- `src/tmux_orchestrator/infrastructure/tmux.py` — 内部 TYPE_CHECKING import
- `tests/test_process_port.py` — `from tmux_orchestrator.process_port import ...`
- `src/tmux_orchestrator/process_port.py` (docstring example) — 自己参照

**移行方針**:
- 実装を `infrastructure/process_port.py` に移動
- 旧パス `tmux_orchestrator.process_port` は Strangler Fig shim として残す

### 3. `messaging.py` → `infrastructure/messaging.py`

**理由**: `Mailbox` クラスはファイルシステムを使ったメッセージ永続化アダプター。
I/O 操作（`json.dumps`, `shutil.move`, `Path.mkdir`）を含む外部システム依存実装。

**依存分析** (インポート先):
- `src/tmux_orchestrator/factory.py` — `from tmux_orchestrator.messaging import Mailbox`
- `src/tmux_orchestrator/orchestrator.py` — `from tmux_orchestrator.messaging import Mailbox`
- `src/tmux_orchestrator/agents/base.py` — TYPE_CHECKING 内
- `src/tmux_orchestrator/agents/claude_code.py` — TYPE_CHECKING 内
- `tests/test_messaging.py` — `from tmux_orchestrator.messaging import Mailbox`
- `tests/test_result_routing.py` — `from tmux_orchestrator.messaging import Mailbox`
- `tests/integration/test_orchestration.py` — `from tmux_orchestrator.messaging import Mailbox`

**移行方針**:
- 実装を `infrastructure/messaging.py` に移動
- 旧パス `tmux_orchestrator.messaging` は Strangler Fig shim として残す

## 調査記録 (Research)

### Query 1: Hexagonal Architecture Ports and Adapters in Python

**出典**: Software Patterns Lexicon — "Hexagonal Architecture: Ports and Adapters in Python"
URL: https://softwarepatternslexicon.com/python/architectural-patterns/hexagonal-architecture-ports-and-adapters/

**知見**:
- Port は抽象インターフェース（ABC または Protocol）、Adapter はその具体実装
- Adapter はビジネスロジック（domain/）を外部システムから保護する
- Python では PEP 544 `Protocol` クラスが Port の定義に適している
- ファイルシステム・プロセス・ネットワークは全て「外部システム」で、それぞれ Adapter を持つ

### Query 2: Strangler Fig Pattern for Module Migration

**出典**: Microsoft Azure Architecture Center — "Strangler Fig Pattern"
URL: https://learn.microsoft.com/en-us/azure/architecture/patterns/strangler-fig

**出典**: FreeCodeCamp — "What is the Strangler Pattern in Software Development"
URL: https://www.freecodecamp.org/news/what-is-the-strangler-pattern-in-software-development/

**知見**:
- 旧パスを shim（再エクスポートのみのモジュール）として残すことで後方互換を維持
- テスト・外部コードは旧パスを使い続けられる
- 移行完了後に shim を削除して完全移行

### Query 3: Clean Architecture Dependency Inversion in Python

**出典**: ArjanCodes Blog — "Dependency Inversion Principle in Python Programming"
URL: https://arjancodes.com/blog/dependency-inversion-principle-in-python-programming/

**出典**: Cosmic Python — "Repository Pattern" (Percival & Gregory, O'Reilly, 2020)
URL: https://www.cosmicpython.com/book/chapter_02_repository.html

**知見**:
- git subprocess はファイルシステム操作と同様に「外部システム」として扱う
- Infrastructure アダプターはドメインの Port（Protocol）に依存し、逆は禁止
- Dependency Injection により application/ は infrastructure/ を直接参照しない

自己レビュー（コンテキストなしサブエージェントは利用不可のため、設計者によるクリーンレビュー）

### 指摘 1: `StdioProcessAdapter` は infrastructure/ に属するか？

`StdioProcessAdapter` はテスト専用のフェイクであり、外部システムに依存しない。
厳密には infrastructure/ ではなく tests/ に置くべき。
ただし、`ProcessPort` と `TmuxProcessAdapter` と同一ファイルにある現状を分割すると、
既存テストの import パスが全て変わるリスクがある。

> 対応: 今回は `StdioProcessAdapter` を `infrastructure/process_port.py` に含める。
> shim 経由で既存テストは変更不要。将来のイテレーションで `tests/fakes.py` 等への移動を検討する。

### 指摘 2: `infrastructure/tmux.py` 内の `process_port` import が旧パス参照

`infrastructure/tmux.py` line 206 に `from tmux_orchestrator.process_port import TmuxProcessAdapter` があり、shim 経由となっている。canonical パスへの直接参照が望ましい。

> 対応: shim は機能的に同一オブジェクトを返すため、動作上の問題はない。
> 将来の clean-up イテレーションで `infrastructure.tmux` → `infrastructure.process_port` に直接参照を更新する。

### 指摘 3: `worktree_integrity.py` は対象外

`worktree_integrity.py` は `WorktreeManager` に依存するが、今回の移行対象外としている。
shim 経由で `WorktreeManager` をインポートしているため、実害はない。

> 対応: 次のイテレーション候補として `DESIGN.md §11` に追記する。
